# Module: Announcement

## 1. Purpose & Current Usage

- Company-wide announcement system: HR/admins create, schedule, publish, archive announcements targeted at ALL employees, specific departments, locations, or individual employees; employees see a read-tracked feed.
- Entities: `Announcement` (brello_server/src/modules/announcement/entities/announcement.entity.ts), `AnnouncementTarget` (announcement-target.entity.ts), `AnnouncementRead` (announcement-read.entity.ts), `AnnouncementAttachment` (announcement-attachment.entity.ts).
- Services: `AnnouncementService` (admin CRUD/lifecycle) and `EmployeeAnnouncementService` (employee feed + read tracking), both backed by a shared `AnnouncementRepository`.
- Controllers: `AnnouncementController` (`/announcements`, admin) and `EmployeeAnnouncementController` (`/employee/announcements`).
- Registered in `brello_server/src/app.module.ts:86`. No other backend module imports `AnnouncementService`/`EmployeeAnnouncementService`/`AnnouncementModule` (verified via repo-wide grep) — the module is a leaf, called only from its own controllers.
- Webapp calls every exposed endpoint: `brello_webapp/src/features/announcement/api/announcementApi.ts` wraps create/list/get/update/delete/publish/archive (admin) and list/markRead (employee), consumed by `brello_webapp/src/pages/announcement/AnnouncementPage.tsx`, `EmployeeAnnouncementPage.tsx`, `CreateAnnouncementModal.tsx`, and surfaced on the dashboard via `AnnouncementCard.tsx` / `useDashboard.ts`. Global search integration (`SearchIndexingService.indexAnnouncement`/`removeAnnouncement`, `brello_server/src/modules/global-search/services/search-indexing.service.ts:238,258`) is called from `announcement.service.ts:76,212,230` on create/publish/archive and consumed by `brello_webapp/src/features/search/components/SearchItem/SearchItem.tsx`.
- Dead/unused: `AnnouncementRepository.findDueScheduled()` (`repositories/announcement.repository.ts:219`) is never called from anywhere in the codebase — it's the only place that would drive the scheduled-publish flow, and nothing invokes it (see §4).
- The `LOCATION` audience type is fully modeled (enum, DTO, entity, frontend "Specific locations" picker in `CreateAnnouncementModal.tsx:45`, `location_ids` in `announcementSchema.ts:18` and `announcementTypes.ts`) but has no effect end-to-end (see §4) — UI + storage exist, delivery does not.

## 2. Intended / Ideal Usage

- A PRD exists at `brello_server/docs/prd/announcement.md` and is detailed and current. Intended flow: Admin creates announcement (INSTANT or SCHEDULED) → if SCHEDULED, a **cron worker checks every minute** for due announcements, flips them to PUBLISHED, and **enqueues a notification job** (push/email/in-app) → employees see it in their feed filtered by ALL/DEPARTMENT/LOCATION/EMPLOYEE targeting, with read tracking.
- PRD's intended module structure (docs §"Module Structure") explicitly lists `services/announcement-query.service.ts` and `services/announcement-scheduler.service.ts` (cron worker) — **neither file exists**; all listing logic lives inline in `AnnouncementService`, and there is no scheduler at all.
- PRD "Background Jobs" section: "Scheduler runs every minute: finds SCHEDULED where scheduled_at <= now(), publishes + sends notifications" and a separate "Notification Worker (queue-based)" supporting push/email/in-app — not implemented anywhere in this module or connected from it.
- PRD "Security Considerations" mandates server-side sanitization of `description_html` (strip `<script>`, `on*` handlers, `javascript:` hrefs) using something like `sanitize-html` — not implemented (no `sanitize-html`/`DOMPurify` reference anywhere under `brello_server/src`).
- PRD "Permissions" table defines seven granular permissions (`announcement.create/view/edit/delete/publish/archive/view.analytics`) mirroring the `@RequirePermission('MODULE', 'action')` pattern already used elsewhere (e.g. `brello_server/src/modules/attendance/controllers/weekly-off.controller.ts:33`) — the announcement controllers use only `@UseGuards(JwtAuthGuard)` with no permission checks at all.
- PRD's `analytics` response contract for `GET /:id` includes `total_recipients` and `unread_count` alongside `read_count` — the actual service (`announcement.service.ts:145-147`) returns only `read_count`.

## 3. Cross-Module Connections

**Depends on:**
- `GlobalSearchModule` / `SearchIndexingService` — indexes announcements on create/publish, removes on archive (`announcement.module.ts:10,19`, `announcement.service.ts:76,212,230`).
- `User` entity/repository from the `user` module — resolves creator display names and an employee's `department_id` (`announcement.service.ts:91-97`, `employee-announcement.service.ts:23-24`).
- `AuditContextService` / `@AuditLog` decorator from the `audit` module — for CREATE/UPDATE/DELETE/PUBLISH/ARCHIVE actions (`announcement.controller.ts:19-21,34,55,65,72,79`).

**Depends on this module:** nothing else in the backend (`AnnouncementModule` only self-registers in `app.module.ts`); consumption is entirely from the webapp via HTTP.

**Missing/expected connections:**
- No call to any `NotificationService` anywhere in the module (grep confirms zero hits besides an unrelated field name `notification_clicked` on `AnnouncementRead`) despite `send_push`/`send_email` flags being stored on every announcement and exposed in the API — these flags are pure dead data with no consumer. This is the same class of gap as the PRD's unimplemented "Notification Worker."
- No cron/scheduler module references `findDueScheduled()` — compare to the working cron pattern used in `brello_server/src/modules/attendance/services/attendance-cron.service.ts:22` or `brello_server/src/modules/payroll/services/payroll-auto-run.cron.ts:41`. SCHEDULED announcements are created and stored but nothing ever transitions them to PUBLISHED.
- `AuditLog` decorator is applied to `update`, `publish`, and `archive` (`announcement.controller.ts:55-56,72-73,79-80`), but the decorator's own doc comment (`audit/decorators/audit-log.decorator.ts:15-16`) says: "Use ONLY for CREATE and DELETE — for UPDATE, APPROVE, REJECT, call auditService.log() manually... after fetching old_value." The service only calls `auditContext.setPreValue()` in `update()` and `remove()` (`announcement.service.ts:161,242`), never in `publish()`/`archive()` — so audit trail entries for publish/archive are missing the pre-change value, inconsistent with the codebase's own documented convention.

## 4. Gaps

### Structural
- **Scheduled publishing is entirely unimplemented.** `AnnouncementRepository.findDueScheduled()` (`repositories/announcement.repository.ts:219-226`) is dead code — no cron service calls it anywhere in the repo. Any announcement created with `publish_type: SCHEDULED` sits in `SCHEDULED` status forever unless an admin manually hits `POST /:id/publish`. This directly contradicts the PRD's core scheduling feature and its explicit `announcement-scheduler.service.ts` component that was never built.
- **No permission/RBAC enforcement on any endpoint.** Both controllers use only `@UseGuards(JwtAuthGuard)` (`announcement.controller.ts:29`, `employee-announcement.controller.ts:21`) with no `@RequirePermission` calls, even though the same pattern exists and is used elsewhere (`attendance/controllers/weekly-off.controller.ts:33`) and the PRD defines a 7-entry permission table. Any authenticated user of any role can create/edit/delete/publish/archive company-wide announcements.
- **LOCATION targeting is a fully-built dead end.** Enum (`enums/announcement.enum.ts:19-24`), entity, DTO, and webapp UI (`CreateAnnouncementModal.tsx:45`, `announcementSchema.ts:18`) all support `LOCATION`/`location_ids`, and `AnnouncementRepository.saveTargets` (`announcement.repository.ts:241-244`) happily persists them — but `findPublishedForEmployee` (`announcement.repository.ts:161-190`) only ever filters on `ALL`, `EMPLOYEE`, and conditionally `DEPARTMENT`; `LOCATION` is never referenced in the visibility WHERE clause, and there is no `location_id` field on `User` at all (confirmed absent from `user/entities/user.entity.ts`). Any announcement targeted at a location is created successfully but is invisible to every employee, silently.

### Coding
- **No HTML sanitization of `description_html`.** It's stored and returned verbatim (`announcement.service.ts:61,134,177`; `employee-announcement.service.ts:44`) with no `sanitize-html`/`DOMPurify`-equivalent call anywhere under `brello_server/src`, despite the PRD explicitly requiring it and the field being rendered as raw HTML on the employee feed. This is a stored-XSS vector — any HR/admin user (or anyone whose account is compromised) can inject `<script>`/`onerror=` payloads that execute in every employee's browser.
- **`findAll` search uses raw `LIKE` with a manually interpolated wildcard** (`announcement.repository.ts:52`: `` a.title LIKE :search `` with `search: `%${query.search}%`` as a bound parameter — this is parameterized so not SQL-injectable, but it silently ignores the codebase's own dedicated full-text search module (`GlobalSearchModule`/pg_trgm) that this same file already imports for indexing, duplicating simplistic substring search instead of reusing FTS.
- **Analytics response is incomplete vs. its own documented contract.** PRD's `GET /:id` response includes `analytics.total_recipients` and `analytics.unread_count`; the actual implementation (`announcement.service.ts:145-147`) returns only `read_count`, so the webapp/admin UI can never show unread counts or total audience size as originally specified.
- **`update()` swallows partial-update semantics for `audience`/`attachments`.** `dto.audience` has no `@IsOptional()` on `CreateAnnouncementDto` (`dto/create-announcement.dto.ts:80-82`), yet `UpdateAnnouncementDto extends PartialType(CreateAnnouncementDto)` making it optional for updates — however `AnnouncementRepository.update()` treats "audience provided" as "replace all targets" (`announcement.repository.ts:126-129`) and "attachments provided" (even as `[]`) as "delete and replace" (`announcement.repository.ts:131-145`); there's no way for a caller to update just the title without accidentally being forced to also resend the full existing audience/attachments, since the DTO layer gives no signal distinguishing "omitted" from "intentionally cleared" once it passes through `class-transformer`'s partial handling — a fragile implicit contract.

### Technical
- **No logging/observability in the entire module.** Zero `Logger` usage across all 15 files (service, controller, repository) — failures in `SearchIndexingService` calls (which are fire-and-forget, not awaited: `announcement.service.ts:76,212,230` call `.indexAnnouncement(...)`/`.removeAnnouncement(...)` without `await` or `.catch()`) would be silently lost with no trace.
- **`getReadStatusForEmployee` builds an O(n) `where` array of per-id objects** (`announcement.repository.ts:198-200`) instead of a single `In(announcementIds)` + `employee_id` query — inefficient but functionally fine at current scale; worth noting as it will generate a large parameterized `OR` query as announcement/employee volume grows.
- **No test coverage.** No `*.spec.ts` files exist anywhere under `brello_server/src/modules/announcement/` (confirmed via directory listing) — none of the lifecycle rules (status transition guards, scheduled-date validation, audience resolution) have automated tests.

## 5. Top 3 Priorities

1. **Implement the scheduled-publish cron worker.** `findDueScheduled()` already exists and is correct; wiring a `@Cron` job (matching the existing pattern in `attendance-cron.service.ts`/`payroll-auto-run.cron.ts`) to call it and transition + index announcements is the single highest-value fix — right now the "SCHEDULED" feature is completely non-functional in production.
2. **Add permission enforcement (`@RequirePermission`) to both controllers.** Currently any logged-in user, including regular employees, can create/edit/delete/publish/archive company-wide announcements — a straightforward but serious authorization gap given the PRD already specifies the permission set.
3. **Sanitize `description_html` server-side before persisting.** This is a stored-XSS hole affecting every employee who views the announcement feed, and the PRD already names the exact mitigation (`sanitize-html`) that was never wired in.
