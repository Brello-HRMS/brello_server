# Module: Audit

## 1. Purpose & Current Usage

The module lives at `brello_server/src/modules/audit/` and provides a single centralized, org/enterprise-scoped audit trail (`system_audit_logs`) for the whole backend:

- **Entity**: `entities/system-audit-log.entity.ts` — one flat, append-only table (no `updated_at`) with actor snapshot (name/email/role captured at write time), module/action enums, entity type+id, `old_value`/`new_value` JSONB, computed `changed_fields`, and request context (ip/user-agent/device/request_id). Six composite indexes for org+time, org+module, org+actor, entity history, org+action, enterprise+time.
- **Enums**: `AuditLogModule` (29 values: AUTH, SESSION, EMPLOYEE…PLATFORM_SETUP, LETTER_MANAGEMENT/OFFER_LETTER/LETTER_SIGNATORY) and `AuditAction` (29 values covering CRUD, workflow, payroll lifecycle, auth, data ops, letter actions).
- **Services**: `AuditService` (writes + all query methods), `AuditActorResolver` (userId → {name,email} with 5-min in-memory cache, falls back to "Deleted User (xxxx…)"), `AuditContextService` (AsyncLocalStorage carrying per-request actor/ip/preValue/bulkEntities).
- **Decorator + Interceptor**: `@AuditLog(module, action, entityType?, opts?)` sets metadata; the global `AuditInterceptor` (registered via `APP_INTERCEPTOR`) reads it, unwraps the `{success,data,timestamp}` response envelope, resolves entity id/display name, and fires `auditService.log()` — intended for CREATE/DELETE only. For UPDATE/APPROVE/REJECT, services call `auditContext.setPreValue(existing)` before mutating so the interceptor can diff old vs new.
- **Controllers**: `AuditController` (`/audit-logs`, org-scoped, gated by `RequirePermission('AUDIT_LOG','view')`) and `PlatformAuditController` (`/platform/audit-logs`, gated by `PlatformAdminGuard`) — both read-only (list, filter-options, stats, entity history).
- **Repository**: `AuditLogRepository` — plain TypeORM query builder, no raw SQL beyond `ANY(changed_fields)` and ILIKE search; no update/delete methods exist at all.
- **Utils**: `diff.util.ts` (shallow key-diff) and `sanitize.util.ts` (redacts password/otp/token/account_number-ish keys with `[REDACTED]`).

**Who calls it today** (repo-wide grep for `@AuditLog(`, `AuditService`/`auditService`, `setPreValue`/`setBulkEntities`):
Attendance, Attendance Rule, Shift, Admin Attendance, Organization, RBAC (user-role assign/revoke), Leave Balance, Holiday, Holiday Calendar, Role, Enterprise, Plan, Platform Department/Designation, Designation, Department, Employee (partial — see gaps), Document (DELETE only), Project/Client-Project, Company Policy (+ type), Letter Management (categories, signatories, templates, issued-letters — manual `auditService.log()` calls for GENERATE/VIEW/DOWNLOAD/ACKNOWLEDGE), Leave Config, Payroll (run, adjustment, preparation, processing, component-master, salary-template, employee-salary, pf-config), Announcement, Leave Request, Client, Billing **Subscription** (not Payment/Invoice — see gaps), Reimbursement, Auth (login/login_failed/logout via manual `logAuthEvent`), Employee (CREATE on onboarding only, in `employee.service.ts`).

That is genuinely broad, real coverage — this is not a shelf-ware module. The frontend (`brello_webapp/src/features/audit/`, `pages/audit/AuditPage.tsx`, `PlatformAuditPage.tsx`) also already implements the list, diff viewer, detail drawer, and stats hooks described in the PRD, contradicting the PRD's "webapp has zero audit log pages" framing — that part of the PRD is stale.

**Dead/unused parts**: `AuditLogModule.BILLING`, `AuditLogModule.PERMISSION`, `AuditLogModule.SESSION`, `AuditLogModule.HR_TEMPLATE`, and `AuditAction.EXPORT`, `IMPORT`, `WITHDRAW`, `RESTORE` have zero call sites anywhere in `brello_server/src` (confirmed by repo-wide grep) — these are PRD-catalog values that were stubbed into the enums but never wired to any controller or service.

## 2. Intended / Ideal Usage

Per `docs/prd/audit-log.md`: a single unified table replacing 4 fragmented per-module audit implementations, with >95% coverage of state-changing operations, DB-level immutability (revoked UPDATE/DELETE grants + RLS), day-one partitioning by `created_at`, a 90-day-max-range query API, CSV/XLSX export, and a 5-week phased rollout (infra → migrate legacy tables → wire all missing modules → export → frontend). The actual implementation matches the PRD's schema/enum/service shape almost exactly (same table name, same index names, same DTO shape) — this was clearly built directly from the PRD. Where it diverges is in the operational hardening pieces (immutability enforcement, partitioning, export) and in system-initiated (non-HTTP) events, both covered below.

## 3. Cross-Module Connections

**Depends on** (imports/injects):
- `modules/user/entities/user.entity.ts` (`User`) — `AuditActorResolver` queries it directly for name/email snapshots.
- `modules/rbac/rbac.module.ts` (`RbacModule`) — imported so `AccessGuard` (used on `AuditController`/`PlatformAuditController`) can resolve `PermissionResolverService`.
- `core/guards/jwt-auth.guard.ts`, `core/guards/access.guard.ts`, `core/guards/platform-admin.guard.ts` — for endpoint gating.

**Depended on by** (every module found in the grep in §1): Attendance (+rule, shift, admin), Organization, RBAC/UserRoleMap, Leave Balance, Holiday (+calendar), Role, Enterprise, Plan, Platform (department/designation), Designation, Department, Employee/User, Document, Project/Client-Project, Company Policy (+type), Letter Management (categories/signatories/templates/issued-letters), Leave Config, Payroll (run/adjustment/preparation/processing/component-master/salary-template/employee-salary/pf-config), Announcement, Leave Request, Client, Billing (Subscription only), Reimbursement, Auth. `AuditCoreModule` is `@Global()` and exports `AUDIT_SERVICE_TOKEN`, `AuditContextService`, `AuditInterceptor` so any module can inject `IAuditService` without a direct module import.

**Missing/expected connections**:
- **Billing → Payment/Invoice**: `PaymentController` (initiate/link/verify) and `InvoiceController` never call the audit service, and `RazorpayWebhookController` (webhook-driven payment success/failure) has zero audit wiring — see Gaps.
- **Employee lifecycle**: `/activate`, `/system-access`, `/offboarding/initiate`, `/offboarding`, `/employment`, `/payroll` endpoints on `employee.controller.ts` have no `@AuditLog` and their services never call `auditService.log()` manually — only `/personal` is decorated.
- **Cron/system jobs**: none of `modules/billing/crons/*`, `modules/payroll/services/payroll-auto-run.cron.ts`, `modules/payroll/services/payroll-reminder.cron.ts`, `modules/attendance/services/attendance-cron.service.ts`, `modules/user/services/offboarding-cron.service.ts` call the audit service — system-initiated mutations (subscription auto-expiry, auto payroll runs, offboarding automation) are invisible to the audit trail.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **The entire audit-context mechanism is HTTP-request-bound and has no story for system-initiated actions.** `AuditInterceptor.intercept()` bails out immediately if there's no logged-in user (`interceptors/audit.interceptor.ts:39-40`: `const user = request.loggedInUser; if (!user) return next.handle();`), and there is no `SYSTEM_ACTOR_ID` constant or documented pattern anywhere in the codebase for logging cron/webhook-driven events (confirmed: zero `AuditService`/`auditService.log` references in any of the 8 cron files or in `razorpay-webhook.controller.ts`). This matters because subscription auto-expiry, auto-run payroll, and payment webhooks are exactly the kind of unattended, high-stakes mutation an audit trail exists to catch, and today none of them can even reach `AuditService` — not because someone forgot to call it, but because the actor-context plumbing has nowhere for them to plug in.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

- **Billing money-movement is entirely unaudited.** `modules/billing/controllers/payment.controller.ts` (initiate/link/verify) and `invoice.controller.ts` have no `@AuditLog` decorator or manual `auditService.log()` call anywhere; `razorpay-webhook.controller.ts:handle()` processes payment success/failure via `paymentService.handleWebhookEvent(event)` with no audit call either. This is the single most compliance-sensitive category (real money changing hands) and the PRD's own event catalog (§7.16) explicitly requires `PAY`/`CREATE` events for it, yet it's fully unimplemented.
- **Several sensitive Employee-lifecycle mutations are unaudited.** In `modules/user/controllers/employee.controller.ts`, only `PATCH :id/personal` (line 103-104) carries `@AuditLog`; `POST :id/activate` (156), `PATCH :id/system-access` (235), `POST :id/offboarding/initiate` (244), `PATCH :id/offboarding` (253), `PATCH :id/employment` (130), and `PATCH :id/payroll` (139) have none, and their services don't call `auditService.log()` manually either. `system-access` (login/access grants) and offboarding are called out by name in the PRD's own events catalog (§7.2) as must-audit.
- **Dead enum values reveal unfinished wiring.** `AuditLogModule.BILLING`, `PERMISSION`, `SESSION`, `HR_TEMPLATE` and `AuditAction.EXPORT`, `IMPORT`, `WITHDRAW`, `RESTORE` (`enums/audit-log-module.enum.ts`, `enums/audit-action.enum.ts`) have zero call sites anywhere in `brello_server/src`. `PERMISSION`/`SESSION` in particular map to PRD-mandated events (granular permission grant/revoke, session expiry) that never got implemented — role permission changes are only visible indirectly as part of a generic `ROLE` `UPDATE` diff, not as their own `PERMISSION` `GRANT`/`REVOKE` events as the PRD's events catalog (§7.5) specifies.
- **`PlatformAuditController.getEntityHistory` silently degrades instead of validating.** `controllers/platform-audit.controller.ts:34-41` takes `organization_id` as a bare `@Query()` param (not routed through `PlatformAuditLogQueryDto`, so no `@IsUUID` validation) and passes `organizationId ?? ''` straight into the repository filter — omitting the param doesn't error, it just silently returns an empty result set, which will look like "no history" to a platform admin rather than a clear 400.

### Technical (performance, security, scalability, test coverage, observability/logging)

- **The table is not actually immutable at the database level.** The PRD's own §12.1 calls for `REVOKE UPDATE, DELETE ON system_audit_logs FROM brello_app_user` plus RLS policies blocking UPDATE/DELETE, but the real migration (`docs/migrations/2026-06-21-system-audit-logs.sql`) contains neither — it only creates the table, enums, and indexes. Immutability today is enforced purely by convention (no repository method exists to update/delete a row), not by the database, so the "tamper-proof, compliance-grade" claim in the PRD's overview is not actually backed by the DB.
- **No partitioning despite explicit PRD instruction to do it "from day one."** PRD §15 is explicit that partitioning by `created_at` range should happen at table creation specifically so archival is a cheap partition-detach later; the actual migration creates a plain, non-partitioned table. Retrofitting partitioning onto a live table later is a much more expensive migration than doing it up front.
- **No export endpoint at all.** PRD §9.4 (`GET /audit-logs/export?format=csv|xlsx`) is a core requirement for the module's stated compliance/HR-audit purpose; `AuditController` and `PlatformAuditController` only expose `findAll`/`filter-options`/`stats`/`entity history` — there is no export route in either controller.
- **Zero test coverage.** There are no `*.spec.ts` files anywhere under `modules/audit/` (confirmed — 9 spec files exist repo-wide, none for audit), so `computeChangedFields`, `sanitizeForAudit`, and the interceptor's payload-unwrapping/entity-id-resolution logic (each with several branches and fallbacks) are entirely unverified by automated tests.

## 5. Top 3 Priorities

1. **Wire Billing payments/invoices/webhooks into the audit log.** This is the biggest blind spot in an otherwise well-covered module — real money movement (payment initiation, verification, webhook-confirmed success/failure, invoice generation) leaves no trace today, directly undermining the compliance use case the PRD was written to solve.
2. **Enforce immutability at the database level** (revoke UPDATE/DELETE grants on `system_audit_logs` from the app DB role, or add RLS policies) — right now "append-only" is a convention enforced only by the repository's method set, not a guarantee, which is a meaningful gap for a module whose entire premise is tamper-proof compliance records.
3. **Establish a system-actor pattern and wire crons/webhooks through it**, then backfill the remaining sensitive Employee endpoints (`/activate`, `/system-access`, `/offboarding/*`) — without this, subscription expiry, auto-payroll runs, and account access changes remain permanently invisible to the audit trail regardless of how well HTTP-driven mutations are covered.
