# Module: Auth

## 1. Purpose & Current Usage

`brello_server/src/modules/auth/` implements login, session/JWT lifecycle, OTP verification, password management, multi-app switching, and a fully parallel platform-admin authentication flow.

- **Entities**: `entities/session.entity.ts` (refresh-token session record, device fingerprint, expiry), `entities/otp.entity.ts` (hashed OTP, purpose, attempt counter).
- **Repositories**: `repositories/session.repository.ts`, `repositories/otp.repository.ts` — thin TypeORM wrappers.
- **Services**: `services/auth.service.ts` (password login, OTP login, refresh, logout, update-password, forgot/reset-password, resend-OTP, switch-app), `services/platform-admin-auth.service.ts` (separate register/login/OTP flow for platform admins), `services/token.service.ts` (JWT + session issuance), `services/cookie.service.ts` (HttpOnly refresh-token cookie).
- **Controllers**: `controllers/auth.controller.ts` (`/auth/*`), `controllers/platform-admin-auth.controller.ts` (`/auth/platform-admin/*`).
- **Guards/Strategies**: `guards/jwt-auth.guard.ts` (respects `@Public()`), `guards/jwt-refresh-auth.guard.ts`, `strategies/jwt.strategy.ts` (Bearer access token), `strategies/jwt-refresh.strategy.ts` (refresh JWT from HttpOnly cookie).
- **Tasks**: `tasks/otp-cleanup.task.ts` — hourly OTP purge, daily expired-session purge.
- **Utils**: `utils/crypto.util.ts` (bcrypt hash/verify), `utils/otp.util.ts` (6-digit OTP via `Math.random`), `utils/expiration.util.ts`, `utils/app.util.ts` (available-app resolution + default app selection).

Who calls in: virtually every guarded controller in the app imports `JwtAuthGuard` from `modules/auth/guards/jwt-auth.guard.ts` (60+ controllers, e.g. `src/modules/user/controllers/user.controller.ts`, `src/modules/billing/controllers/invoice.controller.ts`, `src/modules/project/controllers/project.controller.ts`). `AuthService` itself is consumed outside the module by `src/modules/organization/services/organization.service.ts:57-58,298` (via `forwardRef`) to mint a fresh token after company-setup completes.

Dead/unused code: `SessionRepository.findActiveSessionsByUserId` (`repositories/session.repository.ts:35-44`) is never called anywhere in the codebase (`grep` confirms zero call sites) and is also logically broken (see Gaps).

## 2. Intended / Ideal Usage

An auth module of this kind should: hash and never log secrets (mostly followed — bcrypt via `crypto.util.ts`); issue short-lived access tokens + rotating refresh tokens bound to a server-side session record (partially followed — rotation happens, but see gaps on revocation); rate-limit and lock out credential-guessing endpoints (login, OTP verify); detect/alert on new-device or new-location logins; notify the user out-of-band on every security-sensitive change (password change/reset, new session); and audit every such event immutably. Code comments show real intent here — `dto/auth-response.dto.ts:13` and `services/cookie.service.ts:12-16` explicitly document the HttpOnly/Secure/SameSite/XSS/CSRF rationale for the refresh-token cookie, and `strategies/jwt-refresh.strategy.ts:11-13` documents why the refresh token isn't Bearer-carried. No TODOs exist in the module referencing rate limiting, lockout, or device-alerting — these are gaps of omission rather than documented-but-unfinished work.

## 3. Cross-Module Connections

**Depends on** (imports/injects): `UserService`/`EmployeeService` (`modules/user`), `UserRoleMap` entity (`modules/rbac`), `Role` entity (`modules/role`), `App` entity (`modules/app`), `NotificationService` (`modules/notification`), `IAuditService`/`AUDIT_SERVICE_TOKEN` (`modules/audit`) — see `auth.module.ts:19-25` and `auth.service.ts:20-55`. Auth imports `UserRoleMap`/`Role`/`App` **entities directly** via `TypeOrmModule.forFeature` (`auth.module.ts:38-44`) and hand-builds raw query-builder joins in `utils/app.util.ts:20-107` rather than calling an RBAC/App service — a layering violation that spreads role/app schema knowledge into the auth module.

**Depended on by**: essentially every guarded controller (60+, listed above) via `JwtAuthGuard`; `organization.service.ts` via `forwardRef(() => AuthService)` (`organization.service.ts:57-58`) to call `buildAuthResponse` after org setup — this creates a genuine **circular dependency** between `auth` and `organization` modules.

**Missing/expected connections**:
- No connection from `updatePassword()` / `verifyOtpAndResetPassword()` to `NotificationService` — a password change/reset sends no confirmation email (contrast with login-OTP, which does notify).
- No connection from `updatePassword()` / `verifyOtpAndResetPassword()` / `switchApp()` to the audit service — only `LOGIN`, `LOGIN_FAILED`, and `LOGOUT` are ever audited (`auth.service.ts:123,222,404`); there is no `PASSWORD_CHANGE` value in `AuditAction` at all (`modules/audit/enums/audit-action.enum.ts`).
- `PlatformAdminAuthService` never calls the audit service at all (no `IAuditService` injected) — platform-admin login/registration, the highest-privilege actions in the system, are completely unaudited.

## 4. Gaps

### Structural
- **Duplicate `JwtAuthGuard` implementations with different behavior.** `src/core/guards/jwt-auth.guard.ts` is a second, independent `AuthGuard('jwt')` subclass that does **not** check the `@Public()` reflector, while `src/modules/auth/guards/jwt-auth.guard.ts:31-39` does. 16 controllers (e.g. `modules/payroll/controllers/payroll.controller.ts`, `modules/reimbursement/controllers/reimbursement.controller.ts`, `modules/audit/controllers/audit.controller.ts`, `modules/announcement/controllers/announcement.controller.ts`) use the `core` guard while 40+ others use the `auth` module's guard. Any future `@Public()` route added under a `core`-guarded controller will silently fail to bypass auth (or worse, developers may assume parity and misconfigure a route). Matters because it's a hidden inconsistency in the single most security-critical cross-cutting mechanism in the app.
- **Circular module dependency**: `organization.service.ts:57-58` requires `forwardRef(() => AuthService)` to re-mint a token after org setup, meaning `auth` and `organization` modules depend on each other. Matters because it couples two modules that should be one-directional (auth should not need to know about org setup, and vice versa) and is a sign `buildAuthResponse` should be reachable via a narrower interface/event instead of a full service import.
- **Auth reaches directly into RBAC/Role/App schemas.** `utils/app.util.ts:20-107` builds three raw query-builder joins across `UserRoleMap`, `Role`, `App`, and `role_apps` instead of delegating to an RBAC/App service. Matters because any schema change in those modules (e.g. renaming `roleApps`) silently breaks auth's login response with no compile-time signal from the owning module.

### Coding
- **`SessionRepository.findActiveSessionsByUserId` is both dead and wrong.** `repositories/session.repository.ts:35-44` filters `expires_at: LessThan(new Date())`, which returns **expired** sessions, not active ones (should be `MoreThan`/`>=`). It is never called anywhere in the codebase. Matters because if this method is ever wired up for a "manage your sessions" UI, it will return the opposite of what's asked, and a reviewer might assume it already works since it's named correctly.
- **`expires_in` is hardcoded to `900` in three places** — `services/token.service.ts:85`, `services/auth.service.ts:460` (`refreshToken`), and `services/auth.service.ts:384` (`switchApp`) — while the actual access-token lifetime is driven by config (`auth.JWT_ACCESS_EXPIRATION`, default `'15m'`, `token.service.ts:33-34`). Matters because changing that config value (e.g. to `30m` for a specific environment) makes the client-reported expiry silently wrong, causing premature or delayed refresh-timer behavior on the frontend.
- **`switchApp()` never persists the new `appId` to the session, nor rotates the refresh token.** `auth.service.ts:331-386` issues a new **access** token scoped to the new app but leaves the refresh-token JWT (and the `sessions.app_id` column) pointing at the original login-time app. A subsequent `POST /auth/refresh` (`auth.service.ts:440-447`) copies `appId` straight from the old refresh-token payload, silently reverting the user's app context back to the default app. Matters because this is a real, reachable functional bug (switch app → sit idle past access-token expiry → refresh → back on the wrong app) not just a theoretical one.
- **Inconsistent password-strength enforcement across the three places a password can be set.** `dto/update-password.dto.ts:11-14` and `dto/platform-admin.dto.ts:44-47` both require the uppercase/lowercase/digit/special-char regex, but `dto/forgot-password.dto.ts:21` (`VerifyOtpAndResetPasswordDto.new_password`) only enforces `Length(8,100)` with no complexity `@Matches`. Matters because the "forgot password" path — the one most attractive to attackers doing credential-stuffing recovery — is the weakest-enforced path to set a password.

### Technical
- **No rate limiting or lockout on any auth endpoint.** A repo-wide search for `Throttle`/`RateLimit`/`throttler` returns zero hits. `loginWithPassword` (`auth.service.ts:102-126`) has no attempt counter, delay, or lockout at all — only OTP flows cap attempts (`maxAttempts`, default 5, `auth.service.ts:184-190,570-574`), and that cap is per-OTP-record, trivially reset by requesting a new OTP (`resendOtp`, `auth.service.ts:607-664`, has no cooldown either). Matters because password login and OTP resend/verify are both open to unlimited automated guessing from a single IP with no throttle, CAPTCHA, or backoff.
- **Platform-admin OTP verification has no attempt cap at all.** `PlatformAdminAuthService.validateAndConsumeOtp` (`services/platform-admin-auth.service.ts:207-234`) calls `incrementAttempts` on failure but never checks `otpRecord.attempts_count` against any max before accepting a guess — unlike the near-identical logic in `AuthService.loginWithOtp` (`auth.service.ts:184-190`) and `verifyOtpAndResetPassword` (`auth.service.ts:570-574`), which both explicitly gate on `maxAttempts`. Matters because this is the highest-privilege account type in the system (`is_platform_admin`) and its OTP has strictly weaker brute-force protection than a regular employee's.
- **Insecure PRNG used to generate the refresh-token secret.** `Math.random().toString(36).substring(2)` is used for the raw refresh-token value in `token.service.ts:57` and `auth.service.ts:432` (also `utils/otp.util.ts:6` for OTP digits). `Math.random()` is not a CSPRNG and is a known-weak source for security tokens; `crypto.randomBytes`/`randomUUID` should be used instead. Matters because this value, once hashed, is the sole secret that lets a stolen/leaked refresh-JWT be distinguished from a revoked one — weak entropy here narrows the security margin of the whole refresh flow.
- **Hardcoded fallback JWT secrets.** `strategies/jwt.strategy.ts:29-30`, `strategies/jwt-refresh.strategy.ts:37-39`, and `services/token.service.ts:36-38,47-49` all fall back to the literal strings `'default-secret'` / `'default-refresh-secret'` if `auth.JWT_SECRET`/`auth.JWT_REFRESH_SECRET` config is missing. Matters because a misconfigured deployment (missing env var / config key) fails open to a publicly-known secret instead of failing to start, silently allowing anyone to forge valid access/refresh tokens.
- **No new-device/new-location detection despite capturing the data needed for it.** `device_fingerprint` is stored on every session (`session.entity.ts:27-28`, populated in `token.service.ts:63`) but is never read back or compared against a user's prior sessions anywhere in `auth.service.ts` — confirmed by inspecting every use of `deviceFingerprint`/`device_fingerprint` in the module. Matters because this is the standard signal used to trigger a "new device login" security email, and the data is already being collected and simply discarded.
- **`updatePassword()` invalidates all sessions but sends no notification.** `auth.service.ts:465-501` calls `sessionRepository.deleteAllUserSessions(userId)` (line 498) but never calls `notificationService.send(...)`. `verifyOtpAndResetPassword()` (`auth.service.ts:551-603`) has the identical gap. Matters because a user (or attacker) changing a password should always trigger an out-of-band alert to the account owner; today an attacker who compromises a session and changes the password leaves the real owner with no signal that it happened.
- **Password-change/reset and app-switch events are not audited.** Confirmed by inspecting every `logAuthEvent`/`auditService.log` call site in `auth.service.ts` — only `LOGIN` (lines 123, 222), `LOGIN_FAILED` (line 115), and `LOGOUT` (line 404) are logged; `updatePassword`, `forgotPassword`, `verifyOtpAndResetPassword`, `resendOtp`, and `switchApp` have none, and `AuditAction` enum has no `PASSWORD_CHANGE`/`PASSWORD_RESET` value to log even if wired up. Matters because these are exactly the events a security/compliance audit trail exists to capture.
- **No test files found.** `find` over `modules/auth` for `*.spec.ts`/`*.test.ts` returns nothing — the module with the highest blast radius in the app (every authenticated request depends on it) has zero automated test coverage.

## 5. Top 3 Priorities

1. **Add rate limiting/lockout to `login`, `login/verify-otp`, `verify-otp`, and `resend-otp`, and add an attempts cap to `PlatformAdminAuthService.validateAndConsumeOtp`.** This is the largest live exposure — password login and platform-admin OTP verification can be brute-forced today with zero friction.
2. **Notify + audit on password change/reset, and use the already-captured `device_fingerprint` to detect and alert on new-device logins.** Three related, cheap fixes (wire `NotificationService`/`IAuditService` into `updatePassword`/`verifyOtpAndResetPassword`, add a `PASSWORD_CHANGE` audit action, compare fingerprint against the user's recent sessions) close the biggest "silent account takeover" blind spot.
3. **Replace `Math.random()` with a CSPRNG for refresh-token/OTP generation, and remove the hardcoded `'default-secret'`/`'default-refresh-secret'` JWT fallbacks (fail closed instead of failing open).** Low-effort, high-value hardening of the core token-issuance primitives that everything else in the module trusts.
