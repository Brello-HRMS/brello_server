# Module: Departments

## 1. Purpose & Current Usage

- **Entity**: `Department` (`brello_server/src/modules/departments/entities/department.entity.ts`) extends `BaseEntity` (`brello_server/src/common/entities/base.entity.ts`), which supplies `code`, `status`, `description`, `enterprise_id`, `organization_id`, `created_at`/`updated_at`, `modified_by`/`modified_at`, `deleted_by`/`deleted_at`. The module's own file adds only `name`, `icon`, `is_deleted`, `is_default`. A unique index covers `(organization_id, code)`. Unlike the sibling `Designation` entity, `Department` has no `@ManyToOne` relation object to `Organization` — just the raw inherited `organization_id` column — and no self-referential parent field, so there is no hierarchy concept at all.
- **Controller** (`controllers/department.controller.ts`): standard CRUD at `/departments`, guarded only by `JwtAuthGuard`. Create/Update/Delete are wrapped in `@AuditLog(...)`.
- **Service** (`services/department.service.ts`): resolves `organization_id` from the JWT-derived `LoggedInUser`, enforces `code` uniqueness per org on create, soft-deletes via repository, and pushes index/remove calls to `SearchIndexingService`. `findAll` additionally fetches up to 3 member avatars per department and merges them into each row as `memberAvatars`.
- **Repository** (`repositories/department.repository.ts`): TypeORM query builder methods — `create`, `getListingQueryBuilder`, `findAllByOrg`, `findOneByOrg`, `findByCode`, `update`, `softDelete` (sets `status = INACTIVE`, `is_deleted = true`), `countByOrg`.
- **Who calls in today**:
  - `brello_server/src/modules/organization/services/organization.service.ts:249-263` — `setupCompany()` Step 8 clones all `Department` rows where `is_default = true AND is_deleted = false` (these platform templates have `organization_id = NULL`, created via `PlatformDepartmentService`, which never sets `organization_id`/`enterprise_id`) into the new org, writing `name`, `code`, `description`, `icon`, `status`, `enterprise_id`, `organization_id`, `is_default: false`.
  - `brello_server/src/modules/user/entities/user.entity.ts:50-54` — `User.department_id` / `ManyToOne(() => Department)`; consumed by `modules/user/repositories/department.repository.ts` (a second, separate `UserDepartmentRepository` for map/unmap and listing queries), `modules/user/services/employee.service.ts:204,446,543` (writes `department_id` on create/update with no existence check), and `modules/user/services/user.service.ts:258-281` (`mapDepartmentAndDesignation` — also no existence check, see Gaps).
  - `brello_server/src/modules/letter-management/shared/services/variable-resolver.service.ts:54` — resolves the `{{department}}` letter variable as `employee.department?.name ?? ''`; the relation is eager-loaded via `modules/user/repositories/user.repository.ts:35` (`relations: [..., 'department', ...]`), registered in `modules/letter-management/shared/registry/variable-registry.ts:68-72`.
  - `brello_server/src/modules/global-search/services/search-indexing.service.ts:134-156` — indexes/removes department documents (route `/organisation/departments/:id`, permission `department.read`).
  - `brello_server/src/modules/org-setup/org-setup.service.ts:32-33,61,72` — counts non-deleted departments per org directly via `@InjectRepository(Department)` (bypassing `DepartmentService`/`DepartmentRepository`) to compute onboarding-checklist completion.
  - `brello_server/src/modules/platform/services/platform-department.service.ts` + `platform-department.controller.ts` — a **completely separate** platform-admin CRUD surface (`/platform-admin/departments`) for the `is_default = true` template rows, injecting `Repository<Department>` directly rather than going through `DepartmentService`/`DepartmentRepository`.
  - `brello_server/src/modules/attendance/services/rule-assignment.service.ts` and `dto/assign-departments.dto.ts` — assigns attendance rules to a list of `department_ids` with no check that those ids exist or belong to the caller's org.
  - `brello_server/src/modules/announcement/services/announcement.service.ts:252-253` — announcement targeting stores raw `department_ids` with no referential check either.
  - Frontend: `brello_webapp/src/pages/department/DepartmentPage.tsx`, `DepartmentDetailPage.tsx`, `features/department/**`, `features/platform/departments/**` all consume the two API surfaces above.
- **Dead/unused code**:
  - `DepartmentResponseDto` (`dto/department-response.dto.ts`) is defined but never imported/used anywhere — the controller returns raw `Department` entities straight from TypeORM (`controllers/department.controller.ts:41,61,72`).
  - `DepartmentRepository.findAllByOrg` (`repositories/department.repository.ts:26-54`) and `.countByOrg` (`:97-104`) are defined but never called; `DepartmentService.findAll` builds its own query via `getListingQueryBuilder` + the shared `ListingHelper` instead.
  - `UserDepartmentRepository.findAvatarsByDepartmentIds` (`modules/user/repositories/department.repository.ts:64-84`) is commented out dead code; the real implementation actually used by `DepartmentService.findAll` lives on the unrelated `UserRepository` (`modules/user/repositories/user.repository.ts:106-128`), leaving two near-duplicate implementations of the same query in two files, one of them commented-out and forgotten.

## 2. Intended / Ideal Usage

An org admin manages a flat list of departments (create/rename/deactivate), employees get assigned a `department_id`, and that assignment flows into HR listing/filtering, attendance rule assignment, announcement targeting, and offer-letter generation. Platform admins maintain a curated set of `is_default` template departments that get cloned into every new org at signup so orgs start with a sensible baseline. Deleting or deactivating a department that still has active employees should be blocked (or should force a reassignment step) at the API layer, not just in the UI, and any module that stores a `department_id` (designations, attendance rule assignments, announcements) should validate it against the department table before persisting.

## 3. Cross-Module Connections

- **Depends on**: `UserModule` (for the avatar-lookup query in `findAll`), `GlobalSearchModule`/`SearchIndexingService` (search indexing), `AuditModule` (audit logging via decorator + `AuditContextService`).
- **Depended on by**: `user` module (`User.department_id`, `UserDepartmentRepository`, `employee.service.ts`, `user.service.ts`), `organization` module (clone-on-signup in `setupCompany()`), `designations` module (`Designation.department_id`, optional linkage), `letter-management` module (`{{department}}` variable resolver), `attendance` module (rule-to-department assignment), `announcement` module (department-targeted announcements), `org-setup` module (onboarding checklist), `global-search` module (indexing target), `platform` module (duplicate CRUD over the same table).
- **Missing/expected connections**:
  - No referential-integrity check on delete or deactivate: `DepartmentService.remove()` (`services/department.service.ts:155-175`) soft-deletes unconditionally — the check is literally written out as a commented-out `TODO (Phase 2)` block (lines 161-167) and never implemented.
  - No hierarchy/parent-department concept on the entity at all (no `parent_department_id`), even though the sibling `Designation` entity has a `department_id` FK pointing *into* this module, implying departments were expected to be a structural anchor.
  - No validation anywhere (designations, employee assignment, attendance rule assignment, announcement targeting) that a `department_id` supplied by a client actually exists, belongs to the same org, or is still active before being persisted.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **Platform admin bypasses the module entirely.** `PlatformDepartmentService` (`modules/platform/services/platform-department.service.ts:1-67`) injects `@InjectRepository(Department)` directly and re-implements code-uppercasing, uniqueness-checking, and delete logic instead of using `DepartmentRepository`/`DepartmentService`. `DepartmentModule` exports `DepartmentService` (`department.module.ts:18`) specifically so other modules can inject it, but the platform surface ignores that and duplicates the logic, so the two paths can (and already do — see Coding below) drift apart.
- **No relation object to `Organization`.** `Department` relies solely on the bare `organization_id: string` column inherited from `BaseEntity` (`common/entities/base.entity.ts:16-17`) with no `@ManyToOne(() => Organization)` (contrast with `Designation.organization` in `designations/entities/designation.entity.ts:18-20`), so there is no way to eager-load or navigate to the owning org from a `Department` instance, and no DB-level FK to catch an orphaned `organization_id`.
- **Two independent `department_id`-lookup repositories for `User`.** `UserDepartmentRepository` (`modules/user/repositories/department.repository.ts`) and query logic embedded directly in `UserRepository` (`modules/user/repositories/user.repository.ts:106-128`) both build department-scoped user queries; `DepartmentService.findAll` calls into `UserRepository`, not the purpose-built `UserDepartmentRepository`, so the "department-aware user repository" abstraction isn't actually used for the one place that most needs it.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

- **Deletion/deactivation of a department with active employees is not enforced server-side.** `DepartmentService.remove()` (`services/department.service.ts:161-167`) has the guard written as a comment (`// TODO (Phase 2): Block deletion if active employees are assigned to this department`) and never executes it — any caller hitting `DELETE /departments/:id` directly (bypassing the UI) can soft-delete a department while `User.department_id` rows still point at it, and `update()` (same file, no code path checks employee count before an INACTIVE status change either).
- **The employee-count guard that does exist lives only in the frontend, and is inconsistently applied.** `brello_webapp/src/pages/department/DepartmentPage.tsx:96-101` blocks *deactivate* by checking `dept.memberAvatars?.length > 0`, but `handleDeleteClick` (same file, ~line 105) opens the delete-confirmation modal unconditionally with no such check — so the one client-side safety net that exists doesn't even cover the delete action, and the backend has none for either action.
- **`memberAvatars` is capped at 3 and is being used as a proxy for "does this department have members".** `UserRepository.findAvatarsByDepartmentIds` (`modules/user/repositories/user.repository.ts:106-128`) intentionally caps the array at 3 avatars (`if (result[user.department_id].length < 3 && photo)`), yet the frontend displays `{selectedDepartment?.memberAvatars?.length} active employees` verbatim in the "Cannot Deactivate" alert (`brello_webapp/src/pages/department/DepartmentPage.tsx:310-312`) — for any department with more than 3 members the displayed count is capped/wrong, and there is no real `employeeCount` field anywhere in the response.
- **`DepartmentResponseDto` is unused dead code**, and because it's unused the controller returns raw `Department` entities (`controllers/department.controller.ts:41,61,72`), leaking internal columns (`is_deleted`, `is_default`, `enterprise_id`, `modified_by`, `deleted_by`, `deleted_at`) straight to API consumers.
- **Inconsistent soft-delete semantics between the two department code paths.** `DepartmentRepository.softDelete()` sets `status = INACTIVE, is_deleted = true` (`repositories/department.repository.ts:90-95`), while `PlatformDepartmentService.remove()` sets `status = DELETED, is_deleted = true` directly (`services/platform-department.service.ts:60-66`). Same conceptual operation, two different terminal `status` values depending on which controller was hit.
- **`deleted_by`/`deleted_at` columns are never populated.** Both soft-delete paths set `is_deleted`/`status` but never touch the `deleted_by`/`deleted_at` columns that `BaseEntity` provides (`common/entities/base.entity.ts:44-48`), so there is no record of who deleted a department or when, despite the schema having room for it.
- **`enterprise_id` is written but never read.** `DepartmentService.create` sets `enterprise_id: user.enterpriseId` (`services/department.service.ts:60`), but every repository read method (`findOneByOrg`, `findByCode`, `findAllByOrg`, `countByOrg`) scopes only by `organization_id`, never `enterprise_id` — the field is captured for indexing purposes only and otherwise inert in this module's own queries.

### Technical (performance, security, scalability, test coverage, observability/logging)

- **Cross-tenant leak risk via the shared `ListingHelper`.** `DepartmentService.findAll` (`services/department.service.ts:74-107`) delegates org-scoping to `ListingHelper.apply` (`common/utils/listing.helper.ts:22-27`), which only adds the `organization_id` filter `if (!loggedInUser.isPlatformAdmin && loggedInUser.organizationId)`. `DepartmentController` (`controllers/department.controller.ts:27-28`) is guarded only by `JwtAuthGuard` (no role/org guard), so a JWT flagged `isPlatformAdmin: true` hitting the ordinary `GET /departments` endpoint receives departments across every organization, not just its own.
- **No test coverage found.** No `*.spec.ts` files exist anywhere under `brello_server/src/modules/departments/` for the entity, service, repository, or controller.
- **No logging/audit signal on the clone-on-signup path.** `organization.service.ts:249-263` clones default departments inside `setupCompany()` but there is no dedicated log line indicating how many departments were cloned or if zero defaults were found — a silent zero-department org would only surface later via `org-setup.service.ts`'s `DEPARTMENTS` checklist flag, with no direct link back to the cause.

## 5. Top 3 Priorities

1. **Implement the referential-integrity check that is already sketched as a TODO.** `DepartmentService.remove()` (`services/department.service.ts:161-167`) needs a real active-employee check before soft-deleting (and ideally before deactivating too), and it needs to live server-side — right now the only guard is in the frontend, and it doesn't even cover the delete button, so any direct API call bypasses it entirely.
2. **Consolidate `PlatformDepartmentService` onto `DepartmentService`/`DepartmentRepository`.** Two independent implementations of code uniqueness and soft-delete already disagree on terminal `status` (`INACTIVE` vs `DELETED`); left alone this will keep drifting and complicate any future query that filters departments by status.
3. **Close the `isPlatformAdmin` org-scoping gap in `ListingHelper`, or add an explicit org guard to `DepartmentController`.** As written, a platform-admin-flagged JWT can call the tenant-facing `GET /departments` and read every organization's departments, which is a real multi-tenant data exposure, not just a hypothetical one.
