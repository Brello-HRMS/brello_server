# Module: Designations

## 1. Purpose & Current Usage

- **Entity**: `Designation` (`brello_server/src/modules/designations/entities/designation.entity.ts`) extends `BaseEntity` (`brello_server/src/common/entities/base.entity.ts`), which supplies `code`, `status`, `description`, `enterprise_id`, `organization_id`, `created_at`/`updated_at`, `modified_by`/`modified_at`, `deleted_by`/`deleted_at`. The module's own file adds `org_id` (nullable — null = platform-default template), `department_id`, `title`, `is_deleted`, `is_default`. A unique index `IDX_DESIGNATION_ORG_CODE` covers `(org_id, code)`.
- **Controller** (`controllers/designation.controller.ts`): standard CRUD at `/designations`, guarded only by `JwtAuthGuard`, org id pulled from JWT via `@CurrentUser`. All mutating routes are wrapped in `@AuditLog(...)`.
- **Service** (`services/designation.service.ts`): org-existence check via `OrganizationService.findOne`, uppercases + uniqueness-checks `code` on create, strips `code` from updates (immutability rule), soft-deletes via repository, and pushes index/remove calls to `SearchIndexingService` (global search).
- **Repository** (`repositories/designation.repository.ts`): plain TypeORM query builder methods — `findAllByOrg` (with search/status/department filters), `findById`, `findByOrgAndCode`, `update`, `softDelete` (sets `status = INACTIVE`, `is_deleted = true`), `existsById`.
- **Who calls in today**:
  - `brello_server/src/modules/organization/services/organization.service.ts:268-285` — `setupCompany()` Step 9 clones all `Designation` rows where `is_default = true AND is_deleted = false` (org_id IS NULL) into the new org, writing `org_id`, `organization_id`, `enterprise_id`, `title`, `code`, `description`, `status`; it does **not** copy `department_id`.
  - `brello_server/src/modules/user/entities/user.entity.ts:57-62` — `User.designation_id` / `ManyToOne(() => Designation)`; used throughout the employee module (`modules/user/repositories/designation.repository.ts` — a second, separate `UserDesignationRepository` for map/unmap and listing queries; `modules/user/services/employee.service.ts:205,257,373-375,393,447,544`; `modules/user/services/user.service.ts:258-307` map/unmap endpoints; `modules/user/controllers/user.controller.ts:68-85`).
  - `brello_server/src/modules/letter-management/shared/services/variable-resolver.service.ts:53` — resolves the `{{designation}}` letter variable as `employee.designation?.title ?? ''`, a live-loaded relation (`modules/user/repositories/user.repository.ts:35` eager-loads `relations: ['designation', ...]`), registered in `modules/letter-management/shared/registry/variable-registry.ts:59-62` and used across every starter letter template in `modules/letter-management/shared/seed/starter-letter-content.ts`.
  - `brello_server/src/modules/global-search/services/search-indexing.service.ts:158-182` — indexes/removes designation documents (route `/organisation/designations/:id`, permission `designation.read`).
  - `brello_server/src/modules/leave-balance/repositories/leave-balance.repository.ts:161,259,275` — raw joins against the `designations` table for reporting (`ds.title` as `designation_name`).
  - `brello_server/src/modules/platform/services/platform-designation.service.ts` + `platform-designation.controller.ts` — a **completely separate** platform-admin CRUD surface for the `is_default = true` template rows, injecting `Repository<Designation>` directly rather than going through `DesignationService`/`DesignationRepository`.
- **Dead/unused**: `DesignationResponseDto` (`dto/designation-response.dto.ts`) is defined but never imported/used anywhere in the codebase — the controller returns raw `Designation` entities straight from TypeORM.

## 2. Intended / Ideal Usage

An org admin creates designations (optionally under a department, optionally with a seniority `level`), assigns them to employees, and those designations flow into HR reporting, letters, and org-chart/hierarchy views. Platform admins maintain a curated set of `is_default` template designations that get cloned into every new org at signup so orgs start with a sensible baseline. Deleting a designation should be blocked or should cleanly detach dependent employees, and all designation reads/writes (org-side and platform-side) should go through one service so validation rules (code uniqueness, org scoping, active/inactive status) are enforced consistently in exactly one place.

## 3. Cross-Module Connections

- **Depends on**: `OrganizationModule`/`OrganizationService` (org existence check), `GlobalSearchModule`/`SearchIndexingService` (search indexing), `AuditModule` (audit logging via decorator + `AuditContextService`).
- **Depended on by**: `user` module (`User.designation_id`, `UserDesignationRepository`, `employee.service.ts`, `user.service.ts`), `organization` module (clone-on-signup), `letter-management` module (variable resolver + starter templates), `leave-balance` module (reporting joins), `global-search` module (indexing target), `platform` module (duplicate CRUD over the same table).
- **Missing/expected connections**:
  - No referential-integrity check on delete: `DesignationService.remove()` (`services/designation.service.ts:141-151`) soft-deletes unconditionally with no check for `User` rows still pointing at the `designation_id`, and `User.designation` (`modules/user/entities/user.entity.ts:60-61`) has no `onDelete` behavior configured.
  - No hierarchy/seniority model: `CreateDesignationDto.level` (`dto/create-designation.dto.ts:39-43`) and `DesignationResponseDto.level` (`dto/designation-response.dto.ts:28-29`) reference a `level` field that does not exist as a column on the `Designation` entity at all — see Gaps below.
  - No validation that `department_id` on a designation, or `designation_id` assigned to an employee, actually belongs to the same org (or exists at all).

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **Platform admin bypasses the module entirely.** `PlatformDesignationService` (`modules/platform/services/platform-designation.service.ts:1-73`) injects `@InjectRepository(Designation)` directly and re-implements code-uppercasing, uniqueness-checking, and delete logic instead of using `DesignationRepository`/`DesignationService`. This duplicates business rules in two places that can (and already do, see Coding below) drift apart, and defeats the purpose of `DesignationModule` exporting `DesignationService` "for injection in other modules" (`designation.module.ts:20-21`).
- **Duplicate org-scope columns on the entity.** `Designation` defines its own `org_id` (`entities/designation.entity.ts:15-16`, used everywhere in this module's queries and the unique index) alongside the inherited `BaseEntity.organization_id` (`common/entities/base.entity.ts:16-17`), which is also populated (by `organization.service.ts:279` and `platform-designation.service.ts` implicitly leaves it null) but never read by any query in this module. Two columns carry the same meaning with inconsistent population, which is a landmine for future joins/filters that assume one or the other.
- **`code`/`status`/`description` live on `BaseEntity`, but the field's real semantics are module-specific.** E.g. `status` is reused for both "Active/Inactive" business status and a workaround delete signal (see Coding below); nothing in `BaseEntity` documents that overload, and other modules using `BaseEntity` may not share this convention, making it a fragile shared abstraction.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

- **`level` is a dead field end-to-end.** `CreateDesignationDto.level` and `UpdateDesignationDto` (via `PartialType`) accept and validate a `level` string (`dto/create-designation.dto.ts:39-43`), and `DesignationResponseDto.level` claims to expose it (`dto/designation-response.dto.ts:28-29`), but `Designation` entity has no `level` column — TypeORM silently drops it on `create`/`save` (`repositories/designation.repository.ts:22-25,85-91`). Any client sending `level` gets no error and no persistence; this is the concrete manifestation of "no hierarchy/seniority levels" in the module. Frontend (`brello_webapp/src/features/designation/**`) doesn't even send `level`, confirming it was never wired up end-to-end.
- **`DesignationResponseDto` is unused dead code**, and because it's unused the controller returns raw `Designation` entities (`controllers/designation.controller.ts:59,69,82`), leaking internal columns (`is_deleted`, `is_default`, `enterprise_id`, `modified_by`, `deleted_by`, `deleted_at`, the redundant `organization_id`) straight to API consumers.
- **Inconsistent soft-delete semantics between the two designation code paths.** `DesignationService.remove()` sets `status = INACTIVE, is_deleted = true` (`repositories/designation.repository.ts:97-99`), while `PlatformDesignationService.remove()` sets `status = DELETED, is_deleted = true` (`services/platform-designation.service.ts:66-72`) directly. Same conceptual operation, two different terminal `status` values depending on which controller was hit — anything filtering by `status` downstream (e.g. `FindDesignationsDto.status`, `find-designations.dto.ts:16-19`) will treat org-deleted and platform-deleted designations differently for no principled reason.
- **No org/existence validation for `department_id` on create/update.** `DesignationService.create`/`update` never verify `dto.department_id` belongs to the same org (or exists at all) — `services/designation.service.ts:41-71,115-139` only validates the org itself and code uniqueness.
- **No validation when assigning a designation to an employee.** `EmployeeService` writes `dto.designationId` straight to `User.designation_id` with no existence/org/active check (`modules/user/services/employee.service.ts:205,447,544`), and `UserService.mapDepartmentAndDesignation` → `UserDesignationRepository.mapDesignation` does the same (`modules/user/services/user.service.ts:270-271`, `modules/user/repositories/designation.repository.ts:53-56`). A caller can attach a nonexistent, wrong-org, or already-soft-deleted designation id to an employee undetected.

### Technical (performance, security, scalability, test coverage, observability/logging)

- **Race condition on platform-default code uniqueness.** The unique index `IDX_DESIGNATION_ORG_CODE` is on `(org_id, code)` (`entities/designation.entity.ts:12`), but Postgres treats `NULL org_id` values as distinct from each other in a unique index, so it provides no protection for platform-default rows (`org_id IS NULL`). `PlatformDesignationService.create` only guards this with an app-level `findOne` + `save` (`services/platform-designation.service.ts:36-56`), which is a check-then-act race under concurrent requests.
- **No test coverage found.** No `*.spec.ts` files exist anywhere under `brello_server/src/modules/designations/` for entity, service, repository, or controller — a module three other modules (`user`, `organization`, `letter-management`) depend on has zero automated tests.
- **No logging/audit signal on the clone-on-signup path.** `organization.service.ts:268-285` clones designations inside `setupCompany()` but only logs a generic "Company setup completed successfully" at the end (`organization.service.ts:294-296`); if the default-designation clone loop silently produces zero rows (e.g., no defaults seeded), there is no dedicated log line to surface that during onboarding debugging.

## 5. Top 3 Priorities

1. **Add a referential-integrity check (or `onDelete` policy) before/when soft-deleting a designation.** Today `DesignationService.remove()` (`services/designation.service.ts:141-151`) will happily soft-delete a designation still referenced by active employees, and letters/reports will keep resolving its `title` indefinitely since the row still exists — this is a data-integrity risk with no guardrail at all.
2. **Consolidate `PlatformDesignationService` onto `DesignationService`/`DesignationRepository`.** Two independent implementations of code uniqueness and soft-delete already disagree on terminal `status` (`INACTIVE` vs `DELETED`); left alone this will keep drifting and complicate any future query that filters designations by status.
3. **Either implement `level`/hierarchy properly (add the column, wire it through repository + response DTO + frontend) or remove it from the DTOs.** Right now the API silently accepts and discards a field it advertises as supported, which is a confusing, hard-to-detect bug for any client relying on it — and it's the concrete gap behind "no seniority/hierarchy" in this module.
