# Module: Document

## 1. Purpose & Current Usage

The module lives at `brello_server/src/modules/document/` and is the shared file-storage abstraction for the whole backend — a single `Document` table plus a thin S3 wrapper, meant to be the one place every other module goes through to store/retrieve a file.

- **Entity**: `entities/document.entity.ts` — one table (`document`) covering metadata (`original_name`, `file_name`, `extension`, `mime_type`, `size`), storage location (`storage_provider`, `bucket`, `object_key`), a `folder_type` enum used to derive the S3 key layout, `is_public`, `checksum`, `version`, and an optional `bytea file_data` column (`select: false`) for DB-backed storage. Composite indexes on tenant (`enterprise_id`+`organization_id`) and `employee_id`, unique constraint on `(enterprise_id, organization_id, object_key)`.
- **Enums**: `StorageProvider` (S3, LOCAL, GCS, DATABASE — only S3 and DATABASE are actually implemented) and `FolderType` (9 values driving the S3 path scheme in `generateObjectKey`).
- **Services**: `DocumentService` — orchestrates the two-phase upload flow (`generateUploadUrl` → client PUTs to S3 or `uploadFileContent` for DB mode → `confirmUpload`), plus `uploadDocument` (single-call server-side upload used by backend modules), `getSignedUrl`, `findOne`, `getFileData`, `remove` (soft delete), and `buildViewUrl`. `StorageService` — raw AWS SDK v3 wrapper: `generatePresignedUploadUrl`, `generatePresignedDownloadUrl`, `uploadFile`. **No delete method exists on `StorageService` at all** (confirmed: zero occurrences of `DeleteObjectCommand` anywhere under `modules/`).
- **Controllers**: `DocumentController` (`/documents/upload-url`, `/documents/:id/confirm`, `/documents/:id/upload`, `/documents/:id`, `/documents/:id/signed-url`, `/documents/:id/download`, `DELETE /documents/:id`) — all behind `JwtAuthGuard` only (no tenant/ownership guard). `DocumentViewController` (`/documents/:id/view`) — public, HMAC-signature-gated (for `<img>` tags that can't send an Authorization header).
- **Repository**: `DocumentRepository` — plain CRUD wrapper (`create`, `findById`, `findByIdWithContent`, `update`, `softDelete`); `findById` filters only on `status != DELETED`, no tenant scoping.
- **Utils**: `document-signature.util.ts` — HMAC-SHA256 signing/verification for the DB-view route (15-min default TTL).

**Who calls it today** (repo-wide grep for `DocumentService`/`documentService`/`StorageService`, `modules/document/enums/document.enum.ts` `FolderType` usage, and frontend `folderType` call sites):
- `modules/letter-management/issued-letters/services/issued-letter.service.ts:46,131,200,216,282` — uploads generated letter PDFs (`FolderType.LETTER_DOCUMENT`), fetches signed download URLs.
- `modules/letter-management/signatories/services/signatory.service.ts:24,35,38` — uploads signature images (`FolderType.LETTER_SIGNATURE`).
- `modules/user/services/employee.service.ts:59,90,110,1062,1065` — uploads employee photos (`FolderType.EMPLOYEE_IMAGE`), calls `buildViewUrl`.
- `modules/organization/services/organization-profile.service.ts:15,27,28,48,146` — validates `logo_id` via `documentService.findOne` on create/update (does not itself upload).
- `modules/project/services/project.service.ts:20,33,235,292` — resolves a `documentId` into a project contract, and removes the `Document` row when a contract is deleted.
- `modules/billing/services/invoice-pdf.service.ts:3,12` and `modules/payroll/services/payslip-pdf.service.ts:4,28` — inject `StorageService` directly and bypass `DocumentService`/the `Document` entity entirely (see §3/§4 Structural).
- Frontend (`brello_webapp/src/api/documents.ts`) calls the generic `/documents/upload-url` → `/documents/:id/upload` → `/documents/:id/confirm` flow directly for **Feedback attachments** (`SubmitFeedbackModal.tsx:124`, `folderType: 'FEEDBACK_ATTACHMENT'`), **Reimbursement receipts** (`AddReimbursementModal.tsx:110`, `folderType: 'REIMBURSEMENT_DOCUMENT'`), org logo (`OrgProfileHeader.tsx`), project contracts (`ContractTab.tsx`), and employee document steps (`DocumentsStep.tsx`) — none of these have a dedicated backend service wrapper, they go straight through the generic controller.

**Dead/unused parts**:
- `StorageProvider.LOCAL` and `StorageProvider.GCS` are declared but have no implementation anywhere — `getStorageProvider()` only branches on S3 vs. DATABASE (`services/document.service.ts:41-48`).
- `Document.is_public` (`entities/document.entity.ts:56`) is read once in `findOne()` (`services/document.service.ts:299`) but never written anywhere and never used to gate any access check — it's always `false` and enforces nothing.
- `Document.checksum` and `Document.version` columns exist on the entity but are never set or read anywhere in the module (no integrity verification, no versioning logic despite the column existing).
- `document.module.ts:23` exports `DocumentRepository` alongside `DocumentService`, but a repo-wide grep found zero external modules importing `DocumentRepository` directly — it's exported but nothing uses that escape hatch today (still a layering risk, see Gaps).
- A separate, parallel `UserDocument` entity/table (`modules/user/repositories/user-document.repository.ts`, `modules/user/entities/user-document.entity.ts`) exists alongside the shared `document` table and is unrelated to it — `employee.service.ts` uses both `documentRepository` (this local `UserDocumentRepository`) and `documentService` (the shared `DocumentService`) in the same class, which is confusing (see Gaps).

## 2. Intended / Ideal Usage

A shared document/storage service used by payroll payslips, signatures, issued letters, and general attachments across a multi-tenant HR system should, at minimum:
- Scope every read/write/delete by the caller's tenant (`enterprise_id`/`organization_id`, and `employee_id` where relevant) so one org's authenticated user can never fetch, download, or delete another org's files by ID.
- Enforce file-type and size limits server-side (both at the presigned-URL/validation layer and at the multipart-upload layer), not just document metadata fields that are trusted at face value.
- Issue short-lived, purpose-scoped signed URLs (upload vs. download) and never return a permanent public S3 URL for private tenant data.
- Actually delete the underlying S3 object (or DB blob) when the owning `Document` row is soft/hard-deleted, and clean up the previous file when a reference is replaced (e.g., logo/photo update) — otherwise storage cost and blast radius grow forever.
- Confirm the upload actually happened (e.g., a HEAD/existence check against S3) before flipping status to ACTIVE, and validate uploaded bytes against the declared checksum/size.
- Provide one consistent code path for every consumer, rather than some modules going through the `Document` entity+service and others hitting `StorageService` directly with hand-rolled keys.

## 3. Cross-Module Connections

**Depends on**:
- `@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner` (`services/storage.service.ts:3-8`) — S3 client, presigned PUT/GET URLs.
- `@nestjs/config` (`ConfigService`) — `storage.provider`, `aws.s3.bucketName`, `aws.region`, `auth.JWT_SECRET`. Note `src/core/properties/dev.properties.yaml` has no `storage:` key at all, so dev defaults to `StorageProvider.DATABASE` (bytea column) per `document.service.ts:41-48`, while `sample.properties.yaml:56-57` sets `storage.provider: 'S3'` for prod-like configs — the two backends have materially different security/expiry characteristics (see Gaps).
- `modules/enterprise` (`EnterpriseService.findOneById`) and `modules/organization` (`OrganizationService.findOne`, via `forwardRef`) — for tenant validation and slugging the S3 key prefix.
- `modules/audit` (`AuditContextService`) — captures pre-value on delete for the audit diff.

**Depended on by** (from the grep in §1): Letter Management (`issued-letters`, `signatories`), User/Employee (`employee.service.ts` photo upload), Organization (`organization-profile.service.ts` logo validation), Project (`project.service.ts` contracts), and — indirectly, via the generic `/documents/*` HTTP endpoints only, with no backend service wrapper — Feedback attachments and Reimbursement receipts. Billing (`invoice-pdf.service.ts`) and Payroll (`payslip-pdf.service.ts`) depend only on the low-level `StorageService`, not on `DocumentService` or the `Document` entity.

**Missing/expected connections**:
- **No orphan cleanup of S3 objects anywhere.** `StorageService` has no `deleteFile`/`DeleteObjectCommand` method at all (confirmed by grep), so `DocumentService.remove()` (`services/document.service.ts:418-435`) only flips `status` to `DELETED` in Postgres — the S3 object (or `file_data` bytea row, which isn't cleared either) is retained forever. `project.service.ts:292` explicitly calls `documentService.remove()` when a contract is deleted, but that call can never actually free the underlying storage.
- **No cleanup when a reference is replaced.** `employee.service.ts` `setEmployeePhoto` (`~line 1055-1069`) uploads a new photo and overwrites `profile.photo_id` with the new document id, but the previous photo's `Document` row/S3 object is never deleted or marked DELETED — every photo change leaks the old file. Same pattern in `organization-profile.service.ts:update()` (`~line 145-146`) for `logo_id`.
- **Signatories have no delete path at all** — `SignatoryService` has no `remove`/`delete` method (confirmed by grep), so even the soft-delete-only cleanup that exists elsewhere isn't available for signature images.
- **Billing/Payroll bypass the `Document` entity entirely.** `invoice-pdf.service.ts:18` and `payslip-pdf.service.ts:76` call `storage.uploadFile()` directly with hand-built keys (`invoices/{orgId}/{invoiceNumber}.pdf`, `payslips/{orgId}/{runId}/{itemId}.pdf`) and store the raw S3 key on their own entities (`invoice.pdf_s3_key`, `payroll_run_item.payslip_pdf_key`) — these files have no `Document` row, so they're invisible to any future "list all documents for this tenant" or unified retention/cleanup job, and — because `StorageService.uploadFile` always talks to S3 regardless of the configured `storage.provider` — they always land in real S3 even when the rest of the app is running in `DATABASE` mode (e.g. local dev), which is an inconsistent, easy-to-miss behavior difference from every other consumer.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **Two divergent storage code paths with no shared invariants.** Billing and Payroll inject `StorageService` directly and never create a `Document` row (`modules/billing/services/invoice-pdf.service.ts:3,12,18`, `modules/payroll/services/payslip-pdf.service.ts:4,28,76`), while every other consumer goes through `DocumentService`. This means tenant scoping, soft-delete, and any future audit/cleanup logic built into `DocumentService`/`Document` simply doesn't apply to invoices or payslips — a second, un-abstracted storage convention living beside the "shared" one.
- **`DocumentModule` exports `DocumentRepository` directly** (`document.module.ts:23`), alongside `DocumentService` — this lets any future consumer skip the service's tenant/status logic and hit the table directly. Not yet exploited (grep found no external importers), but the module boundary doesn't prevent it.
- **A parallel, unrelated `UserDocument` table exists alongside `document`.** `employee.service.ts` mixes `this.documentRepository` (the local `UserDocumentRepository`, a completely separate table for onboarding doc metadata) and `this.documentService` (the shared module) in the same class (`services/employee.service.ts:86,90`); the developer's own comment at `employee.service.ts:906` — `// docId from parameter could be the user_document table ID, not S3 doc ID` — flags that even the author wasn't sure which ID space `removeDocument(id, docId)` operates in. This is exactly the kind of confusion a single shared document module is supposed to prevent.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

- **`confirmUpload` and `uploadFileContent` never verify the file actually arrived.** `services/document.service.ts:260-283` (`confirmUpload`) blindly sets `status: ACTIVE` after a client-reported "I uploaded it" call, with no HEAD/existence check against S3 and no comparison of actual bytes/size/checksum against what was declared in `generateUploadUrl`. A client can register a document, never upload anything (or upload something totally different), and still get it marked ACTIVE.
- **`is_public`, `checksum`, and `version` are vestigial.** `is_public` (`entities/document.entity.ts:56`) is surfaced in `findOne()`'s response but never set or enforced anywhere — dead column that looks like an access-control feature but isn't one. `checksum`/`version` are declared columns with zero read/write sites in the module.
- **Orphan photo/logo references** — see §3 above (`employee.service.ts` photo update, `organization-profile.service.ts` logo update) — each is also a coding-level miss: the update methods have the old id in hand (`profile.photo_id`, `existingProfile.logo_id`) at the moment they overwrite it, but never call `documentService.remove()` on it.

### Technical (performance, SECURITY, scalability, test coverage, observability/logging)

- **No tenant/ownership check on any document read/write/delete route — cross-tenant IDOR.** `DocumentController` (`controllers/document.controller.ts:28-29`) is guarded only by `JwtAuthGuard` (validates the JWT, nothing else); `DocumentService.findOne`, `getSignedUrl`, `getFileData`, `remove`, and `confirmUpload` (`services/document.service.ts:260-435`) all fetch by `id` alone via `DocumentRepository.findById` (`repositories/document.repository.ts:19-23`, filters only on `status != DELETED`) with **no comparison against the caller's `enterprise_id`/`organization_id`/`employee_id`**. Document ids are routinely returned to authenticated clients in other entities' JSON responses — e.g. `IssuedLetter.pdf_document_id` is serialized straight through by `GET /letter-management/issued-letters` (`modules/letter-management/issued-letters/controllers/issued-letter.controller.ts` `findAll`) — so any authenticated user in *any* tenant can call `GET /api/v1/documents/:id`, `GET /api/v1/documents/:id/signed-url`, `GET /api/v1/documents/:id/download`, or `DELETE /api/v1/documents/:id` with a document id from a different org and it will be served or deleted with no ownership check. This is the module's single biggest risk: callers like `IssuedLetterService`/`SignatoryService` do their own org-scoping *before* calling into `DocumentService` (e.g. `issued-letter.service.ts:findOne` scopes by `user.organizationId`), but the generic `/documents/*` endpoints provide no equivalent protection of their own, so the safety of the whole module currently depends entirely on every caller remembering to pre-check — and the generic controller is reachable directly by any authenticated frontend request.
- **No server-side file-size or mime-type limits anywhere in the upload path.** `GenerateUploadUrlDto` (`dto/generate-upload-url.dto.ts:36-38`) only requires `size` to be `@IsNumber()`/`@IsNotEmpty()` — no `@Max()` — and `mimeType` is a free-form `@IsString()` with no enum/whitelist. The multipart endpoint (`controllers/document.controller.ts:52-53`, `@UseInterceptors(FileInterceptor('file'))`) is configured with no `limits.fileSize` option (confirmed: none of the three `FileInterceptor(...)` call sites in the repo pass limits), so an authenticated user can upload arbitrarily large files through `/documents/:id/upload`, and for the S3 path the presigned `PutObjectCommand` (`services/storage.service.ts:45-49`) sets no `ContentLength`/policy condition, so the declared `size`/`mimeType` are never actually enforced against what gets stored.
- **`buildViewUrl` returns a permanent, unsigned public S3 URL for S3-backed documents.** `services/document.service.ts:56-72`: when `storage_provider === S3`, it returns `https://{bucket}.s3.{region}.amazonaws.com/{object_key}` directly — no signature, no expiry — while the DATABASE-mode branch correctly returns an HMAC-signed, time-limited view URL. This only produces a working (private) file if the bucket/object is itself private and some other layer intercepts it, since this bare virtual-hosted-style URL has no auth of its own; it's inconsistent with the signed-URL discipline used everywhere else in the module (`getSignedUrl`, `document-signature.util.ts`).
- **Zero test coverage.** No `*.spec.ts` files exist anywhere under `modules/document/` — `generateObjectKey`'s per-`FolderType` branching, the S3-vs-DATABASE provider fork, and the signature-verification utility are all unverified by automated tests.
- **No orphan-cleanup / retention job.** As detailed in §3, nothing ever issues a `DeleteObjectCommand`, so every soft-deleted or superseded document (letters, signatures, photos, logos, reimbursement receipts, feedback attachments) accumulates in storage indefinitely with no reconciliation mechanism.

## 5. Top 3 Priorities

1. **Add tenant/ownership scoping to every `DocumentService` read/write/delete path** (or a guard on `DocumentController` that checks `document.enterprise_id`/`organization_id`/`employee_id` against the caller) — this is a live cross-tenant IDOR on shared infrastructure serving payslips, signed letters, and signature images, and it's reachable through a generic, always-mounted controller rather than a narrow, hard-to-hit edge case.
2. **Implement `StorageService.deleteFile` and wire it into `remove()`, photo/logo replacement, and any future parent-entity deletion.** Right now "delete" is DB-only everywhere in the module, so storage cost and exposed-file count only ever grow, and a compliance request to actually purge a person's documents cannot currently be honored.
3. **Enforce file-size and mime-type limits server-side** (Multer `limits`, DTO validation, and S3 upload policy conditions) — today the upload surface accepts arbitrarily large files of any declared type with no verification against what's actually written, which is both a storage-cost and DoS exposure on a route hit by every module in the app.
