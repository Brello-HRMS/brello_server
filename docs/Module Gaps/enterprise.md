# Module: Enterprise

## 1. Purpose & Current Usage

- **Entities**: `Enterprise` (`entities/enterprise.entity.ts`) — a top-level tenant record (`name`, `domain` [unique], `logo`, `favicon`, `status`) that does **not** extend the shared `BaseEntity` (it hand-duplicates `status/code/description/timestamps/modified_by/deleted_by/deleted_at` instead). `EnterpriseApp` (`entities/enterprise-app.entity.ts`) is a join table mapping `enterprise_id` ↔ `app_id` (unique composite index), and does extend `BaseEntity`.
- **Service**: `EnterpriseService` (`services/enterprise.service.ts`) provides `create` (validates domain via DNS lookup + uniqueness, then seeds `enterprise_app` rows for *every* app in the system), `findAll`/`findOneById` (join in apps via `EnterpriseAppRepository`), `update`, `remove` (soft-delete cascaded to `enterprise_app`), and `assignAppsToEnterprise`.
- **Repositories**: `EnterpriseRepository` and `EnterpriseAppRepository` are thin CRUD wrappers, nothing unusual.
- **Controller**: `controllers/enterprise.controller.ts` exposes `/enterprises` CRUD, gated by `JwtAuthGuard` + `PlatformAdminGuard` — i.e. this is a **platform-admin-only** management surface (confirmed on the frontend: `brello_webapp/src/features/platform/enterprises/*`, linked from the sidebar at `/platform/enterprises`).
- **Real caller #1 (self-serve signup)**: `brello_server/src/modules/organization/services/organization.service.ts:105-115`, inside `setupCompany()`. This is the path every new tenant actually goes through, and it does **not** call `EnterpriseService.create()` at all — it does `manager.create(Enterprise, { name, domain: dto.subdomain, status: ACTIVE })` directly on the transaction's `EntityManager`. This bypasses all of `EnterpriseService`'s validation (DNS existence check, domain-uniqueness check) and its app-seeding step (no `enterprise_app` rows get created for a freshly self-served enterprise).
- **Real caller #2 (org listing)**: `organization.service.ts:340-349` (`findByEnterpriseId`) calls `EnterpriseService.findOneById()` purely as an existence check before listing an enterprise's organizations.
- **Real caller #3 (plan upgrade app-provisioning)**: `plan/services/organization-subscription.service.ts:44-64` calls `EnterpriseService.assignAppsToEnterprise()` when a subscription is created, to grant the enterprise access to the apps included in the new plan; consumed downstream by `app/services/app.service.ts:33` (`findAllForEnterprise`), which is the one place `enterprise_app` data actually gates something (which apps show up for the enterprise).
- **Dead/unused parts**: Every public method on `EnterpriseService` accepts a `user?: LoggedInUser` parameter (`create`, `findAll`, `findOneById`, `update`, `remove`, `assignAppsToEnterprise` — `enterprise.service.ts:32-157`) that is never read inside any method body. It exists only to match a call-site convention; there is no per-user authorization logic anywhere in this service.

## 2. Intended / Ideal Usage

- The schema clearly intends "one Enterprise → many Organizations" (`Organization.enterprise_id` + `@ManyToOne(() => Enterprise)` in `organization/entities/organization.entity.ts:14-16`), and `BaseEntity` (`common/entities/base.entity.ts`) puts `enterprise_id` on essentially every domain entity in the codebase (confirmed: 100+ files reference `enterprise_id`).
- However, there is **no cross-org, enterprise-level feature** built anywhere on top of this. Every place that reads domain data (attendance, payroll, announcements, reimbursements, leave, letters, etc.) filters by `organization_id`, and everywhere `enterprise_id` also appears in a `WHERE`/`andWhere` clause it is always paired with an `organization_id` filter on the *same* query (e.g. `announcement.repository.ts:41-42`, `payroll/repositories/employee-salary.repository.ts:154`, `attendance/repositories/rule-assignment.repository.ts:100-101`, `reimbursement.repository.ts:66`). Since one organization belongs to exactly one enterprise, the `enterprise_id` predicate is redundant in every one of these queries — it filters nothing that `organization_id` doesn't already filter.
- The only queries that filter by `enterprise_id` **alone** (i.e., genuinely enterprise-wide, cross-org) are: `EnterpriseAppRepository` (apps assigned to the enterprise) and `PlanRepository.findAll` (`plan/repositories/plan.repository.ts:22-35`, an optional `enterprise_id` filter for enterprise-specific custom plans). Neither is "cross-org reporting/data sharing" in the sense of aggregating business data (attendance, payroll, headcount, etc.) across an enterprise's multiple organizations — no such feature exists.
- Conclusion: the "multi-org-per-enterprise" concept is wired at the schema/FK level and used for exactly one real thing today — enterprise-wide app/plan provisioning — but functionally every business module still operates as if `organization_id` were the only tenant boundary.

## 3. Cross-Module Connections

- **Depends on**: `app` module (`AppRepository`, via `forwardRef` in `enterprise.module.ts:9,14` and `enterprise.service.ts:17,26-27` — used to seed all apps on enterprise creation), `audit` module (`AuditContextService`, `AuditLog` decorator/enums).
- **Depended on by**: `organization` module (creation/lookup, `organization.service.ts`), `plan` module (`organization-subscription.service.ts` → `assignAppsToEnterprise`), `app` module (`app.service.ts` → `findAllForEnterprise` reads `EnterpriseAppRepository` directly, not through `EnterpriseService`). Virtually every other entity in the codebase *carries* `enterprise_id` via `BaseEntity`, but per §2 above almost none of them actually query on it independently of `organization_id` — it is populated on write (from `LoggedInUser.enterpriseId` / `ctx.enterpriseId`) but is vestigial for read/filter purposes in modules like attendance, payroll, leave, announcement, reimbursement, letter-management, holiday, timesheet, etc.
- **Missing/expected connection**: `EnterpriseController`'s CRUD (create/update/remove) is never invoked by `OrganizationService.setupCompany()` — the two enterprise-creation code paths (platform-admin API vs. self-serve signup) are entirely disjoint, duplicating entity-construction logic with different validation guarantees (see Gaps below).

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)
- **Duplicated, divergent enterprise-creation logic.** `organization.service.ts:105-115` constructs and saves a bare `Enterprise` row directly via `manager.create/save`, completely bypassing `EnterpriseService.create()` — so the DNS-validation (`validateDomainExists`, `enterprise.service.ts:159-167`), domain-uniqueness check (`validateDomainUniqueness`, `enterprise.service.ts:169-176`), and automatic `enterprise_app` seeding (`enterprise.service.ts:44-52`) never run for self-serve tenants. This matters because it means the two on-ramps to "create an Enterprise" (platform-admin API vs. signup flow) produce inconsistent, differently-validated records, and self-serve enterprises silently start with zero `enterprise_app` mappings.
- **Enterprise entity does not extend `BaseEntity`.** `entities/enterprise.entity.ts:13-59` hand-copies `status/code/description/created_at/updated_at/modified_by/modified_at/deleted_by/deleted_at` instead of reusing the shared base every other entity uses; this is inconsistent with the rest of the codebase (e.g. its own child `EnterpriseApp` does extend `BaseEntity`) and is how the type bug below crept in.
- **No enterprise-level cross-org feature exists** despite `enterprise_id` being propagated onto essentially every entity — the abstraction is present in the schema but has no product surface built on it beyond app/plan provisioning (see §2). This is a missing-abstraction gap: there's no service/repository pattern anywhere for "aggregate X across all organizations under my enterprise."

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)
- **`modified_at` has the wrong column type.** `entities/enterprise.entity.ts:42-43` declares `@Column({ type: 'uuid', nullable: true }) modified_at: Date;` — copy-pasted from the `modified_by` (uuid) column above it instead of using `type: 'timestamp'` like `BaseEntity.modified_at` (`common/entities/base.entity.ts`). Any write to this column will fail or silently coerce incorrectly since a `Date` is being persisted into a `uuid` column.
- **`assignAppsToEnterprise` can throw on duplicate-key insert.** `EnterpriseAppRepository.bulkCreate` (`repositories/enterprise-app.repository.ts:14-25`) always does a plain create+save with no existence check, but `EnterpriseApp` has a unique index on `(enterprise_id, app_id)` (`entities/enterprise-app.entity.ts:5`). `OrganizationSubscriptionService.create()` (`plan/services/organization-subscription.service.ts:44-64`) calls this every time a subscription is created for *any* organization under an enterprise; if a second organization in the same enterprise subscribes to a plan sharing an app already mapped for that enterprise (the exact "multi-org-per-enterprise" scenario this module is supposed to serve), the insert violates the unique constraint and the request fails with an unhandled DB error instead of being idempotent.
- **Dead `user` parameter across the whole service.** `EnterpriseService.create/findAll/findOneById/update/remove/assignAppsToEnterprise` (`services/enterprise.service.ts:32-157`) all accept `user?: LoggedInUser` but never reference it — no ownership check, no filtering. It's plumbing with no behavior behind it.
- **`findByEnterpriseId` performs no ownership check.** `OrganizationService.findByEnterpriseId` (`organization/services/organization.service.ts:340-349`) takes a `user` argument but only uses it to call `EnterpriseService.findOneById(enterpriseId, user)` — which, per the previous bullet, ignores `user` entirely. The result: **any authenticated user of any organization can call `GET /organizations/enterprise/:enterpriseId` with an arbitrary enterprise UUID and receive that enterprise's full organization list** (`organization/controllers/organization.controller.ts:54-63`, guarded only by `JwtAuthGuard`, no check that `user.enterpriseId === enterpriseId`).

### Technical (performance, security, scalability, test coverage, observability/logging)
- **`enterprise_id` is not used as a tenant-isolation boundary anywhere — only `organization_id` is.** Every business-data query that includes an `enterprise_id` predicate also includes an `organization_id` predicate in the same query (`announcement/repositories/announcement.repository.ts:41-42,173-174`, `payroll/repositories/employee-salary.repository.ts:154,182`, `payroll/repositories/payroll-source.repository.ts:76`, `attendance/repositories/rule-assignment.repository.ts:100-101`), so `enterprise_id` never independently restricts access — actual isolation rests entirely on `organization_id`. Combined with the previous bug (`findByEnterpriseId` with no ownership check), this means enterprise as a security boundary is effectively unenforced anywhere in the codebase.
- **No test coverage.** There is no `*.spec.ts` file anywhere under `src/modules/enterprise/` — none of `EnterpriseService`'s validation branches (DNS check, uniqueness, transactional rollback, app-seeding) or the IDOR-shaped `findByEnterpriseId` path have any automated coverage.
- **DNS-based domain validation is a live external dependency with no fallback/observability.** `validateDomainExists` (`enterprise.service.ts:159-167`) does `dns.promises.resolve(domain)` synchronously inside the request path with a generic `BadRequestException` on any failure (including transient DNS resolver errors) and no logging distinguishing "domain doesn't exist" from "DNS lookup timed out/failed" — a flaky resolver would incorrectly reject legitimate enterprise creation.

## 5. Top 3 Priorities

1. **Fix the `findByEnterpriseId` / `EnterpriseService` authorization gap** — any logged-in user can enumerate another enterprise's organizations (`organization.controller.ts:54-63`, `enterprise.service.ts` ignoring `user`); this is the only outright security hole found and is cheap to fix (compare `user.enterpriseId` to the requested id, or restrict to platform admins).
2. **Unify the two enterprise-creation paths** — route `OrganizationService.setupCompany()` (`organization.service.ts:105-115`) through `EnterpriseService.create()` (or extract its validation/seeding into a shared method) so self-serve tenants get the same domain validation and `enterprise_app` seeding as platform-admin-created ones, and so there's exactly one place that knows how to construct a valid `Enterprise`.
3. **Make `assignAppsToEnterprise`/`bulkCreate` idempotent** (upsert or pre-filter existing `(enterprise_id, app_id)` pairs) — this is a live correctness bug that will break subscription creation for any second organization onboarded under an existing enterprise, which is precisely the multi-org-per-enterprise scenario the module exists for.
