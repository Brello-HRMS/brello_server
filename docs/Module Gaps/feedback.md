# Module: Feedback

## 1. Purpose & Current Usage
- In-app product feedback / issue-reporting channel. Org users (RBAC-gated) submit `FEEDBACK` or `ISSUE_REPORT` tickets with optional attachments; Platform Admins triage, reply, change status/priority, and view stats across all orgs.
- Entities: `FeedbackTicket` (`entities/feedback-ticket.entity.ts`), `FeedbackComment` (`entities/feedback-comment.entity.ts`), `FeedbackStatusLog` (`entities/feedback-status-log.entity.ts`).
- Services: `FeedbackService` (org-side: create, findAll, findOne, addComment — `services/feedback.service.ts`) and `PlatformFeedbackService` (admin-side: findAll, findOne, update, addComment, getStats — `services/platform-feedback.service.ts`).
- Controllers: `FeedbackController` at `/feedback-tickets` (`JwtAuthGuard` + `AccessGuard` + `RequirePermission('FEEDBACK_REPORT', ...)`, `controllers/feedback.controller.ts`) and `PlatformFeedbackController` at `/platform/feedback-tickets` (`JwtAuthGuard` + `PlatformAdminGuard`, `controllers/platform-feedback.controller.ts`).
- Repositories: `FeedbackTicketRepository`, `FeedbackCommentRepository`, `FeedbackStatusLogRepository` — thin TypeORM wrappers, no leaked query builders outside the repo layer.
- Registered in `src/app.module.ts:38,93`. Real frontend consumer exists: `brello_webapp/src/features/feedback/**` (`feedbackApi.ts`, `useFeedback.ts`, `usePlatformFeedback.ts`, `FeedbackPage.tsx`, `PlatformFeedbackPage.tsx`, `FeedbackDetailDrawer.tsx`, `PlatformFeedbackDetailDrawer.tsx`) — this is a live, used module, not scaffolding.
- No dead code found inside the module itself — every repository method, DTO field, and entity column is reachable from a service or the frontend. The one near-miss is `resolved_at`/`closed_at` on `FeedbackTicket`, which looked unused at first glance but are in fact set in `FeedbackTicketRepository.updateStatus` (`repositories/feedback-ticket.repository.ts:89-90`).

## 2. Intended / Ideal Usage
PRD: `brello_server/docs/prd/feedback-and-report.md`. Implementation matches the PRD closely — routes, DTOs, lifecycle transitions (`enums/feedback-status.enum.ts:19-35` vs PRD "State Transition Rules" table), stats shape, and soft-delete behavior all line up 1:1.
- One correction to the audit brief: the PRD explicitly lists **"Email notifications (in-app only)"** under **Out of Scope (V1)** (PRD lines 41-49, 140). So IN_APP-only notifications in `feedback.service.ts:52,97,121-128` and `platform-feedback.service.ts:84,107,128-134` are **not** a PRD deviation — they match spec. The real gap is that the PRD's own V1 scoping decision (no email) means Platform Admins have no out-of-band way to learn about a new ticket if they miss the in-app notification, which is compounded by the env-var issue below.
- Gap actually present vs. PRD: the PRD's RBAC section states access is "purely RBAC-driven ... via the existing RBAC system without code changes" (PRD line 59, 426), but the `FEEDBACK_REPORT` permission key is never registered anywhere in the RBAC seed/module list (see §3) — so today no role can actually be granted this permission through normal seed/admin flow, contradicting the PRD's core design decision.
- Everything else in scope (org ticket CRUD, platform triage, status log, stats) is implemented as specified. No evidence of SLA timers, roadmap/upvoting, or email-to-ticket ingestion being attempted — correctly left out per PRD's Out-of-Scope list.

## 3. Cross-Module Connections
- **Depends on**: `NotificationModule`/`NotificationService` (`feedback.module.ts:18,24`; used for both submitter and platform-admin notifications), `RbacModule` (imported at `feedback.module.ts:19,25` but only used transitively via `AccessGuard`/`RequirePermission`, not injected into any feedback service), `Organization` entity (`platform-feedback.service.ts:7,34-38`, queried directly via `DataSource.getRepository(Organization)` rather than through an OrganizationService/repository), `Document` module (implicitly, via `document_id` in `dto/create-feedback-ticket.dto.ts:16-17` and `entities/feedback-ticket.entity.ts:53` — no `DocumentModule` import or ownership/existence check in this module).
- **Depends on it**: `brello_webapp/src/features/feedback/**` (org + platform admin UI), nothing else server-side references `FeedbackModule`, `FeedbackService`, or `PlatformFeedbackService` (confirmed via repo-wide grep — only `app.module.ts` wires it in).
- **Missing/expected connections**:
  - RBAC seed gap: `FEEDBACK_REPORT` (the module permission key used in `feedback.controller.ts:35,46,56,63` and the PRD) does not appear anywhere under `src/modules/rbac/**`, nor in any of the `src/seeds/seed-*-module.ts` scripts (e.g. `seed-org-profile-module.ts`, `seed-announcement-module.ts`, `seed-payroll-permissions.ts` all exist; there is no `seed-feedback-module.ts`). `src/seeds/seed-brello-v2-base.ts` seeds `ANNOUNCEMENT`/`EMP_ANNOUNCEMENT` module rows (lines 82, 115) but has no `FEEDBACK`/`FEEDBACK_REPORT` row. Without a seeded module + permission row, `AccessGuard` has nothing to check against for any role, which likely blocks every org user from ever satisfying `RequirePermission('FEEDBACK_REPORT', ...)` in a fresh environment.
  - Platform-admin recipient resolution is a single hardcoded env var, not an RBAC lookup: `feedback.service.ts:19` (`PLATFORM_ADMIN_NOTIFICATION_USER_ID = process.env.PLATFORM_ADMIN_USER_ID ?? null`) and `feedback.service.ts:119-120` (`notifyPlatformAdmin` silently returns/no-ops if unset). This should resolve recipients via the RBAC "platform admin" role/user set (the same concept `PlatformAdminGuard` at `src/core/guards/platform-admin.guard.ts:15` checks via `user.isPlatformAdmin`) instead of one static user ID — today only a single admin (or none, if the env var is unset) is ever notified of new tickets or org follow-up comments, org-wide.
  - EMAIL channel: intentionally out of scope per PRD (see §2), but worth flagging as a real product gap given the above single-recipient/no-op risk — if that one admin misses the in-app notification (or the env var is unset), there is currently no fallback channel at all.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)
- **`PlatformFeedbackService` reaches across module boundaries via raw `DataSource`** instead of an `OrganizationService`/repository abstraction: `platform-feedback.service.ts:25,34-38` does `this.dataSource.getRepository(Organization).find(...)` directly. Matters because it bypasses whatever access-control/service-layer logic `OrganizationModule` might otherwise enforce, and couples this module to `Organization`'s internal entity shape.
- **RBAC permission never registered** (see §3): `feedback.controller.ts:35,46,56,63` reference `'FEEDBACK_REPORT'` but no seed script or RBAC module list defines it, unlike every sibling module (`ANNOUNCEMENT`, `ORG_PROFILE`, payroll permissions all have dedicated seed files). Matters because the module is unusable end-to-end in any environment seeded only from the existing scripts — a role can never be granted access through the documented RBAC flow.
- **Platform-admin notification fan-out is a single env var, not a role-based recipient list**: `feedback.service.ts:19,119-120`. Matters because it's a single point of failure (unset env var = total notification silence, confirmed: `notifyPlatformAdmin` returns early with no logging) and cannot notify more than one admin even when set.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)
- **No ownership/existence validation on attachment `document_id`s**: `dto/create-feedback-ticket.dto.ts:15-27` validates shape (`IsUUID`, `IsString`) but never checks the referenced document exists or belongs to the submitting org/enterprise before persisting it onto the ticket (`feedback.service.ts:37-50`). Matters because a malicious or buggy client could attach an arbitrary `document_id` from another org, and it would be silently stored and later surfaced to a Platform Admin (and resolved to a signed URL by the frontend) without any ownership check.
- **Notification failures are swallowed with no logging**: every `.catch(() => {})` in `feedback.service.ts:129-131` and `platform-feedback.service.ts:136-138` discards the error entirely — no `Logger.error`/`Logger.warn` call. Matters because a real notification-delivery outage (not just the env-var no-op case) would be completely invisible in logs/observability.
- **Silent no-op on missing platform admin config**: `notifyPlatformAdmin` (`feedback.service.ts:119-120`) returns without logging when `PLATFORM_ADMIN_NOTIFICATION_USER_ID` is falsy — same observability gap as above, specific to a misconfiguration case that's easy to hit in a new environment.
- **Inherited-but-unused `BaseEntity` columns on `FeedbackTicket`**: `FeedbackTicket` extends `BaseEntity` (`entities/feedback-ticket.entity.ts:19`), which contributes `status` (a generic `Status` enum, default `ACTIVE`) and `description` (`common/entities/base.entity.ts`) columns that are never read or written by this module — the ticket's real lifecycle state lives in `ticket_status` and its real text lives in `ticket_description`. Minor but creates two same-shaped-but-unrelated columns (`status` vs `ticket_status`, `description` vs `ticket_description`) per row, which is confusing for anyone querying the table directly.

### Technical (performance, security, scalability, test coverage, observability/logging)
- **No test files found**: no `*.spec.ts` exists under `src/modules/feedback/` (confirmed via directory listing) — none of the status-transition validation, type/category validation, or org-scoping logic has automated coverage.
- **`getStats()` runs three unindexed-by-result live aggregate queries with no caching** (`repositories/feedback-ticket.repository.ts:106-144`), explicitly acknowledged as a V1 tradeoff in the PRD (line 436) — acceptable now, but flagged as a scalability gap if ticket volume grows, since there's no cached counter table or materialized view yet.
- **No rate limiting on ticket/comment submission**: `feedback.controller.ts:33-43,61-70` has no throttling decorator, so a single org user could spam-create tickets/comments, each triggering a platform-admin notification attempt.

## 5. Top 3 Priorities
1. **Seed the `FEEDBACK_REPORT` RBAC module/permission** (add a `seed-feedback-module.ts` analogous to `seed-announcement-module.ts`, and add a row in `seed-brello-v2-base.ts`) — without this, the module is likely inaccessible to any org role today, which is a functional blocker, not just a quality issue.
2. **Replace the hardcoded `PLATFORM_ADMIN_NOTIFICATION_USER_ID` env var with an RBAC-driven lookup** (e.g. fan out to all users where `isPlatformAdmin === true`, mirroring `PlatformAdminGuard`'s check) — removes the single point of failure and the silent full-outage mode when the env var is unset.
3. **Add logging to the swallowed notification-failure `.catch()` blocks** in both services — cheap fix that closes the observability gap so a real delivery outage (as opposed to the config no-op) is at least visible in logs.
