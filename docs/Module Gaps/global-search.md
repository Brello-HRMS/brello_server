# Module: Global Search

## 1. Purpose & Current Usage

- **Entities**: `GlobalSearchDocument` (`brello_server/src/modules/global-search/entities/global-search-document.entity.ts`) is a single centralized "searchable document" table (`global_search_documents`) keyed by `(enterprise_id, entity_id, entity_type)` with `title`/`subtitle`/`keywords`, a `route`, a `permissions` array, `module_key`, `is_active`/`is_deleted` flags, and JSONB `metadata`. `RecentSearch` (`entities/recent-search.entity.ts`) is a separate per-user recent-search log table.
- **Services**:
  - `SearchIndexingService` (`services/search-indexing.service.ts`) — one `index<Entity>()`/`remove<Entity>()` method pair per indexable entity type (10 pairs), each building a fire-and-forget `upsert`/`softDelete` call (errors are only `logger.warn`'d, never thrown — see Coding gaps).
  - `GlobalSearchDocumentRepository` (`repositories/global-search-document.repository.ts`) — raw SQL `upsert()` (writes the row, then a second `UPDATE ... SET search_vector = to_tsvector(...)` query), `softDelete()`, and `search()` (combines `plainto_tsquery` FTS with `pg_trgm` `similarity(title, …)`, ranked by exact/prefix/substring match first).
  - `SearchDatabaseInitService` (`services/search-database-init.service.ts`) — `OnApplicationBootstrap` hook that runs raw DDL every app start: `CREATE EXTENSION pg_trgm`, adds the `search_vector` column, adds a unique constraint, creates the GIN indexes.
  - `SearchQueryService` (`services/search-query.service.ts`) — orchestrates `search()` (delegates to the repository, then derives a "Modules" chip list from `SEARCHABLE_MODULES` config) and recent-search read/write.
  - `SearchCleanupService` (`services/search-cleanup.service.ts`) — daily cron (`EVERY_DAY_AT_MIDNIGHT`) that deletes `RecentSearch` rows older than 30 days.
- **Controller**: `SearchController` (`controllers/search.controller.ts`) exposes `GET /search`, `GET /search/recent`, `POST /search/recent`, guarded only by `JwtAuthGuard` (no permission guard — see Gaps §4).
- **Who calls in today (webapp)**:
  - `brello_webapp/src/components/layout/MainLayout.tsx:20-23` — global `keydown` listener opens the modal on **Cmd/Ctrl+`/`** (not Cmd+K as informally assumed) via `useSearchStore().openModal()`. `Header.tsx` has no search icon/button — the keyboard shortcut is the only entry point.
  - `brello_webapp/src/features/search/components/SearchModal/SearchModal.tsx` — renders the modal, debounces the query 200ms, calls `useSearch` / `useRecentSearches` / `useSaveRecentSearch`.
  - `brello_webapp/src/features/search/api/search.ts:12-17` — `GET {BASE}/search?q=`, `GET {BASE}/search/recent`, `POST {BASE}/search/recent`.
- **Entity types actually indexed today** (confirmed via grep for `SearchIndexingService` usage across `src/modules`, all 10 defined index methods are wired to a real call site, not just employees as previously assumed):
  - `employee` — `modules/user/services/employee.service.ts:239` (create), `:459` (on `updateBasicInfo`, only when name changes), `:1101` (soft delete).
  - `department` — `modules/departments/services/department.service.ts:70,151,170`.
  - `designation` — `modules/designations/services/designation.service.ts:69,137,148`.
  - `client` — `modules/client/services/client.service.ts:53,112,128`.
  - `project` — `modules/project/services/project.service.ts:83,193,204`.
  - `announcement` — `modules/announcement/services/announcement.service.ts:76,212,230`.
  - `company_policy` — `modules/company-policy/services/company-policy.service.ts:40,102,111`.
  - `role` — `modules/role/services/role.service.ts:63,199,218`.
  - `holiday` — `modules/holiday/services/holiday.service.ts:44,84,101`.
  - `reimbursement` — `modules/reimbursement/services/reimbursement.service.ts:53,114,133`.
- **NOT indexed** (grepped every module under `src/modules` — 25 modules exist; only the 10 above ever reference `SearchIndexingService`): `letter-management` (issued/offer letters and letter templates — a whole separate documented app area per memory), `feedback` (tickets/reports), `leave-request`, `leave-balance`, `leave-config`, `attendance`, `timesheet`, `document`, `notification`, `payroll`, `billing`, `plan`, `organization` (org profile), `industry-type`, `lead`, `enterprise`, `enterprise-admin`, `audit`. None of these emit any search-index writes on create/update/delete.

## 2. Intended / Ideal Usage

A global search feature like this should let any new module "opt in" with minimal, declarative wiring — e.g. an interface/decorator the entity implements plus a listener on a domain event, so indexing survives refactors of the producing module without that module's service needing to know about `SearchIndexingService` at all. It should also ship a reindex/backfill job (triggerable via CLI or admin endpoint) so: (a) data that existed before this module was added is searchable, (b) the ranking/tokenization logic (`to_tsvector('simple', …)`) can be changed and reapplied to the whole table without a fresh event stream, and (c) an index that silently failed a write (see fire-and-forget `.catch()` in `search-indexing.service.ts:82-90`) can be repaired. None of this exists today — see §3 and §4.

## 3. Cross-Module Connections

- **Depends on**: nothing beyond core Nest/TypeORM/`@nestjs/schedule` and the raw Postgres `DataSource` — it is a leaf module with no imports from other feature modules (`global-search.module.ts:1-31`).
- **Depended on by** (import `SearchIndexingService` and call it directly from their own service layer): `user` (employee), `departments`, `designations`, `client`, `project`, `announcement`, `company-policy`, `role`, `holiday`, `reimbursement` — 10 modules, confirmed by grep above.
- **Missing/expected connections**: `letter-management` (issued letters — arguably the single most search-worthy entity per the module's own architecture docs), `feedback`, `leave-request`, `attendance`/`timesheet`, `document`, `organization`/org-profile, `lead` do not call into `SearchIndexingService` anywhere — creating/issuing/updating any of these entities has zero effect on search results.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **Indexing is wired via direct, bespoke calls scattered across 10 unrelated services rather than an event-driven or self-registering pattern.** Every producer module (`employee.service.ts`, `department.service.ts`, `client.service.ts`, etc.) imports `SearchIndexingService` directly and calls a hand-written `index<X>()`/`remove<X>()` pair (see call sites in §1). Adding search support to an 11th module means editing that module's service by hand and adding a new method pair to `SearchIndexingService` — there is no shared interface, decorator, or event bus, so it's easy (and has already happened 15+ times, per §1's "NOT indexed" list) for a module to simply be forgotten.
- **No reindex/backfill capability exists at all** — grepped for `reindex`/`backfill` across `src/`, the only hits are unrelated letter-content seed scripts. If the index falls out of sync (fire-and-forget failure, entity added to the app before this module existed, tokenization rule change), there is no code path to repair it short of manually re-saving every entity.
- **`SearchDatabaseInitService.onApplicationBootstrap()` (`services/search-database-init.service.ts:11-17`) performs raw DDL (`CREATE EXTENSION`, `ALTER TABLE ADD COLUMN IF NOT EXISTS`, a `DO $$ ... IF NOT EXISTS ... ADD CONSTRAINT` block, `CREATE INDEX IF NOT EXISTS`) on every process boot**, instead of a versioned SQL migration file — the project's own convention, evidenced by `brello_server/docs/migrations/*.sql` for every other schema change (e.g. `2026-07-07-letter-management-consolidation.sql`). Running unversioned DDL from app code on every boot is inconsistent with the rest of the codebase and, under multiple concurrently-starting instances, the check-then-act `IF NOT EXISTS` pattern for the constraint (`search-database-init.service.ts:39-48`) is a race condition (two instances can both see "not exists" and both attempt `ADD CONSTRAINT`, one erroring on the other's DDL depending on lock timing).

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

- **The `permissions` field on `GlobalSearchDocument` (and `SEARCHABLE_MODULES[].permission`) is stored and returned to the client but never enforced anywhere.** `SearchController` (`controllers/search.controller.ts:17-18`) only applies `JwtAuthGuard` — contrast with every real resource controller in the app, e.g. `holiday.controller.ts:36,47,58,...` which uses `@RequirePermission('LEAVE_HOLIDAYS', 'view')` etc. `SearchQueryService.search()` (`services/search-query.service.ts:27-43`) and `GlobalSearchDocumentRepository.search()` (`repositories/global-search-document.repository.ts:111-152`) filter only by `enterprise_id`/`organization_id`/`is_active`/`is_deleted` — never by the requesting user's actual granted permissions. The webapp likewise never reads `item.permissions` to hide anything (`brello_webapp/src/features/search/components/SearchResults/SearchResults.tsx` renders every returned item unconditionally). Net effect: any authenticated user in an org can search and see title/subtitle/route for roles, reimbursements, company policies, etc. regardless of whether their own RBAC role grants `role.read`, `reimbursement.read`, or `policy.read`.
- **Fire-and-forget indexing swallows all errors.** `SearchIndexingService.index()`/`.remove()` (`services/search-indexing.service.ts:82-100`) never `await` the repository call in the caller and only `logger.warn` on failure — a failed index write (bad connection, constraint violation, etc.) is silently dropped with no retry, no dead-letter, and no way for an operator to discover it happened short of grepping logs.
- **Dead-end routes for 4 of the 10 indexed entity types.** `indexAnnouncement` routes to the static `/announcements/list` (`search-indexing.service.ts:252`), `indexPolicy` to `/organisation/policies` (`:278`), `indexRole` to `/access/roles` (`:304`), `indexReimbursement` to `/reimbursement/list` (`:361`) — none carry the entity id. Clicking a search result for a specific announcement/policy/role/reimbursement just opens the generic list page with no deep-link/highlight; confirmed the corresponding webapp features (`brello_webapp/src/features/announcement`, `.../reimbursement`, `.../role`) have no `useSearchParams`/id-based scroll-to logic to consume such a link even if one were added. This is inconsistent with `employee`/`department`/`designation`/`client`/`project`/`holiday`, which all route to a real per-entity detail page.
- **Employee reindexing only fires on name changes.** `employee.service.ts:459` only calls `indexEmployee` when `updateBasicInfo` changed `firstName`/`lastName`/`middleName`; every other update path (`updatePersonalDetails`, `updateEmploymentDetails`, `updateSystemAccess`, `updatePayrollInformation`, etc. — `employee.service.ts:494-1000`) never touches the search index, which is correct only because the search doc's `title`/`subtitle` happen to be name/email — but it is a hidden, undocumented coupling between "which fields are indexed" and "which update methods reindex," easy to violate the next time the indexed fields change.

### Technical (performance, security, scalability, test coverage, observability/logging)

- **Zero automated test coverage.** No `*.spec.ts` file exists anywhere under `brello_server/src/modules/global-search/` for the entity, repository, services, or controller — the FTS ranking query (`global-search-document.repository.ts:118-151`, a hand-tuned `CASE` + `ts_rank` + `similarity` blend) and the upsert/soft-delete/search-vector-refresh sequence have no regression protection at all.
- **No observability beyond `logger.warn` on failure.** There is no metric/counter for index-write failures, no success logging, and no way to tell from monitoring whether the index is keeping up with writes; the daily cleanup cron (`search-cleanup.service.ts:14-30`) does log a deletion count, but the indexing path (the far more failure-prone one) does not.
- **`upsert()` does two round-trips per write** — an `INSERT ... ON CONFLICT` followed by a separate `UPDATE ... SET search_vector = to_tsvector(...)` scoped by `(enterprise_id, entity_id, entity_type)` (`global-search-document.repository.ts:41-90`) — instead of computing `search_vector` inline in the first statement or via a generated column/trigger, doubling write cost and creating a (currently harmless, since nothing else reads the row in between) window where the row exists without a `search_vector`.

## 5. Top 3 Priorities

1. **Enforce `permissions` on the query path.** Right now the only real access control on `/search` is "logged in, same enterprise/org" — any authenticated user can discover titles/subtitles for roles, reimbursements, and policies they have no RBAC read permission for, which is a genuine access-control gap in a codebase that otherwise guards every other resource controller with `@RequirePermission`.
2. **Wire `letter-management` and `feedback` into the index** (and decide, deliberately, which of `leave-request`/`attendance`/`document`/`lead` should follow) — these are documented as major feature areas per project memory, yet a user cannot find an issued letter or a feedback ticket through global search at all, which undermines the feature's basic value proposition.
3. **Replace ad-hoc per-module wiring with a backfill/reindex job plus a lighter-weight registration pattern.** With 15+ entity types already un-indexed and no way to repair drift, every future module addition risks silently missing search support forever unless someone remembers to hand-wire it — a reindex CLI/admin endpoint would at least make the omission fixable after the fact.
