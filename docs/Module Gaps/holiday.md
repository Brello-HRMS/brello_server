# Module: Holiday

## 1. Purpose & Current Usage

The Holiday module manages a per-organization holiday calendar: `HolidayCalendar` (one row per org+year, with a `status` of PENDING/ACTIVE/INACTIVE) and `Holiday` (individual dated entries — `PUBLIC` / `OPTIONAL` / `COMPANY`, belonging to a calendar). It exposes:

- `HolidayCalendarService` / `HolidayCalendarController` (`brello_server/src/modules/holiday/controllers/holiday-calendar.controller.ts`) — CRUD, `activate`/`deactivate` (only one ACTIVE calendar per org per year, enforced via `HolidayCalendarRepository.deactivateOthersInYear`, `brello_server/src/modules/holiday/repositories/holiday-calendar.repository.ts:70-84`), and `clone` (copy a prior year's holidays into a new calendar).
- `HolidayService` / `HolidayController` (`brello_server/src/modules/holiday/controllers/holiday.controller.ts`) — CRUD on individual holidays scoped to a calendar, plus `getMonthView`.
- `EmployeeHolidayController` (`brello_server/src/modules/holiday/controllers/employee-holiday.controller.ts`) — employee self-service "upcoming holidays" list (`GET /employee/holidays`), backed by `HolidayService.getEmployeeHolidays`.

**Who actually calls into it:**
- **Attendance** — `AttendanceMaterializationService` (`brello_server/src/modules/attendance/services/attendance-materialization.service.ts`) is fully holiday-aware. Its daily cron (`attendance-cron.service.ts:22-27`, `@Cron('0 1 * * *', { name: 'daily-absent-marking' })` → `runDailyAbsentMarking`) calls `isHolidayForOrg()` (materialization service lines 490-504) before marking an employee ABSENT, and applies priority `weekly_off > holiday > leave > absent` (lines 179-194). It does **not** go through `HolidayService`/`HolidayRepository` — it injects `Repository<Holiday>` directly (line 6, 63-64) and re-implements the holiday-lookup query itself.
- **Holiday → Attendance (reverse direction)** — `HolidayService.create`/`.remove` (`brello_server/src/modules/holiday/services/holiday.service.ts:31-56`, `88-102`) call `AttendanceMaterializationService.syncHolidayToAttendance` / `.reverseHolidaySync` directly, so creating/deleting a holiday retroactively stamps/reverts `HOLIDAY` attendance records for the whole org (best-effort, errors only logged, lines 47-53, 92-99).
- **Leave Request** — `LeaveRequestService` (`brello_server/src/modules/leave-request/services/leave-request.service.ts`) also injects `Repository<Holiday>` directly (lines 36, 76-77) rather than using `HolidayService`. `getHolidayDates()` (lines 1081-1096) loads active-calendar holiday dates for the leave's year, and `computeBreakdown()` (lines 1228-1277) excludes those dates from `billable_days` (holidays are "never billable, even under the sandwich rule", line 1265-1268).
- **Global Search** — `SearchIndexingService.indexHoliday`/`removeHoliday` (`brello_server/src/modules/global-search/services/search-indexing.service.ts:316-338`) is wired and called from `HolidayService` on create/update/delete (`holiday.service.ts:44,84,101`) — actively indexed, not dead.
- **Frontend** — full UI exists under `brello_webapp/src/features/holidays/` (`AddCalendarDialog.tsx`, `AddHolidayDialog.tsx`, `HolidayCard.tsx`, `useHolidays.ts`) and `brello_webapp/src/pages/holidays/` (`HolidaysPage.tsx`, `HolidayCalendarView.tsx`), plus a dashboard "upcoming holidays" widget (`features/dashboard/components/HolidaysCard/`).

**Dead / unused parts:**
- `HolidayCalendar.holiday_count` (`entities/holiday-calendar.entity.ts:21`) is populated via `loadRelationCountAndMap` in `findAllByOrg` but there is no evidence any frontend list view consumes it beyond a count badge — low-risk, not flagged further.
- `HolidayService.remove()` calls `this.findOne(user, id)` twice back-to-back (`services/holiday.service.ts:89,91`) — the second call's result is discarded; dead code.

## 2. Intended / Ideal Usage

The model is: one holiday calendar per organization per calendar year, with exactly one ACTIVE calendar at a time, containing PUBLIC/OPTIONAL/COMPANY-typed dated entries that (a) drive attendance auto-marking (weekly-off > holiday > leave > absent priority) and (b) are excluded from leave-day billing. Cloning lets HR roll a calendar forward year-to-year without re-entering fixed dates. There is no concept of region/location-specific calendars (e.g., a Mumbai office vs. a Bangalore office with different state holidays) — the schema is strictly org+year scoped, and no recurring-yearly-holiday feature exists (each year's dates must be re-entered or cloned+edited manually).

## 3. Cross-Module Connections

**Depends on:**
- `RbacModule`, via `AccessGuard`/`RequirePermission` (`holiday.module.ts:12,19`).
- `GlobalSearchModule` — `SearchIndexingService` (`holiday.module.ts:13,20`).
- `AttendanceModule` — imported so `HolidayService` can inject `AttendanceMaterializationService` (`holiday.module.ts:14,21`; `holiday.service.ts:17,28`).
- `AuditModule` — `AuditContextService` for pre-value snapshotting (`holiday.service.ts:16`, `holiday-calendar.service.ts:15`).

**Depended on by:**
- `AttendanceModule` — imports `Holiday` entity directly into its own `TypeOrmModule.forFeature` (`attendance/attendance.module.ts:15,68`), bypassing the holiday module's public API.
- `LeaveRequestModule` — same pattern (`leave-request/leave-request.module.ts:13,27`).
- This creates a **bidirectional coupling**: `HolidayModule` imports `AttendanceModule` for the service call, while `AttendanceModule` and `LeaveRequestModule` independently reach into `HolidayModule`'s `Holiday` entity/table. No `HolidayRepository` (the module's own repository abstraction) is used by any external consumer — both attendance and leave-request re-implement their own "is this date a holiday" query against the raw entity instead of calling a shared method.

**Missing or expected connections:**
- Both external consumers duplicate the same query shape (`join calendar, filter status=ACTIVE, is_deleted=false, date=:date`) independently — `attendance-materialization.service.ts:494-502` and `leave-request.service.ts:1085-1094` — instead of calling a single `HolidayRepository.isHoliday(orgId, date)` / `.findByOrgAndYear` (which already exists at `holiday.repository.ts:81-94` but is unused outside the holiday module itself).
- Weekend detection is **inconsistent across modules**: attendance's `isWeeklyOffDay()` reads a configurable `AttendanceRule.weekly_off` field (per employee/department), but leave-request hardcodes Sat/Sun (`leave-request.service.ts:1256`, `dow === 0 || dow === 6`, and the comment at line 1226: `// weekend = Sat/Sun`). If an org configures a non-Saturday/Sunday weekly-off pattern, attendance and leave-day billing will disagree on which days are "off," even though both correctly consult the holidays table.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

1. **No shared holiday-lookup abstraction; two modules duplicate the query.** `attendance-materialization.service.ts:490-504` (`isHolidayForOrg`) and `leave-request.service.ts:1081-1096` (`getHolidayDates`) both hand-roll the same `holiday ⋈ calendar WHERE org, ACTIVE, not deleted, date=X` query directly against `Repository<Holiday>`, instead of calling the holiday module's own `HolidayRepository.findByOrgAndYear` (`holiday.repository.ts:81-94`), which is never imported outside `src/modules/holiday/`. Matters because any future change to "what counts as an active/valid holiday" (e.g., adding location scoping) must be updated in three places instead of one, and already risks drift (see weekend-logic mismatch below).
2. **Bidirectional module coupling.** `HolidayModule` imports `AttendanceModule` to call `AttendanceMaterializationService` (`holiday.module.ts:14,21`; `holiday.service.ts:17,28,48,94`), while `AttendanceModule` and `LeaveRequestModule` import the `Holiday` entity straight from `holiday/entities` (`attendance.module.ts:15,68`, `leave-request.module.ts:13,27`). Matters because it breaks clean layering — a change to the `Holiday` entity's shape can silently break attendance/leave-request compile-time, and there's no single owner of "who is allowed to read holiday data."
3. **No region/location-specific or recurring-holiday model.** `HolidayCalendar` (`entities/holiday-calendar.entity.ts`) is scoped only to `organization_id` + `year`; there is no `location_id`/`office_id` column and no "recurs every year" flag anywhere in the entity or DTOs (`dto/create-holiday.dto.ts`, `dto/create-holiday-calendar.dto.ts`). Matters for any multi-location org with state-specific holidays — today it's one calendar for the whole company, and every year's dates must be manually re-entered or cloned.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

1. **`getMonthView` has no organization ownership check — cross-org data leak (IDOR).** `HolidayController.getMonthView` (`holiday.controller.ts:89-99`) passes `calendarId` straight to `HolidayService.getMonthView` (`holiday.service.ts:104-123`), which calls `this.holidayRepo.findByCalendar(calendarId, { month })` with **no** verification that `calendarId` belongs to `user.organizationId` — unlike every other method in the same service (`create`, `findAll`, `findOne` all call `calendarRepo.findOneByOrg(calendarId, user.organizationId)` first, e.g. lines 32-35, 59-61). Any authenticated user who guesses/enumerates another org's calendar UUID can read that org's holiday names/dates via `GET /holidays/calendars/:calendarId/month-view`. Matters because it's a real, exploitable cross-tenant data leak in an otherwise consistently-guarded module.
2. **`getMonthView`'s "group by date" is a no-op — it never actually groups.** The comment at `holiday.service.ts:107` says `// Group by date`, but the code (`108-116`) does `holidays.map(h => ({ date: h.date, holidays: [{...}] }))` — a 1:1 map, not a `groupBy`. If two holidays fall on the same date, the response contains two separate day entries instead of one entry with two holidays. Matters because it silently produces a wrong/duplicated calendar view for any month with more than one holiday on the same day (e.g., a company holiday overlapping a public holiday).
3. **RBAC permission module code mismatch — the entire admin Holiday API is likely unusable by any non-platform-admin role.** Every endpoint in `HolidayController` and `HolidayCalendarController` uses `@RequirePermission('LEAVE_HOLIDAYS', ...)` (e.g. `holiday-calendar.controller.ts:36,46,56,67,79,90,101,113`; `holiday.controller.ts:36,47,58,69,81,91`), and the frontend agrees (`brello_webapp/src/routes/adminRoutes.tsx:197,205`, `src/enum/modules.ts:20`, `ModuleCode.LEAVE_HOLIDAYS`). But the seeded module catalog that `PermissionResolverService` resolves against only defines `code: 'HOLIDAY'` (`brello_server/src/seeds/seed-brello-v2-base.ts:65`) — there is no `LEAVE_HOLIDAYS` entry anywhere in any seed file. `PermissionResolverService.hasPermission` (`rbac/services/permission-resolver.service.ts:118-131`) matches on `mod.code === moduleCode` and falls through to `return false` (line 130) when no module matches, and `AccessGuard` (`core/guards/access.guard.ts:73-80`) throws `403 Forbidden` on any `false`. Net effect: for every organization user except platform admins (who bypass the guard entirely, `access.guard.ts:57-58`), no role can ever be granted access to `LEAVE_HOLIDAYS` because that module code doesn't exist to attach a permission to — the entire Holiday Calendar admin CRUD surface (create/view/update/delete/activate/clone calendars and holidays) is unreachable in practice. Matters because this is a hard functional break, not a theoretical one — it blocks the feature for real users, not just an edge case.
4. **Repository method doesn't defensively scope by org.** `HolidayRepository.findByCalendar` (`holiday.repository.ts:19-45`) filters only by `calendar_id`, relying entirely on callers to have already checked org ownership. Every current caller does check first (`holiday.service.ts:59,63` and `104-105` — except the `getMonthView` bug above), but the repository method itself offers no defense-in-depth; a future caller that forgets the ownership check reproduces the same IDOR class of bug.

### Technical (performance, security, scalability, test coverage, observability/logging)

1. **`syncHolidayToAttendance` is O(all active employees in org) per holiday create, run synchronously in the request but fired-and-forgotten.** `HolidayService.create` (`holiday.service.ts:46-53`) does `void this.materialization.syncHolidayToAttendance(holiday.id).catch(...)` — for a large org this touches every active employee and writes/updates an attendance record + audit row each (`attendance-materialization.service.ts:382-438`), with no batching/queueing. Matters for scalability on large tenants — a single "add holiday" API call can trigger thousands of writes with only a logged error on partial failure, no retry, and no way for the caller to know it partially failed.
2. **No automated tests found for the holiday module.** No `*.spec.ts` file exists under `src/modules/holiday/` (only source files listed under entities/dto/services/controllers/repositories). Matters because the module drives payroll-relevant attendance status and leave billing — both areas where a silent regression (e.g., the `getMonthView` grouping bug above) has real business impact and would not be caught by CI.
3. **No structured logging/observability on the RBAC gap.** The `AccessGuard` logs a generic `Access denied` warning (`access.guard.ts:74-76`) but nothing distinguishes "role legitimately lacks permission" from "the module code doesn't exist in the catalog at all" — the `LEAVE_HOLIDAYS` misconfiguration (Coding #3) would be indistinguishable from a normal permission denial in logs, making it hard to detect without reading the seed data.

## 5. Top 3 Priorities

1. **Fix the `LEAVE_HOLIDAYS` vs `HOLIDAY` module-code mismatch in RBAC.** This is a functional outage, not a nice-to-have — non-platform-admin users almost certainly cannot use any Holiday Calendar admin endpoint today, and the fix (align the seed's module code, or align the controllers/frontend to `HOLIDAY`) is small and isolated.
2. **Add the missing org-ownership check in `HolidayService.getMonthView`.** It's a live cross-tenant data leak (IDOR) sitting next to three sibling methods that already do the check correctly — a one-line fix (`calendarRepo.findOneByOrg` guard) closes it.
3. **Consolidate the duplicated holiday-lookup queries in attendance and leave-request behind a shared `HolidayRepository` method.** Both modules already inject `Repository<Holiday>` directly and hand-roll the same join; centralizing this removes the drift risk seen in the weekend/weekly-off inconsistency and makes any future holiday-model change (e.g., location scoping) a one-place fix instead of three.
