# Module: Industry Type

## 1. Purpose & Current Usage
- A minimal lookup/reference module: one entity (`IndustryType`, `brello_server/src/modules/industry-type/entities/industry-type.entity.ts:4-8`) with a single `name` column (unique, extends `BaseEntity` for `id`/`status`/timestamps). Standard repository/service/controller CRUD (`repositories/industry-type.repository.ts`, `services/industry-type.service.ts`, `controllers/industry-type.controller.ts`) exposed at `/industry-types`.
- Registered in `IndustryTypeModule` (`industry-type.module.ts:8-14`), imported by `app.module.ts:16,66` and by `OrganizationModule` (`src/modules/organization/organization.module.ts:13,30`), which injects `IndustryTypeService` into `OrganizationProfileService` to validate `industry_type_id` on create/update (`src/modules/organization/services/organization-profile.service.ts:16,29,52-53,147-148`).
- `OrganizationProfile` entity has a `ManyToOne` FK to `IndustryType` via `industry_type_id` (`src/modules/organization/entities/organization-profile.entity.ts:52-56`).
- Consumed at signup (`business_type_id` → `industry_type_id` mapping in `src/modules/organization/services/organization.service.ts:131`, fed by `brello_webapp/src/features/auth/components/Lead/LeadForm.tsx:105-110` via `useIndustryTypes` hook) and in the Org Profile "Company" card dropdown (`brello_webapp/src/features/organization/components/OrgCompanyCard/OrgCompanyCard.tsx:39-43`).
- Also has a platform-admin management screen in the webapp (`brello_webapp/src/pages/platform/PlatformIndustryTypePage.tsx`, `brello_webapp/src/features/platform/industryTypes/*`) that calls the very same public `/industry-types` REST endpoints for create/update/delete (`brello_webapp/src/features/platform/industryTypes/api.ts:9-21`) — there is no dedicated `platform-admin/industry-types` backend controller, unlike the equivalent Department/Designation platform features (see §3).
- `Seed: brello_server/src/seeds/seed-industry-types.ts` inserts 20 baseline industries directly via raw SQL (`INSERT ... ON CONFLICT (name) DO NOTHING`), run manually/out-of-band from the app.
- No dead code inside the module itself — every file (entity, both DTOs, repository, service, controller) is used. `countOrgsUsing` (`repositories/industry-type.repository.ts:53-59`) is only exercised by `remove()` guard logic.

## 2. Intended / Ideal Usage
- Today it is purely a flat, global name → id lookup used to populate a dropdown at signup and on the org profile page. There is no industry-specific behavior anywhere: no default policy sets, no letter-template variations, no payroll-component defaults keyed off `industry_type_id`. A `grep` for `industry_type_id`/`industryType` outside this module and `organization-profile`/signup turns up nothing in payroll, letter-templates, or policy modules.
- Ideal usage (matching the pattern already built for Departments/Designations — see `src/modules/platform/services/platform-department.service.ts`) would be: platform admins curate a global "default" catalog of industries, org data drives industry-specific behavior (e.g. suggested departments/policies), and admin mutation is served from a properly guarded, audited `platform-admin/*` controller distinct from the public read endpoint.

## 3. Cross-Module Connections
- **Depends on:** `BaseEntity`/`Status` common enum (`common/entities/base.entity.ts`, `common/enums`), `PlatformAdminGuard` and `JwtAuthGuard` (imported but effectively unused — see Gaps).
- **Depended on by:** `OrganizationModule` / `OrganizationProfileService` (FK validation + relation), organization signup flow (`organization.service.ts:131`), webapp signup lead form, webapp org-profile company card, webapp platform-admin industry-types page.
- **Missing/expected connections:**
  - No dedicated platform-admin controller/service (`platform-industry-type.controller.ts`/`.service.ts`) mirroring `PlatformDepartmentController`/`PlatformDesignationController`, even though the webapp already has a platform-admin UI for it.
  - No `AuditLog` integration (`src/modules/audit/decorators/audit-log.decorator.ts`, `AuditContextService`) despite Department/Designation platform CRUD logging every create/update/delete (`platform-department.controller.ts` uses `@AuditLog(...)` on all three mutating endpoints; `platform-department.service.ts` calls `auditContext.setPreValue(...)`). Industry Type mutations are invisible to the audit trail entirely.
  - No `is_default` (or org-scoping) column the way `Department`/`Designation` entities have (`src/modules/departments/entities/department.entity.ts:15-16`, `src/modules/designations/entities/designation.entity.ts:16-17,26-27`), so the model can't distinguish platform defaults from org-specific/custom entries even though the rest of the platform-admin feature set is built around that distinction.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)
- **No platform-admin/public split, unlike sibling modules.** `IndustryTypeController` (`controllers/industry-type.controller.ts:21-61`) serves both the public read used at signup and the platform-admin CRUD from the same `@Controller('industry-types')` with no controller-level guard, whereas Departments/Designations split this into a public controller and a separate `@Controller('platform-admin/departments')` class wrapped in `@UseGuards(JwtAuthGuard, PlatformAdminGuard)` (`src/modules/platform/controllers/platform-department.controller.ts:22-23`). This inconsistency means the webapp's "platform admin" Industry Types page is, on the backend, indistinguishable from an anonymous public endpoint.
- **No `is_default`/org-scoping abstraction**, unlike `Department.is_default`/`Designation.is_default` (`department.entity.ts:15-16`, `designation.entity.ts:16-17,26-27`) — Industry Type can't model "platform default vs org custom" even though its sibling lookup tables were explicitly built with that abstraction.
- **Schema managed only by `synchronize: dev` + a manual seed script**, with no migration file (`grep` over `docs/migrations` for `industry_type` returns nothing) — the table's existence in any non-dev environment depends entirely on someone remembering to run `seed-industry-types.ts`, unlike audited schema changes elsewhere in the codebase (`src/config/database.config.ts:36-37` explicitly warns `synchronize: true` must never be used in production, yet this table has no tracked migration as a fallback).

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)
- **`@UseGuards(JwtAuthGuard, PlatformAdminGuard)` on service methods is a no-op.** `IndustryTypeService.create`, `.update`, `.remove` (`services/industry-type.service.ts:23,62,96`) decorate plain service methods with `@UseGuards`; NestJS only reads guard metadata off controller classes/handlers, never off injectable service methods, so these decorators do nothing at runtime. Combined with the controller having zero guards of its own (`controllers/industry-type.controller.ts:25-60`), `POST/PATCH/DELETE /industry-types` are reachable by **any unauthenticated caller** — a real gap given the equivalent Department/Designation platform endpoints require `JwtAuthGuard` + `PlatformAdminGuard` at the controller level.
- **`LoggedInUser` params on `update`/`remove` are unused.** The controller injects `@LoggedInUser() user` on `update()` and `remove()` (`controllers/industry-type.controller.ts:48,57`) but never passes it to the service or uses it — dead parameter, likely a leftover from a partially-applied audit/ownership check.
- **Naming inconsistency across layers**: DTO/entity field is `industry_type_id`, but the signup DTO calls it `business_type_id` (`src/modules/organization/dto/setup-company.dto.ts:31`, mapped in `organization.service.ts:131`), and the webapp form field is named `industry` (`LeadForm.tsx:21,57`) — three different names for the same concept across the stack, increasing the chance of future mapping bugs.

### Technical (performance, security, scalability, test coverage, observability/logging)
- **Security: unauthenticated write access** (same root cause as above) — this is the most severe issue since anyone with network access to the API can create, rename, or soft-delete industry types that feed the organization signup flow and org-profile FK.
- **No audit logging** on create/update/delete, unlike Department/Designation platform CRUD which wraps every mutation in `@AuditLog(AuditLogModule.PLATFORM_SETUP, ...)` (`platform-department.controller.ts`) — Industry Type changes leave no trace in `system_audit_logs`.
- **No test coverage**: no `*.spec.ts` files exist anywhere under `src/modules/industry-type/`.
- **`countOrgsUsing` uses a raw SQL string literal against `organization_profile`** (`repositories/industry-type.repository.ts:53-59`) instead of the injected repository/query builder — bypasses TypeORM's schema awareness, so a future rename of the `organization_profile` table or `industry_type_id` column would silently break this guard without a compile-time signal.

## 5. Top 3 Priorities
1. **Fix the missing authorization** — move `JwtAuthGuard`/`PlatformAdminGuard` onto the controller (ideally by splitting into a public read-only controller plus a `platform-admin/industry-types` controller, mirroring Departments/Designations) so `create`/`update`/`remove` are no longer open to anonymous callers. This is a live security hole, not a stylistic nit.
2. **Add `AuditLog` decorators to the mutating endpoints**, consistent with Department/Designation platform CRUD, so industry-type changes are traceable — cheap to add once the controller split above is done.
3. **Add a tracked migration for the `industry_type` table** (and drop reliance on `synchronize: dev` + manual seed run) so schema provisioning is reproducible and reviewable like the rest of the schema.
