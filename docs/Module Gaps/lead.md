# Module: Lead

## 1. Purpose & Current Usage
- The module implements public marketing-site signup/waitlist capture: a visitor submits `POST /leads/register` (`lead.controller.ts:32-40`), which creates a `Lead` row and emails a 6-digit OTP (`lead.service.ts:40-70`). The visitor confirms via `POST /leads/verify-otp` (`lead.controller.ts:42-50`), which validates the OTP and, inside a DB transaction, marks the lead verified and creates a real `User` account from the lead's data (`lead.service.ts:227-256`).
- It also exposes internal admin endpoints — `GET /leads`, `GET /leads/:id`, `PATCH /leads/:id/status` (`lead.controller.ts:52-79`) — backed by `LeadRepository.findAll/findById/update` (`lead.repository.ts:16-39`).
- Callers today: the public marketing/signup flow in `brello_webapp/src/features/auth/api/auth.ts:25-32` (`/leads/register`, `/leads/verify-otp`), and the platform-admin pipeline UI in `brello_webapp/src/pages/platform/PlatformLeadsPage.tsx` + `brello_webapp/src/features/platform/leads/api.ts` (list/get/update-status). Both consumers are real, nothing here is unreferenced at the frontend level.
- Dead/unused parts: `CreateLeadDto.source` is validated (`@IsIn(['website'])`, `create-lead.dto.ts:47-50`) but never read — `registerLead` hardcodes `source: LeadSource.WEBSITE` (`lead.service.ts:60`), making the field and the `LeadSource.ADMIN_PANEL` enum member (`lead-source.enum.ts:3`) effectively dead. The `user?: LoggedInUser` parameter on `registerLead`/`verifyLeadOtp` (`lead.service.ts:41-42`, `73-76`) is accepted but never used — and since both endpoints are `@Public()`, `req.user` is never populated anyway, so the parameter is inert.

## 2. Intended / Ideal Usage
- Correct usage is a full lead-to-customer funnel: public capture → OTP verification → automatic conversion into a `User` (this part exists), with the resulting `User` traceably linked back to the originating `Lead` for attribution/reporting, and with the sales/platform-admin team notified in real time so they can work the pipeline (`lead_status` progression from `NEW` through `WON`/`LOST`, `lead-status.enum.ts:1-12`) without having to poll the admin list.

## 3. Cross-Module Connections
- Depends on: `NotificationService` (`../../notification/notification.module`, `lead.module.ts:11,17`) to send the OTP email; `OtpRepository` from the `auth` module (`lead.service.ts:13`, reused directly rather than via an `AuthService` abstraction); `User` entity from the `user` module, written to directly via `manager.getRepository(User)` inside `verifyLeadAndCreateUser` (`lead.service.ts:234-245`); `BaseEntity` (`common/entities/base.entity.ts`) for `enterprise_id`/`organization_id`/soft-delete columns.
- Depends on it: the platform-admin lead pipeline UI (`brello_webapp/src/features/platform/leads/*`, `PlatformLeadsPage.tsx`) and the public signup UI (`brello_webapp/src/features/auth/api/auth.ts`).
- Missing/expected connection confirmed: there is no notification to platform admins when a new lead is captured. `lead.service.ts:179-184` only sends a `NotificationType.EMAIL` to `target_email: email` (the lead's own address) to deliver the OTP — there is no second `notificationService.send(...)`/`broadcastAllChannels(...)` call targeting platform-admin users, and no corresponding entry exists in the event catalog: `src/common/enums/notification-event-type.enum.ts` enumerates `LEAVE_*`, `REIMBURSEMENT_*`, `ATTENDANCE_*`, `PAYROLL_REMINDER`, `EMPLOYEE_*`, `BILLING_*`, `AUTH_OTP`, `LETTER_*` — no `LEAD_CAPTURED`/`LEAD_CREATED` event type. This confirms the gap: admins only discover new leads by manually opening the `GET /leads` list.
- Also missing: no link from the created `User` back to the source `Lead` (no `lead_id` column on `User`, verified via repo-wide grep) — conversion analytics can only be reconstructed indirectly by matching email addresses between the two tables.

## 4. Gaps

### Structural
- No platform-admin RBAC on admin endpoints: `LeadController` (`lead.controller.ts:27-30`) applies only `@UseGuards(JwtAuthGuard)` at the class level; `GET /leads`, `GET /leads/:id`, and `PATCH /leads/:id/status` have no `PlatformAdminGuard`, unlike comparable platform-admin controllers, e.g. `platform-department.controller.ts:24` (`@UseGuards(JwtAuthGuard, PlatformAdminGuard)`). This matters because any authenticated user of any tenant — not just platform admins — can list every lead's email/phone/location/device and change pipeline status.
- No conversion linkage between `Lead` and `User`: `verifyLeadAndCreateUser` (`lead.service.ts:227-245`) creates a `User` from lead fields but stores no reference back to `lead.id`; matters because it makes lead-to-customer conversion reporting (the module's stated ideal use) unimplementable without fragile email-matching joins.
- OTP handling is duplicated ad hoc rather than delegated to a shared auth/OTP service: `LeadService` directly injects and drives `OtpRepository` (`lead.service.ts:13,33,161-225`) with its own hash/expiry/attempt logic, duplicating what should likely be a shared `OtpService` used by both `auth` and `lead` modules — a layering/reuse smell rather than a true dependency inversion.

### Coding
- Dead validation: `CreateLeadDto.source` (`create-lead.dto.ts:47-50`) is validated but discarded — `registerLead` always sets `LeadSource.WEBSITE` (`lead.service.ts:60`), so `LeadSource.ADMIN_PANEL` (`lead-source.enum.ts:3`) can never be produced by any code path.
- Unused parameters with inconsistent convention: `registerLead`/`verifyLeadOtp` accept `user?: LoggedInUser` and never use it (`lead.service.ts:41-42,73-76`), while the admin methods correctly prefix the unused parameter as `_user` (`lead.service.ts:96,104,115`) — the public-endpoint methods should either drop the parameter or follow the same underscore convention.
- Debug OTP logging left in: `lead.service.ts:186-187` logs the plaintext OTP (`this.logger.warn('[DEV] OTP for ${email}: ${otp}')`) with a `// TODO: Remove OTP logging before production` comment directly above it — matters because this is a plaintext-secret log line still active in the code path used by the public registration endpoint.
- Dev OTP bypass hardcoded in production code path: `lead.service.ts:214-216` accepts OTP `'123456'` whenever `configService.get('brello.environment') === 'dev'` — a magic bypass value baked into shared logic rather than isolated to test tooling; if the `dev` environment flag is ever misconfigured in a reachable deployment, it becomes a universal OTP.
- Race condition on duplicate email: `ensureEmailNotTaken` (`lead.service.ts:127-132`) does a read-then-write check; between the `findByEmail` check and the `leadRepository.create` call (`lead.service.ts:48-65`) a concurrent request can pass the same check, and the resulting unique-constraint violation on `email` (`lead.entity.ts:8-10`) is not caught, so it would surface as an unhandled 500 instead of the intended `ConflictException`.
- No validation that `plan_id` (required, `create-lead.dto.ts:60-62`) actually references an existing plan — `registerLead` stores it as-is (`lead.service.ts:63`) with no lookup against a plans repository/service, so a bad or stale `plan_id` silently persists on both the `Lead` and later the converted `User` (`lead.service.ts:242`).

### Technical
- Public, unauthenticated endpoints with no rate limiting: `POST /leads/register` and `POST /leads/verify-otp` are both `@Public()` (`lead.controller.ts:33,43`), and a repo-wide search found no `Throttler`/`@Throttle` usage anywhere in `brello_server` — there is no rate limiting or CAPTCHA on either endpoint, meaning both the signup form (spam lead creation, mass emailing via the OTP send) and the OTP verification endpoint (brute-forcing the 6-digit code, bounded only by the `maxAttempts` counter in `validateAndConsumeOtp`, `lead.service.ts:208-212`) are exposed to abuse at the infrastructure level, not just the application-level attempt cap.
- PII exposure via missing RBAC (also a security issue, cross-referenced from Structural above): full lead PII (email, phone, location, device, password_hash — though password_hash is stripped in `findAll`/`findOne`, `lead.service.ts:99,108`) is readable by any authenticated user given the missing `PlatformAdminGuard`.
- No test coverage: no `.spec.ts` files exist alongside any file in `src/modules/lead/` (verified via directory listing), so none of the OTP/transaction/conflict logic has automated regression coverage.
- No structured/observable metrics: the only signals are `Logger.log/warn/error` calls (`lead.service.ts:28,44,69,77,89,187,248-251`); there is no counter/event for conversion rate, OTP failure rate, or lead-source breakdown, so the admin team has no way to observe funnel health beyond manually reading the leads list.

## 5. Top 3 Priorities
1. Add `PlatformAdminGuard` to `GET /leads`, `GET /leads/:id`, `PATCH /leads/:id/status` — right now any logged-in user of any tenant can view and mutate the entire sales pipeline and lead PII, which is the most severe and cheapest-to-fix gap.
2. Add rate limiting (and ideally CAPTCHA) to `POST /leads/register` and `POST /leads/verify-otp` — both are unauthenticated public endpoints with no throttling anywhere in the codebase, exposing them to spam signups, OTP-triggered email flooding, and OTP brute-forcing.
3. Emit a platform-admin-facing "new lead captured" notification (and add a `LEAD_CAPTURED` entry to `NotificationEventType`) alongside the existing OTP email in `generateAndSendOtp` (`lead.service.ts:179-184`) — today admins have zero push signal that a lead exists and must poll the list manually, undermining the pipeline's usefulness.
