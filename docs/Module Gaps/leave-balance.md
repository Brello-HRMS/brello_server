# Module: Leave Balance

## 1. Purpose & Current Usage

- **Entities**: `LeaveBalance` (`entities/leave-balance.entity.ts`) — one row per `employee_id` + `leave_type_id` + `leave_year`, storing `allocated_days`, `accrued_days`, `carry_forward`, `adjustment`, `used_days`, `pending_days`, and an `is_unlimited` flag for Loss-of-Pay (LWP). `LeaveBalanceLedger` (`entities/leave-balance-ledger.entity.ts`) — append-only audit trail of every credit/debit against a balance (`entry_type`, `direction`, `days`, `running_balance`).
- **Service** (`services/leave-balance.service.ts`, 1051 lines): initializes yearly balances (`initializeForEmployee`, `bulkInitialize`), reads (`getMyBalance`, `getBalanceForEmployee`, `listBalances`, `getBalanceById`, `getLedger`), and mutates (`updateBalance` for reallocation, `adjustBalance` for manual credit/debit, `recompute` to resync `used_days`/`pending_days` from leave requests). It also exposes two primitives used by other modules: `ensureBalanceForRequest` (get-or-create a balance row under a pessimistic row lock) and `writeLedger` (append a ledger entry) — see `services/leave-balance.service.ts:662` and `:719`.
- **Controller** (`controllers/leave-balance.controller.ts`): REST surface at `/leave-balances` — `GET /me`, `GET /employee/:employeeId`, `POST /initialize`, `POST /initialize/bulk`, `GET /`, `GET /:id`, `GET /:id/ledger`, `PATCH /:id/adjust`, `POST /:id/recompute`, `PATCH /:id`. All mutation endpoints are wrapped in `@AuditLog(...)`.
- **Who calls into it today**: `modules/leave-request/services/leave-request.service.ts` is the only external consumer, calling `ensureBalanceForRequest` + `writeLedger` at submit (`holdBalance`, line 926), approve (line 686), and release/cancel/reject paths (`releasePending` line 962, `releaseApproved` line 996). On approval it also directly mutates `LeaveBalance.pending_days`/`used_days` via `manager.update` (`leave-request.service.ts:659`) rather than going through a `LeaveBalanceService` write method — the balance table is written from two different modules.
- **Dead/unused parts**:
  - `LeaveBalanceRepository.list()` (`repositories/leave-balance.repository.ts:150-239`) and its `BalanceListRow`/`toListRowView` counterpart in the service (`services/leave-balance.service.ts:997-1049`) are fully implemented but never called — `listBalances` uses `listEmployeesWithBalances` instead. ~90 lines of dead repository/service code.
  - `LedgerEntryType.MONTHLY_ACCRUAL` and `LedgerEntryType.LAPSE` (`enums/ledger.enum.ts:4,9`) are defined but never referenced anywhere in the codebase (verified via repo-wide grep) — no code path produces these ledger entries.

## 2. Intended / Ideal Usage

- Yearly balances should be seeded once (`initializeForEmployee`/`bulkInitialize`), then **kept current automatically**: monthly-accrual leave types should top up `accrued_days` every month, unused balances should carry forward (capped per policy) or lapse at year-end, and low/near-zero balances should proactively notify the employee.
- Debits (hold on submit, consume on approve) and credits (release on cancel/reject) should be strictly serialized per `(employee_id, leave_type_id, leave_year)` so concurrent requests can never push `available_days` negative, and the ledger should be the single source of truth reconciled by `recompute`.
- Cross-module writes to `LeaveBalance` should route exclusively through `LeaveBalanceService` so business rules (locking, ledger writes, non-negative invariant) are enforced in one place rather than duplicated by callers.

## 3. Cross-Module Connections

- **Depends on**: `leave-config` (`LeaveConfig`, `LeaveType` — policy: `days`, `accrual` enum `none|monthly`, `is_paid`; `services/leave-balance.service.ts:12-14,92-95`), `user` (`User` entity, active-employee checks), `leave-request` (`LeaveRequest`/`LeaveRequestStatus`, used to aggregate `used`/`pending` days for LWP synthesis and `recompute`, `services/leave-balance.service.ts:30-31`), and `audit` (`AuditContextService` for diffing pre/post values).
- **Depended on by**: `leave-request` module only (see §1). `LeaveBalanceModule` exports `LeaveBalanceService` (`leave-balance.module.ts:33`) specifically for this consumer.
- **Missing/expected connections**:
  - **No `NotificationService` reference anywhere in `leave-balance/` or `leave-request/`** (verified: `grep -rl NotificationService modules/leave-balance modules/leave-request` returns nothing). The PRD (`docs/prd/notification-system.md:135-136`) specifies two triggers this module should originate: "Leave balance running low (< 2 days remaining)" (weekly Monday cron) and "Leave balance credited (accrual / manual)" — neither exists.
  - **No `@Cron` job for this module at all.** Repo-wide search for `@Cron` finds jobs only in `billing`, `org-setup`, `attendance`, `payroll`, and `user` (offboarding) — nothing under `leave-balance` or `leave-request`. This confirms the prior audit finding: the weekly low-balance reminder cron does not exist, and neither does any monthly-accrual or year-end-lapse cron (consistent with `MONTHLY_ACCRUAL`/`LAPSE` being dead enum values, see §1).
  - `leave-config`'s `LeaveType`/`LeaveRules` entities have no carry-forward cap, no encashment field, and no lapse/expiry rule (`entities/leave-type.entity.ts`, `entities/leave-rules.entity.ts`) — so `computeAutoCarryForward` (`services/leave-balance.service.ts:864-882`) carries forward 100% of the prior year's remaining balance uncapped, because there is no policy field to cap it against.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **Balance mutated from outside the owning service.** `leave-request.service.ts:659-663` and similarly at lines 940-944, 976-980, 1010-1014 call `manager.update(LeaveBalance, ...)` directly on `pending_days`/`used_days` instead of through a `LeaveBalanceService` method; only the ledger write goes through `writeLedger`. This means the non-negative/locking invariants enforced in `adjustBalance` (see below) are not guaranteed for the request-driven debit/credit path — two different code paths update the same columns with different levels of safety.
- **Dead repository/service code**: `LeaveBalanceRepository.list()` (~90 lines, `repositories/leave-balance.repository.ts:150-239`) and `LeaveBalanceService.toListRowView` (`services/leave-balance.service.ts:997-1049`) are unreachable — `listBalances` calls `listEmployeesWithBalances` instead. This doubles the maintenance surface for the same feature with no benefit.
- **No accrual/lapse abstraction.** `LedgerEntryType.MONTHLY_ACCRUAL` and `LAPSE` exist in the enum (`enums/ledger.enum.ts:4,9`) signaling the module was designed to support periodic accrual and year-end expiry, but no service method, cron, or scheduler produces them — the abstraction is half-built.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling)

- **`accrued_days` is frozen at creation time and never re-accrued.** `computeAccruedDays` (`services/leave-balance.service.ts:884-895`) computes a pro-rated snapshot (`monthsElapsed/12 * annualDays`) only inside `initializeForEmployee`/`ensureBalanceForRequest`, at the moment the balance row is first created. `recompute` (`:596-658`) only resyncs `used_days`/`pending_days` from leave requests — it never touches `accrued_days`. For a `monthly`-accrual leave type, an employee whose balance row is created in January is permanently stuck with ~1/12 of their annual entitlement for the rest of the year unless an admin manually calls `adjustBalance`/`updateBalance`. This makes the "monthly accrual" policy non-functional after initial creation.
- **TOCTOU race on leave-request submission can overdraw a balance.** The sufficiency check in `leave-request.service.ts:1196-1209` (`validateForSubmit`) reads the balance via `fetchBalance` *before* any transaction/lock is taken. The actual hold (`holdBalance`, `:926-960`) later acquires a pessimistic row lock via `ensureBalanceForRequest` but **does not re-check `available >= totalDays` after acquiring the lock** — it unconditionally adds `ctx.totalDays` to `pending_days`. Two concurrent submissions for the same employee/leave-type can both pass the pre-check against stale data, then both commit their holds serially, producing `pending_days` that exceeds `available_days` (i.e., a negative balance once both are later approved). Contrast with `adjustBalance` (`services/leave-balance.service.ts:536-593`), which correctly locks first and recomputes `projectedAvailable` under the lock before writing.
- **`adjustBalance` days-increment validation is fragile.** `dto.days % 0.5 !== 0` (`services/leave-balance.service.ts:532`) is a floating-point modulo check on user input (e.g. `2.3 % 0.5` due to binary floating point can misbehave for some values), which is an unreliable way to validate half-day increments.

### Technical (performance, security, scalability, test coverage, observability/logging)

- **Zero automated test coverage.** No `*.spec.ts` or `*.test.ts` file exists anywhere under `modules/leave-balance` (confirmed via repo-wide filename search) for entities, repositories, or the 1000+ line service that contains the balance-arithmetic and locking logic described above.
- **N+1 query pattern in list endpoints.** Both `getBalanceForEmployee`/`listBalances` loop over employees and, per employee, call `findForEmployee` plus `findLwpLeaveTypes` plus (per LWP type) `aggregateRequestsForLwp` (`services/leave-balance.service.ts:349-374`) — for a paginated list of 20 employees this is potentially dozens of extra queries per page, and `findLwpLeaveTypes` re-runs an identical query for every employee in the loop instead of being hoisted out.
- **No notification hooks / no cron observability.** Because there's no `NotificationService` call and no `@Cron` job (see §3), there's also nothing to log, alert on, or monitor for the low-balance/accrual/lapse features — the gap is invisible operationally, not just functionally.

## 5. Top 3 Priorities

1. **Fix the TOCTOU hold race** (`leave-request.service.ts:926-960`) by re-validating `computeAvailable` against `ctx.totalDays` *after* the pessimistic lock is acquired in `holdBalance`, mirroring the pattern already used correctly in `adjustBalance` — this is a correctness bug that can silently push balances negative today.
2. **Make monthly accrual actually recur** — add a scheduled job (there is precedent in `billing/crons`, `payroll/services/*.cron.ts`) that re-derives `accrued_days` for `accrual = 'monthly'` balances, since currently it is computed once and frozen forever (`services/leave-balance.service.ts:884-895`, `recompute` never touches it).
3. **Wire up the two PRD-specified notification triggers** (weekly low-balance reminder, credit notification) per `docs/prd/notification-system.md:135-136` — currently there is no `NotificationService` reference and no `@Cron` anywhere in `leave-balance`/`leave-request`, so this PRD-committed feature is entirely unbuilt.
