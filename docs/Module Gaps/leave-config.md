# Module: Leave Config

## 1. Purpose & Current Usage

- Defines organization-level leave policy: `LeaveConfig` (total_leave + status draft/active, `brello_server/src/modules/leave-config/entities/leave-config.entity.ts:1-20`), `LeaveType` (name/days/accrual/half-day/is_paid, `entities/leave-type.entity.ts:1-33`), and `LeaveRules` (approval_required, max_per_month, allow_backdated/max_backdated_days, sandwich_rule, `entities/leave-rules.entity.ts:1-32`). Together they back the HR "stepper" wizard described in the PRD (leave year → total leave → leave type allocation → rules → review).
- `LeaveConfigService` (`services/leave-config.service.ts`) exposes: `createDraft` (line 27), `updateConfig` with a transactional replace-strategy for leave types (line 49, delegates to `LeaveConfigRepository.updateWithRelations` at `repositories/leave-config.repository.ts:47-88`), `activateConfig` with allocation/backdated/max-per-month validation (line 104), `findOne` (line 154) and `findCurrent` (line 170).
- `LeaveConfigController` (`controllers/leave-config.controller.ts:25-80`) exposes `POST /leave-configs`, `GET /leave-configs/current`, `GET /leave-configs/:id`, `PUT /leave-configs/:id`, `POST /leave-configs/:id/activate`, all gated by `RequirePermission('LEAVE_SETUP', ...)` and audit-logged via `@AuditLog(AuditLogModule.LEAVE_CONFIG, ...)`.
- **Nobody outside this module calls `LeaveConfigService` or `LeaveConfigRepository`.** `leave-request`, `leave-balance`, and `org-setup` all re-declare `TypeOrmModule.forFeature([LeaveConfig, LeaveType, LeaveRules])` in their own modules (`leave-request/leave-request.module.ts:9-11,21-27`, `leave-balance/leave-balance.module.ts:9-10,18-23`, `org-setup/org-setup.module.ts:25`) and inject raw `Repository<LeaveConfig>` / `Repository<LeaveType>` directly (`leave-request/services/leave-request.service.ts:70-71`, `leave-balance/services/leave-balance.service.ts:92-93`, `org-setup/org-setup.service.ts:42-43`). The controller/service/repository trio in this module is therefore only used by the admin-facing config wizard, and read access to the same entities is duplicated three more times elsewhere.
- No DELETE endpoint exists; `BaseEntity`'s `deleted_at`/`deleted_by` columns are present but never referenced anywhere in this module (no soft-delete logic in `leave-config.repository.ts`).

## 2. Intended / Ideal Usage

- `brello_server/docs/prd/leave-config.md` documents only the front-end stepper UX (5 steps: leave year, total leave, leave type allocation, rules, review/save) and explicitly lists "Policies, Policy assignment, Unpaid leave config, Future leave limit, Weekly off/holiday toggles" as **out of scope** (`docs/prd/leave-config.md:359-367`).
- The PRD has no backend/API contract, no mention of multi-org lifecycle (draft→active transitions, what happens to a prior ACTIVE config on re-activation), and no mention of a "leave year" field at all even though Step 1 of the UX is entirely about selecting a leave cycle month — the `LeaveConfig` entity has no field for it (only `total_leave`); `leave-request.service.ts` independently assumes a **calendar-year (Jan 1–Dec 31) cycle** (`leave-request/services/leave-request.service.ts:1054-1061`), not the "Apr–Mar" style configurable cycle the PRD describes. Implementation and PRD have diverged on a core concept.
- PRD Step 1 (leave year/cycle) and its "Edit Flow" special case (reducing total leave below current allocation) have no corresponding backend validation — `activateConfig` only checks equality of total vs allocated (line 117), not the reduce-below-allocation re-edit scenario.

## 3. Cross-Module Connections

- **Depends on:** `RbacModule` (permission checks), `AuditModule` (`AuditContextService`, `@AuditLog` decorator), `common/entities/base.entity.ts` for org/enterprise scoping and status column.
- **Depends on it (structurally, via entity import, not via service):** `leave-request` module, `leave-balance` module, `org-setup` module (dashboard setup-completion counter, `org-setup/org-setup.service.ts:66`).
- **Missing/expected connection:** `leave-request` and `leave-balance` do read the real `LeaveConfig`/`LeaveType`/`LeaveRules` rows (not hardcoded duplicate values), so the data itself isn't forked — but the **access path and "which config is current" semantics are forked**. `LeaveConfigRepository.findCurrent` (`repositories/leave-config.repository.ts:28-34`) picks the row with the latest `created_at` **regardless of status**, which is what backs `GET /leave-configs/current` used by the admin UI. Meanwhile both `leave-request.service.ts:1281-1293` (`requireActiveConfig`) and `leave-balance.service.ts:753-765` (`requireActiveConfig`, byte-for-byte duplicated logic) independently query `where: { organization_id, status: Status.ACTIVE }`. If an HR user starts (but never activates) a new draft after an existing ACTIVE config, `GET /current` returns the unfinished draft while leave-request/leave-balance keep enforcing the old ACTIVE config — the two code paths can legitimately disagree on "the current policy."
- `approval_required` is read out of `LeaveRules` and copied onto the `LeaveRequest` snapshot (`leave-request/services/leave-request.service.ts:1217`) but is **never branched on** — every submitted request always goes to `PENDING` and requires manual approval (`leave-request.service.ts:126-128`, `156-157`). The flag is persisted but has no runtime effect, i.e. leave-request does not actually enforce this particular policy rule.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)
- **This module's service/repository layer is dead code for every consumer except its own controller.** `leave-request.module.ts:9-11,21-27` and `leave-balance.module.ts:9-10,18-23` both re-register `TypeOrmModule.forFeature` for `LeaveConfig`/`LeaveType`/`LeaveRules` and inject raw repositories instead of importing `LeaveConfigModule`/`LeaveConfigService` (which is exported at `leave-config.module.ts:25` for exactly this purpose). This means any future validation, caching, or business rule added to `LeaveConfigService` (e.g. tenant-safety, single-active-config enforcement) will silently not apply to leave-request/leave-balance/org-setup unless each is updated in parallel.
- **`requireActiveConfig` is duplicated verbatim in two other modules** (`leave-request/services/leave-request.service.ts:1281-1293` and `leave-balance/services/leave-balance.service.ts:753-765`) instead of living once in `LeaveConfigService`/`LeaveConfigRepository`. Any change to "what counts as the active config for an org" has to be made in 3 places (this module's own `findCurrent` is a 4th, differing definition).
- **No entity/service concept of "one active config per org."** Nothing in `activateConfig` (`services/leave-config.service.ts:104-149`) deactivates a previous ACTIVE config before setting the new one to ACTIVE, and there's no unique/partial-index constraint found in the schema for `(organization_id) WHERE status = 'ACTIVE'`. Two ACTIVE configs for the same org would make `leave-request`'s and `leave-balance`'s `findOne({ where: { organization_id, status: ACTIVE } })` non-deterministic (TypeORM returns an arbitrary match).

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)
- **`approval_required` is stored but never enforced**: leave requests always start `PENDING` regardless of the flag (`leave-request/services/leave-request.service.ts:126-128`); the field is only echoed onto the request snapshot at line 1217. Dead policy field from the consumer's perspective.
- **`GET /leave-configs/current` can return a non-active draft.** `findCurrent` (`repositories/leave-config.repository.ts:28-34`) has no `status` filter, unlike every consumer's own "active config" lookup — inconsistent and likely to surface a stale/incomplete draft in the admin UI's "current policy" view.
- **No soft-delete support despite inherited columns.** `deleted_at`/`deleted_by` from `BaseEntity` are never set or filtered anywhere in `leave-config.repository.ts`, and there is no DELETE endpoint on the controller — leave configs are effectively permanent once created, with abandoned drafts left in the table indefinitely (compounding the `findCurrent` issue above).
- **`CreateLeaveConfigDto` imports unused validators** `IsInt`, `Max` (`dto/create-leave-config.dto.ts:1`) and `UpdateLeaveConfigDto` imports unused `IsInt`, `Max` as well (`dto/update-leave-config.dto.ts:1-4`) — minor lint/dead-import noise, but signals the DTOs weren't cleaned up after edits.
- **No leave-year/cycle field on `LeaveConfig`**, contradicting the PRD's Step 1 (leave year selection, `docs/prd/leave-config.md:58-77`) — `total_leave` is the only config-level column (`entities/leave-config.entity.ts:8-9`); `leave-request.service.ts` hardcodes a Jan–Dec calendar year instead (line 1055-1056).

### Technical (performance, security, scalability, test coverage, observability/logging)
- **No test files found for this module** (no `*.spec.ts` under `leave-config/`, confirmed via repo-wide search) — `activateConfig`'s validation logic (allocation match, backdated config, max-per-month) and the transactional `updateWithRelations` replace-strategy have zero automated coverage despite being the module's most business-critical paths.
- **`updateWithRelations`'s LeaveType REPLACE strategy deletes-then-reinserts on every update** (`repositories/leave-config.repository.ts:60-69`), generating new UUIDs for every leave type on every save; any other table that stored a foreign key to a specific `LeaveType.id` (e.g. balances/ledger keyed by `leave_type_id`) would be silently orphaned across an edit — worth checking against `leave_balance.leave_type_id` foreign keys before this module's edit flow is used post-activation.
- Only `Logger.log` calls exist (`services/leave-config.service.ts:39-41,146-148`); no warn/error logging on the `NotFoundException`/`BadRequestException` paths, making failed activations harder to trace in production logs.

## 5. Top 3 Priorities

1. **Route `leave-request`, `leave-balance`, and `org-setup` through `LeaveConfigService`/`LeaveConfigRepository` instead of raw TypeORM repositories on the same entities.** This is the single biggest gap — three modules bypass the module's own abstraction, duplicating "find the active config" logic (with three different definitions of "current"/"active") and making the module's service layer effectively dead code for everything except its own CRUD API.
2. **Fix `findCurrent` to filter by `status = ACTIVE`** (or rename/clarify it as "latest draft or active") so the admin UI's "current config" view can't silently diverge from what `leave-request`/`leave-balance` are actually enforcing.
3. **Enforce `approval_required` in the leave-request submission flow** (or remove the field if auto-approval was intentionally descoped) — right now it's collected in the config wizard, persisted per-request, but has zero effect on request routing, which is a policy rule defined in this module but not honored by its consumer.
