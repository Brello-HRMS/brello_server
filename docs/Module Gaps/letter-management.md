# Module: Letter Management

## 1. Purpose & Current Usage

The module lives at `brello_server/src/modules/letter-management/` and implements an HR letter-issuance pipeline across 6 subfolders:

- **`categories/`** — named buckets for grouping templates (e.g. "Offer Letter"). Entity `categories/entities/letter-category.entity.ts:4-10` (`LetterCategory extends BaseEntity`, adds only `name`). Service `categories/services/letter-category.service.ts` handles create/find/update/archive, blocking archive while PUBLISHED templates reference the category (`:79-84`). Controller at `categories/controllers/letter-category.controller.ts`, routes under `/letter-management/categories`.
- **`templates/`** — letter content authoring: heading/paragraphs/bullets + optional salary table + optional signatory, with `DRAFT → PUBLISHED → ARCHIVED` lifecycle. Entity `templates/entities/letter-template.entity.ts:10-53` (`category_id` FK `RESTRICT`, `signatory_id` FK `SET NULL`, `variables: string[]` cache, `version`, `template_status`). Service `templates/services/letter-template.service.ts` — `create` (`:82-105`), `update` bumps `version` only once PUBLISHED (`:149-159`), `publish` (`:166-192`), `duplicate` (`:194-212`), `preview` (`:252-274`).
- **`signatories/`** — named people (name + designation) with an uploaded signature image reference. Entity `signatories/entities/signatory.entity.ts:6-18` (`signature_document_id`, `is_default`). Service `signatories/services/signatory.service.ts` — `create` uploads via `DocumentService.uploadDocument(..., FolderType.LETTER_SIGNATURE)` (`:28-57`), `update` has no file param (`:76-96`), `setDefault`, `archive` (blocks archiving current default, `:124-128`).
- **`settings/`** — per-org numbering config + defaults. Entity `settings/entities/letter-settings.entity.ts:6-21` (`letter_prefix` default `'BRLO'`, `current_year`, `last_sequence`, `default_signatory_id`, `date_format`, unique `organization_id`). Service `settings/services/letter-settings.service.ts` lazily creates the row on first `get()` (`:17-28`). Repository `settings/repositories/letter-settings.repository.ts` has the transactional pair `lockByOrganizationId` (`:60-69`, pessimistic write) and `createWithManager` (`:77-82`) used by numbering, plus a separate, non-transactional `findOrCreateForOrg` (`:19-40`).
- **`issued-letters/`** — the generation pipeline + immutable issuance record + employee self-service. Entity `issued-letters/entities/issued-letter.entity.ts:20-92` — full point-in-time snapshot columns (`variable_snapshot`, `heading_snapshot`, `paragraphs_snapshot`, `bullets_snapshot`, `salary_snapshot`, `signatory_snapshot`), `pdf_document_id`, `idempotency_key` (unique per org, `:24-27`), `delivery_status` (`ISSUED|VIEWED|ACKNOWLEDGED`). Service `issued-letters/services/issued-letter.service.ts` — `generate` (`:80-197`, one DB transaction: reserve number → render PDF → upload PDF → insert row → fire-and-forget notify + audit), `findMineById` promotes `ISSUED→VIEWED` (`:223-240`), `acknowledgeMine` promotes to `ACKNOWLEDGED` (`:285-325`). Controller exposes org-admin routes (RBAC `LETTER_ISSUED`) and employee `me/*` routes (JWT-only, no RBAC, by design).
- **`shared/`** — cross-cutting services: `variable-registry.ts` (closed set of `{{placeholder}}` keys), `variable-resolver.service.ts` (resolves registry vars from User/Payroll/Organization/Signatory/Settings), `render-model-builder.service.ts` (single substitution implementation, intended to be shared by preview and real generation), `pdf-builder.service.ts` (pdfkit rendering), `numbering.service.ts` (row-locked sequence reservation), `formatters.ts` (`formatDate`/`formatCurrency`), and `seed/starter-letter-content.ts` (starter categories/templates seeding — **just implemented this session, wired into `organization.service.ts:45,288`; not a gap**).

**Callers today**: Backend — only `organization.service.ts:45,288` imports from the module (the new starter-content seed call, excluded from gap analysis). No other backend module imports `letter-management`; `IssuedLettersModule` exports `IssuedLetterService` (`issued-letters/issued-letters.module.ts:26`) but nothing else in the backend imports it. Frontend — the entire `brello_webapp/src/features/letter-management/*` surface plus `pages/letter-management/*` (Categories, Templates, TemplateEditor, Signatories, LetterSettings, LetterSetup, IssuedLetters pages) and `pages/employee-letters/MyLettersPage.tsx`. No other frontend feature references it.

**Dead/unused parts (verified)**:
- Signature image dead-code claim **confirmed**: `shared/services/pdf-builder.service.ts:105-110` prints only `model.signatory.name`/`.designation` as text; `SignatoryModel` (`shared/interfaces/render-model.interface.ts:16-19`) carries no image field at all; `VariableResolverService.getSignatory` (`shared/services/variable-resolver.service.ts:87-101`) never reads `signature_document_id`; no `doc.image(...)` call exists anywhere in the module. `signature_document_id` is written once at signatory creation (`signatories/services/signatory.service.ts:35-39,51`) and never read back for rendering — fully vestigial.
- Archived-signatory resolution claim **confirmed**: `signatories/repositories/signatory.repository.ts:42-49` `findOneByOrg` applies no `status` filter (contrast `findDefaultForOrg` at `:56-64`, which does filter `status: Status.ACTIVE`). Used unfiltered both for template assignment (`templates/services/letter-template.service.ts:70-80`) and for resolution at generation time (`variable-resolver.service.ts:87-101,92`). Frontend only filters archived signatories out of the dropdown for *new* assignments (`SignatoryFormModal`/`TemplateForm.tsx:53-58`) — cosmetic, not enforced server-side.
- Additional dead code found: unused `PreviewLetterTemplateDto` with an `employee_id` field that no controller/service/frontend call ever populates (`templates/dto/letter-template.dto.ts:89-93`); `issued_letters.archived_at` column declared but never set or queried anywhere (`issued-letters/entities/issued-letter.entity.ts:75`); `formatDate`'s `_pattern` argument is accepted but ignored — always renders `DD MMM YYYY` regardless of the org's configured `date_format` (`shared/utils/formatters.ts:7`).

## 2. Intended / Ideal Usage

A correct HR letter-issuance flow: Categories group letter types → Signatories (with real signature images) are defined once and kept current → Templates are authored per category with `{{variable}}` placeholders drawn from a closed, sufficiently broad registry, validated at save/publish, and must be explicitly published before use → Settings hold per-org numbering/defaults, lazily bootstrapped → Generation resolves every variable from Employee/Payroll/Organization data, reserves a number under a row lock, renders a PDF exactly once, and freezes a full field-level snapshot so the legal record survives later edits to template/employee/org data → the employee gets self-service list/view/download/acknowledge access, with view/acknowledge events flowing back to HR → every mutating action is audit-logged. The immutability-via-snapshot design and the numbering row-lock are already solid; where the implementation falls short of this intent is signatory lifecycle enforcement, real signature-image rendering, and variable-registry breadth for non-offer letter types (see Gaps).

**Depends on this module:** `organization.service.ts:45,288` (starter-content seeding, not a gap); the full `brello_webapp` letter-management + employee-letters UI. Nothing else in the backend consumes it — it is currently a leaf feature with no downstream automation.

**Missing/expected connections:**
- No automatic trigger from HR lifecycle events into letter generation — e.g. no code in `employee`/`payroll`/offboarding flows calls `IssuedLetterService.generate` when an employee is relieved or a salary revision is approved; generation is a fully manual admin action (`issued-letters/controllers/issued-letter.controller.ts:60-70`), yet the starter content already ships Relieving/Experience/Promotion/Increment templates (`shared/seed/starter-letter-content.ts:100-176`) that imply this connection was intended.
- `settings/dto/letter-settings.dto.ts:9-11` accepts `default_signatory_id` with only `@IsUUID()` — `settings/services/letter-settings.service.ts:30-45` never validates the id exists or is active against `SignatoryRepository`, unlike `templates/services/letter-template.service.ts:70-80` which does validate signatory existence for template assignment. Inconsistent cross-module validation within the same feature.

## 4. Gaps

### Structural

- **Cross-module entity reach-through bypassing module boundaries.** `categories/repositories/letter-category.repository.ts:73-77` (`countPublishedTemplatesByCategory`) calls `this.repo.manager.getRepository(LetterTemplate)` directly instead of depending on `LetterTemplatesModule`, and `LetterTemplatesModule` is never imported by `categories/letter-categories.module.ts`. Matters because any future change to `LetterTemplate`'s schema or DI registration can silently break categories with no compiler-level signal.
- **`LetterTemplateService.preview()` bypasses the shared render pipeline it was built to reuse.** `templates/services/letter-template.service.ts:20-33,252-274` maintains its own hardcoded `SAMPLE_VALUES` and inline regex substitution instead of delegating to `RenderModelBuilderService.substitute()` (`shared/services/render-model-builder.service.ts:16-19,22-24`), which explicitly documents the goal of "preview and real generation can never drift." Matters because any future substitution-semantics change (escaping, formatting) has to be made in two places or preview silently diverges from what actually gets issued.
- **No distinct abstraction for "check a signatory exists" vs "resolve a signatory for legal-document rendering."** Both authorization-style lookups and generation-time resolution reuse the same unfiltered `SignatoryRepository.findOneByOrg` (`signatories/repositories/signatory.repository.ts:42-49`), which is the structural root cause of the archived-signatory rendering gap below. Matters because there's no single place to fix the archival-status semantics without touching every caller.
- **`issued_letters.archived_at` is schema debt with no feature behind it** (`issued-letters/entities/issued-letter.entity.ts:75`) — never set or queried anywhere in the repository or service. Matters because it signals an abandoned "archive an issued letter" feature and adds confusion for future maintainers reading the entity.

### Coding

- **Signature image collected but never rendered (dead field, confirmed).** `shared/services/pdf-builder.service.ts:105-110` only prints `model.signatory.name`/`.designation`; `SignatoryModel` (`shared/interfaces/render-model.interface.ts:16-19`) has no image field; no `doc.image(...)` call exists in the module. Matters because HR letters typically require a visible signature, and the upload step (`signatories/services/signatory.service.ts:28-57`) misleads admins into thinking it's used.
- **Signatures can't be replaced after creation (confirmed, backend + frontend).** Backend: `signatories/dto/signatory.dto.ts:32-47` (`UpdateSignatoryDto`) has no file field; `signatories/controllers/signatory.controller.ts:75-84` (`PATCH :id`) has no `FileInterceptor`, unlike `POST` (`:35-41`); `signatories/services/signatory.service.ts:76-96` never calls `DocumentService`. Frontend: `SignatoryFormModal.tsx:153` wraps the entire upload block in `{!isEdit && (...)}`. Matters because a signatory whose designation changes, or whose signature image was wrong/expired, can never be corrected without deleting and recreating the record (losing history/references).
- **Archived signatories remain assignable and resolvable at generation time (confirmed).** `signatories/repositories/signatory.repository.ts:42-49` `findOneByOrg` has no `status` filter; used for template assignment checks (`templates/services/letter-template.service.ts:70-80`) and for actual variable resolution during `generate()` (`shared/services/variable-resolver.service.ts:87-101,92`, called from `issued-letters/services/issued-letter.service.ts:115-118`). Frontend dropdown filtering (`TemplateForm.tsx:53-58`) only blocks new assignments through the UI. Matters because a legally-issued letter can carry a signatory who has left the org or is no longer authorized to sign.
- **Dead DTO**: `templates/dto/letter-template.dto.ts:89-93` `PreviewLetterTemplateDto` (with `employee_id`) is never used by any controller, service, or frontend call (`templates/controllers/letter-template.controller.ts:57-62` takes no body). Minor but indicates an abandoned "preview with real employee data" feature.
- **`formatDate`'s pattern argument is ignored.** `shared/utils/formatters.ts:7` always renders `DD MMM YYYY` regardless of `LetterSettings.date_format`, which the settings UI/DTO still exposes as editable (`settings/dto/letter-settings.dto.ts:13-16`). Matters because it's a user-facing setting that silently does nothing.
- **Salary-table currency formatting is inconsistent with the flat CTC variable.** `shared/services/pdf-builder.service.ts:75,87` interpolate raw numbers (`` `₹${component.amount}` ``) while the flat `ctc` variable goes through `formatCurrency`'s `toLocaleString('en-IN', ...)` (`shared/utils/formatters.ts:16-21`, used at `variable-resolver.service.ts:55`). Matters because the same letter can show `₹720000` next to a properly comma-formatted CTC.
- **`default_signatory_id` accepted with no existence/status validation.** `settings/services/letter-settings.service.ts:30-45` never checks the id against `SignatoryRepository`, unlike template assignment which does validate (`templates/services/letter-template.service.ts:70-80`). Matters because an admin can silently set a nonexistent or archived signatory as the org default, breaking the fallback path in `variable-resolver.service.ts:95-97`.
- **Unhandled race on first letter generation for a new org.** `shared/services/numbering.service.ts:18-53`: if two `generate()` calls land concurrently before `LetterSettings` exists for the org, `lockByOrganizationId` returns `null` for both, both attempt `createWithManager` against a column with a unique index on `organization_id` (`settings/entities/letter-settings.entity.ts:5`), and the loser throws an unhandled unique-constraint violation instead of a clean retry/`ConflictException`. Matters because it turns a legitimate concurrency case into a 500 rather than a controlled conflict response.

### Technical

- **Effectively zero test coverage.** The only spec file in the module is `shared/seed/starter-letter-content.spec.ts` (out of scope, just built). No tests exist for `SignatoryService`, `LetterTemplateService`, `IssuedLetterService`, `PdfBuilderService`, `VariableResolverService`, `RenderModelBuilderService`, or `NumberingService`. Matters because the module's core legal guarantee — "an issued letter's snapshot must never change even if the template/employee data changes later" — is exactly the kind of invariant that needs regression tests, and currently has none.
- **Fire-and-forget notification failures are silently swallowed.** `.catch(() => {})` at `issued-letters/services/issued-letter.service.ts:176,251,306` with no `Logger.error`/metric on failure. Matters because a notification-queue outage means nobody ever finds out an employee wasn't notified their letter was ready, or that HR wasn't notified of a view/acknowledge.
- **No pagination on any listing endpoint.** `findAll` for categories (`categories/repositories/letter-category.repository.ts:21-42`), signatories (`signatories/repositories/signatory.repository.ts:19-40`), templates (`templates/repositories/letter-template.repository.ts:19-48`), and issued letters (`issued-letters/repositories/issued-letter.repository.ts:41-74`) all return the entire org's result set with no `LIMIT`/cursor. Matters because `issued_letters` grows unbounded over an org's lifetime and this will degrade both query time and response payload size as adoption grows.
- **Minimal structured logging on the highest-risk paths.** Only scattered `Logger.log` calls exist in `categories`/`signatories`/`settings` services; `IssuedLetterService.generate`, `NumberingService.reserveNumber`, and `PdfBuilderService.build` log no timing, failures, or resolved-variable diagnostics beyond throwing exceptions. Matters for debugging production issuance failures, which are otherwise invisible until a user reports them.

## 5. Top 3 Priorities

1. **Close the signatory-archival gap end-to-end** (repository filter in `findOneByOrg`, or a dedicated "resolve for rendering" method that rejects archived signatories at both template-assignment and generation time) — this is the one gap that can put a legally invalid signatory on an actually-issued document, and the fix is small and localized (`signatories/repositories/signatory.repository.ts`, `templates/services/letter-template.service.ts`, `shared/services/variable-resolver.service.ts`).
2. **Either render the signature image on the PDF or stop collecting it** — right now the module does real file-upload/storage work (`signatories/services/signatory.service.ts:28-57`, `DocumentService`) for a field that has zero effect on the actual document, which is wasted effort and a misleading admin experience; also fix the companion "can't replace a signature" gap while touching this code path.
3. **Add tests around `IssuedLetterService.generate`/`NumberingService`/`VariableResolverService`** — this is the module's core, immutability-guaranteeing, money-and-legal-document-adjacent logic, and it currently has no regression protection at all; a bug here silently corrupts or double-numbers real HR letters.
