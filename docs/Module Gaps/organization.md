# Module: Organization

## 1. Purpose & Current Usage

- The module owns two entities and their CRUD surfaces:
  - **`Organization`** (`organization/entities/organization.entity.ts:6-20`) — the tenant record: `name`, `subdomain` (both unique), `enterprise_id` FK, one-to-one `profile`.
  - **`OrganizationProfile`** (`organization/entities/organization-profile.entity.ts:8-72`) — the tenant's business-card data: `name`, `email` (unique), `phone` (unique), `gst_no`/`registration_no` (unique), address fields, `logo`/`industry_type`/`parent` relations.
  - Two services (`OrganizationService`, `OrganizationProfileService`), two repositories, two controllers (`/organizations`, `/organization-profiles`).
- **`OrganizationService.setupCompany()`** (`organization/services/organization.service.ts:62-309`) is the module's centerpiece: a single `queryRunner` transaction that provisions an entire tenant in ~10 sequential steps — enterprise create-or-link (106-115), organization create (118-123), profile create (126-134), plan-app lookup (137-140), platform-role clone + permission clone + role-app clone (143-196), user-role-map creation (205-212), user update (215-217), 14-day trial subscription (219-231), default payroll component (234-247), department clone (250-266), designation clone (269-285), and letter-management starter content seed (288-291, added this session via `seedStarterLetterContent`).
- **Actual callers today** (backend-internal, confirmed by grep — no other service calls `setupCompany` besides the controller):
  - `OrganizationController.setupCompany()` → `organization.controller.ts:29-33`, the only entry point for the whole provisioning flow (`POST /organizations/setup`).
  - `OrganizationService.findOne()` is the module's most reused export — called from `user.service.ts:60,341`, `designation.service.ts:47,84`, `organization-subscription.service.ts:48`, `document.service.ts:181,352`. It's a simple existence-check dependency used across four unrelated modules.
  - `OrganizationProfileService.findByOrganizationId()` is called once, from `letter-management/shared/services/variable-resolver.service.ts:47`, to resolve template placeholder variables (org name/address) at letter-generation time.
- **Dead / unused parts confirmed:**
  - `OrganizationService.debugUser()` (`organization.service.ts:407-432`) and `repairEnterpriseIds()` (`organization.service.ts:434-553`) — neither is wired to any controller route, nor called from any other file in the repo (`grep -rn "repairEnterpriseIds\|debugUser" src/` outside the service file itself returns nothing). `repairEnterpriseIds` alone is 120 lines of one-off data-repair logic (hardcodes the email `b10@admin.com` at line 449) sitting dead in the production service class.
  - `OrganizationResponseDto` (`organization/dto/organization-response.dto.ts`) is defined but never referenced anywhere outside its own file — the controllers return raw entities, not this DTO, so the "prevents accidental exposure of sensitive data" comment in its header is aspirational, not real.
  - Zero test files: `find src/modules/organization -name "*.spec.ts"` returns nothing — no unit tests exist for either service, either repository, or either controller.

## 2. Intended / Ideal Usage

- A new user signs up, gets a `plan_id` but no `organization_id`, then calls `POST /organizations/setup` exactly once with a company name/subdomain/business type. `setupCompany()` should atomically stand up everything that user's org needs on day one — its own enterprise shell, org record, profile, a starter RBAC role set matching their plan's apps, a trial subscription, and starter reference data (payroll component, departments, designations, letter templates) — then log the user in.
- Because this only ever runs once per organization (guarded by the `user.organization_id` check at line 72-74), it's inherently a low-frequency, all-or-nothing operation — which is exactly why it was implemented as one big transaction. The ideal shape keeps the atomicity guarantee but factors the ~5 conceptually distinct provisioning concerns (identity/org, RBAC, billing, HR reference data, letter content) into named, independently testable steps/helpers invoked from one orchestrating method, rather than one 250-line function body.
- Downstream modules should treat `OrganizationService`/`OrganizationProfileService` purely through their public `findOne`/`findByOrganizationId` methods (as `user.service.ts`, `designation.service.ts`, and `variable-resolver.service.ts` already correctly do) — never reach into `Organization`/`OrganizationProfile` entities directly.

## 3. Cross-Module Connections

- **What `setupCompany()` touches (imports + direct entity manipulation), all in one transaction:**
  - `EnterpriseService` (service call, `organization.service.ts:53`) — but also directly imports and `manager.save`s the `Enterprise` entity itself (line 108-114), so even this one dependency is touched two ways.
  - `plan` module — `PlanService.findOne()` (line 81, service call) and direct `PlanApp` entity read (line 137-139, no service).
  - `rbac`/`role` modules — direct entity manipulation only: `Role` (144-168), `ModuleAccess` (171-183), `RoleApp` (186-194), `UserRoleMap` (206-211). No `RoleService`/equivalent is called at all despite `role/services/role.service.ts` existing.
  - `payroll` module — direct `PayrollComponent` entity create (234-247), no service call, despite payroll having its own service layer elsewhere in the codebase.
  - `departments` module — direct `Department` entity clone (250-266), no call to `department.service.ts` despite it existing.
  - `designations` module — direct `Designation` entity clone (269-285), no call to `designation.service.ts` despite it existing (and despite `designation.service.ts` itself depending on `OrganizationService.findOne` — see below, a two-way relationship where the designations module correctly calls into organization's service layer, but organization does not reciprocate).
  - `letter-management` module — the one dependency done "right": calls the exported function `seedStarterLetterContent(manager, ctx)` (`organization.service.ts:45,288-291`) instead of importing `LetterCategory`/`LetterTemplate` entities directly.
  - `auth` module — `AuthService.buildAuthResponse()` (line 298) to mint the login response after setup.
  - `user` module — direct `User` entity update (215-217, 449-537 in `repairEnterpriseIds`).
- **What depends on this module:** `user.service.ts`, `designation.service.ts`, `organization-subscription.service.ts` (plan module), `document.service.ts` all depend on `OrganizationService.findOne()`; `letter-management`'s `variable-resolver.service.ts` depends on `OrganizationProfileService.findByOrganizationId()`. `OrganizationModule` exports both services (`organization.module.ts:42`) for exactly this purpose.
- **Missing/expected connections:** `RoleService`, `DepartmentService`, `DesignationService`, and any payroll-component service are all bypassed inside `setupCompany()` in favor of raw entity access — the inverse of the pattern the module itself expects of its own consumers (`findOne`-style service calls). This is the clearest architectural inconsistency in the module.

## 4. Gaps

### Structural (architecture)
- **`setupCompany()` is a single 247-line method (`organization.service.ts:62-309`) doing ~10 unrelated provisioning steps in one DB transaction with no decomposition.** Confirmed by direct read — steps are only separated by inline `// Step N` comments, not by function boundaries. This makes the method hard to test in isolation (see Technical section) and hard to reason about — a reviewer must hold the entire enterprise/org/RBAC/billing/HR/letters flow in their head at once.
- **Three near-identical copy-paste clone loops with no shared helper.** Role clone (143-196), department clone (250-266), and designation clone (269-285) all follow the same shape — `manager.find(Entity, {where: {is_default/is_system: true}})` → loop → `manager.create` → `manager.save` — but are written three separate times with no extracted `cloneDefaultsForOrg<T>()` utility. Adding a fourth default-cloneable entity type (a real possibility given the pattern's frequency in this codebase) means copy-pasting a fourth block instead of calling a helper.
- **The service reaches past the public API of four sibling modules (rbac/role, departments, designations, payroll) to manipulate their entities directly**, while those same modules' own services (`designation.service.ts:47,84`) correctly call back into `OrganizationService.findOne()`. Only the `letter-management` integration (`seedStarterLetterContent`, line 45/288-291) follows the intended function-boundary pattern. This is a one-directional layering violation: organization's own consumers respect its service boundary, but organization does not respect theirs.
- **`Designation` clone sets both `organization_id` (from `BaseEntity`) and its own separate `org_id` column to the same value** (`organization.service.ts:279-280`, entity has both fields per `designations/entities/designation.entity.ts:12-19`) — two columns tracking the same relationship with no single source of truth, and `setupCompany` has to know to set both or risk an inconsistent designation row (the unique index `IDX_DESIGNATION_ORG_CODE` is keyed on `org_id`, not `organization_id`).

### Coding (bugs, dead code, inconsistent patterns)
- **Profile creation can throw an unhandled NOT NULL violation.** `OrganizationProfile.phone` is `@Column({..., unique: true})` with no `nullable: true` (`organization-profile.entity.ts:16-17`), but `setupCompany` populates it straight from `user.phone` (`organization.service.ts:130`), and `User.phone` is explicitly nullable (`user/entities/user.entity.ts:29-31`). A user who signed up without a phone number will hit a raw Postgres NOT NULL constraint error mid-transaction instead of a clean `BadRequestException` — the whole setup rolls back with a confusing 500.
- **`debugUser()` and `repairEnterpriseIds()` are dead code sitting in the production service** (`organization.service.ts:407-432`, `434-553`) — not routed, not called anywhere, `repairEnterpriseIds` hardcodes a specific user email (`'b10@admin.com'`, line 449) as a debug artifact. This is a one-off migration/debug script that never got removed or moved to a script folder.
- **`OrganizationResponseDto` is unused** (`organization/dto/organization-response.dto.ts`) — both controllers return raw `Organization`/`OrganizationProfile` entities, so there's no actual response-shaping or field-exclusion happening despite the DTO's stated purpose.
- **Role-clone logic only clones the first matching platform default role per app** — `platformRoles.find((r) => r.app_id === appId)` (`organization.service.ts:154`) uses `.find()`, silently dropping any additional default roles that might exist for the same `app_id`, with no warning logged.
- **No uniqueness pre-check before insert for org name/subdomain vs. the profile's own unique columns** — `setupCompany` explicitly checks `findBySubdomain`/`findByName` for `Organization` (85-96) but does the equivalent check nowhere for `OrganizationProfile.email`/`phone`/`gst_no`/`registration_no`, unlike `OrganizationProfileService.create()` which does check email uniqueness (`organization-profile.service.ts:62-69`) for the *manual* profile-creation path. The two profile-creation paths (auto via setup vs. manual via `POST /organization-profiles`) enforce different levels of validation for the same entity.

### Technical (performance, scalability, test coverage, observability)
- **Zero test coverage on the module's most critical path.** `find src/modules/organization -name "*.spec.ts"` returns no files at all — no test exists for `setupCompany()`, its rollback behavior, its ~10 steps, or either repository/controller. This is the single highest-risk gap: a multi-entity, multi-module transactional method with no automated regression protection.
- **A single long-running transaction spans enterprise, org, profile, plan, RBAC (3 sub-loops with nested `manager.find`/`manager.save` calls each), subscription, payroll, departments, designations, and letter-management** (`organization.service.ts:102-296`) — every `manager.save` call inside the loops (168, 182, 193, 211, 265, 284, plus the letter-management seed's own per-category saves) executes sequentially and awaited one at a time rather than batched, and any single failure anywhere in this chain (e.g., the phone NOT NULL bug above, or a future letter-management seed error) rolls back the entire org creation, including entities from completely unrelated domains (payroll, RBAC) that had already succeeded logically.
- **No structured/observability hooks around the individual steps** — the method logs only a start message (line 83) and a success/failure message (294-296, 300-304); there is no per-step timing or step-name context in the error log, so a failure at, say, step 8 (departments) is reported with the same generic message as a failure at step 2 (org creation), making production triage slower than necessary for a 10-step transaction.

## 5. Top 3 Priorities

1. **Add unit/integration test coverage for `setupCompany()` before making any further changes to it.** It is the module's most complex, most consequential, and currently the only completely untested piece of logic in the codebase's tenant-provisioning path — every session that adds a new step (like this session's letter-management seed) compounds risk with zero safety net.
2. **Decompose `setupCompany()` into named per-concern steps and extract the repeated clone-loop pattern into a shared helper.** The method already reads as ~10 discrete steps by comment; turning those comments into actual method/function boundaries (and collapsing the role/department/designation clone loops into one generic helper) would make the transaction's scope legible and testable without changing its atomicity.
3. **Fix the `OrganizationProfile.phone` NOT NULL vs. nullable `User.phone` mismatch**, either by making `phone` nullable on the profile entity or by validating/defaulting it before the transaction — this is a live, reachable bug (any phone-less signup hits it), not a hypothetical one, and it currently surfaces as an opaque 500 instead of a clean validation error.
