# Module: Plan

## 1. Purpose & Current Usage

`brello_server/src/modules/plan/` defines the tier/entitlement model that the rest of the app reads from to decide what an organization is allowed to see and do.

- **Entities**
  - `entities/plan.entity.ts` — `Plan`: name, `price` (monthly per-employee), `price_per_employee_annual`, `annual_discount_percent`, `discount`, `tier_rank` (0=Free/1=Standard/2=Premium, used to detect upgrade vs downgrade), `billing_cycle_default`, `feature[]`. Inherits `enterprise_id`/`organization_id`/`status` from `BaseEntity` but only `enterprise_id` is actually used (plans can be enterprise-scoped or platform-global); `organization_id` on a `Plan` is dead weight from the shared base class.
  - `entities/plan-app.entity.ts` — `PlanApp`: `(plan_id, app_id, is_active)`, unique per pair. Which top-level apps a plan bundles.
  - `entities/organization-subscription.entity.ts` — `OrganizationSubscription`: `organization_id`, `plan_id`, `sub_status` (Trial/Active/Grace/Expired/Cancelled), `billing_cycle`, `is_trial`, `start_date`/`end_date`/`next_renewal_date`, plus deferred-change fields `pending_plan_id`/`pending_billing_cycle`, and `grace_period_ends_at`.
  - `entities/plan-module.entity.ts` / `plan-module-action.entity.ts` — `(plan_id, module_id[, action_id], enabled)`. These are the actual entitlement tables: which conceptual modules, and which actions within them, a plan permits.
- **Services**: `PlanService`, `OrganizationSubscriptionService`, `PlanModuleService`, `PlanModuleActionService` — all four are close to pure CRUD wrappers around their repositories. Notably, `OrganizationSubscriptionService.update()` (`services/organization-subscription.service.ts:82-97`) only ever mutates `end_date` and `sub_status` — it has no code path that changes `plan_id`. There is no upgrade/downgrade method anywhere in this module.
- **Controllers**: `PlanController` (`/plans`, `GET` is `@Public()`), `OrganizationSubscriptionController` (`/organization-subscriptions`), `PlanModuleController` (`/plan-modules`), `PlanModuleActionController` (`/plan-module-actions`). All four are annotated only with `@UseGuards(JwtAuthGuard)` at the controller level — none narrow further to platform-admin-only.
- **Who actually calls into it today**:
  - `PermissionResolverService.resolve()` (`src/modules/rbac/services/permission-resolver.service.ts:86-112, 181-279`) is the real consumer: on every permission resolution it loads the org's active `plan_id`, then intersects role-based `ModuleAccess` grants against `PlanModule`/`PlanModuleAction` rows for that plan. This is where plan restrictions are actually enforced, and it re-queries the DB per call (no caching), so it will reflect whatever `plan_id` currently sits on the subscription row.
  - `ModuleAccessService.assignByCode()` (`src/modules/app-module/services/module-access.service.ts:120-170`) writes back into `PlanModule`/`PlanModuleAction` to auto-provision entitlement rows when a role is granted access to something the current plan doesn't yet list — a one-way "widen" operation, never a "narrow."
  - `SubscriptionBillingService.changePlan()` (`src/modules/billing/services/subscription-billing.service.ts:43-115`) — the only real "upgrade/downgrade" business logic in the codebase — lives entirely in the **billing** module, not here, and mutates `OrganizationSubscription` via a raw TypeORM repository injected directly into billing (`subOrm`), completely bypassing `OrganizationSubscriptionService`/`OrganizationSubscriptionRepository`.
  - `brello_webapp/src/features/platform/planPermissions/*` and `features/platform/organizations/*` are the only webapp features hitting `/plan-modules`, `/plan-module-actions`, `/organization-subscriptions` — i.e. in practice only the platform-admin UI drives these routes today, even though the backend doesn't require that.
- **Dead/unused**: `OrganizationSubscriptionController`'s generic `PATCH /:id` and the module's own `create()` are effectively bypassed for real subscription lifecycle — billing creates/mutates subscriptions directly. No `*.spec.ts` exists anywhere under `modules/plan/` (confirmed by search) — zero automated test coverage for entitlement logic.

## 2. Intended / Ideal Usage

`docs/prd/plan.md` is not actually about this module — it's an unrelated Reimbursement-module implementation plan that happens to share the filename. The relevant docs are `docs/prd/platform_admin/platform-plans.md` (shipped PRD for the Platform Admin "Plans" page) and `docs/api-contracts/12-plan.md` (API contract).

- The API contract (`docs/api-contracts/12-plan.md:215-219`) documents `PATCH /organization-subscriptions/:id` as accepting a "partial update of creation payload" (implying `plan_id` is patchable), but `UpdateOrganizationSubscriptionDto` (`dto/organization-subscription.dto.ts:32-40`) only exposes `end_date` and `sub_status` — the doc overstates what the endpoint can actually do; real plan changes require the separate billing `ChangePlanDto` flow instead.
- Both docs describe plan CRUD as a "Platform Admin" operation (`docs/prd/platform_admin/platform-plans.md:14-22`, `docs/api-contracts/12-plan.md:15-21`: "Who can call: Platform admin"), but nothing in the actual controllers enforces that — see Gaps.
- No PRD anywhere defines seat/employee-count limits per plan; the `Plan` entity has no `max_employees`/`seat_limit` field and no code in the repo references one (confirmed by search) — "seat limits" as named in the task brief do not exist as a concept in this codebase at all, only module/action entitlement does.

## 3. Cross-Module Connections

**Depends on** (imports/injects):
- `modules/organization/services/organization.service.ts` — to resolve `enterprise_id` when creating a subscription.
- `modules/enterprise/services/enterprise.service.ts` — `assignAppsToEnterprise()` called from `OrganizationSubscriptionService.create()` to push the new plan's bundled apps onto the enterprise.
- `modules/audit/services/audit-context.service.ts` — used by `PlanService` (pre-value capture for `@AuditLog` diffing) but **not** by `OrganizationSubscriptionService`, `PlanModuleService`, or `PlanModuleActionService`.

**Depended on by**:
- `modules/rbac` — `PermissionResolverService` injects `OrganizationSubscription`, `PlanModule`, `PlanModuleAction` repositories directly (`rbac.module.ts:8-9,46-47`) to enforce entitlement — this is the real integration point for "does the plan actually restrict access."
- `modules/app-module` — `ModuleAccessService` reads/writes `PlanModule`/`PlanModuleAction` to keep entitlement in sync when granting role access.
- `modules/billing` — `forwardRef(() => PlanModule)` in `billing.module.ts:64`; owns all real subscription lifecycle logic (`SubscriptionBillingService.changePlan/cancelPendingChange/cancelSubscription`, `RenewalInvoiceCron`, `PaymentService.renewSubscriptionForInvoice`) by injecting `OrganizationSubscriptionRepository`/`PlanRepository` from this module plus a raw `Repository<OrganizationSubscription>`.
- `modules/organization` — `forwardRef(() => PlanModule)`.

**Missing/expected connections**:
- **Notifications**: confirmed zero `NotificationService` references in `modules/plan/services/organization-subscription.service.ts` and, more importantly, zero in `modules/billing/services/subscription-billing.service.ts` — the file that actually executes upgrades/downgrades. The only billing file that imports `NotificationService` is `crons/trial-reminder.cron.ts`, which is unrelated to plan-change events. No org admin is ever notified when a plan upgrade, downgrade, or cancellation happens.
- **Downgrade enforcement is real but delayed and payment-gated, not "revoked" in any explicit sense**: `SubscriptionBillingService.changePlan()` (`subscription-billing.service.ts:74-80`) never touches `plan_id` for a downgrade — it only sets `pending_plan_id`/`pending_billing_cycle` and returns `applied: 'next_cycle'`. The swap to the new (lower) `plan_id` happens only inside `PaymentService.renewSubscriptionForInvoice()` (`billing/services/payment.service.ts:328-353`), which itself only runs when a Razorpay webhook reports the renewal invoice as paid (`payment.service.ts:283-295`). Since `PermissionResolverService` re-reads `plan_id` fresh on every request, access genuinely narrows the moment `plan_id` flips — but if that renewal invoice is never paid (declined, ignored, webhook lost), `plan_id` never flips and the org silently keeps full access to the higher (canceled) tier indefinitely, with nothing in this module or billing detecting or alerting on that stuck state.

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **All real plan-change business logic lives outside the Plan module.** `SubscriptionBillingService.changePlan()` (`src/modules/billing/services/subscription-billing.service.ts:43-115`) is the only place upgrade/downgrade/immediate-vs-deferred logic exists, yet it's in `billing`, injects a raw `Repository<OrganizationSubscription>` (`subOrm`, line 32) alongside the Plan module's own repository, and never calls `OrganizationSubscriptionService`. The Plan module's own service for the same entity (`OrganizationSubscriptionService`) is reduced to CRUD that no real flow uses for plan changes. This matters because there are now two independent write paths into the same table with no shared invariant-checking layer.
- **`OrganizationSubscriptionController`'s `PATCH /:id` cannot actually change plans** (`controllers/organization-subscription.controller.ts:54-62` → `dto/organization-subscription.dto.ts:32-40`, no `plan_id` field) despite the API contract documenting it as a general partial-update endpoint (`docs/api-contracts/12-plan.md:215-219`) — the contract and the code have drifted apart.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

- **No authorization guard beyond a generic JWT check on any mutating Plan endpoint.** `PlanController`, `OrganizationSubscriptionController`, `PlanModuleController`, `PlanModuleActionController` are all decorated with only `@UseGuards(JwtAuthGuard)` (e.g. `controllers/plan.controller.ts:26`, `controllers/plan-module.controller.ts:24`, `controllers/plan-module-action.controller.ts:24`, `controllers/organization-subscription.controller.ts:24`) — no `PlatformAdminGuard` or `RequirePermission` check, unlike comparable platform-only controllers elsewhere (`modules/enterprise/controllers/enterprise.controller.ts`, `modules/platform/controllers/platform-department.controller.ts`, `modules/role/controllers/platform-role.controller.ts` all use `PlatformAdminGuard`). Both PRDs describe these as platform-admin-only operations (`docs/prd/platform_admin/platform-plans.md:14`, `docs/api-contracts/12-plan.md:15-21`), but the code lets any authenticated user of any organization hit `POST /plan-modules`, `POST /plan-module-actions`, or `POST /organization-subscriptions` for any plan/org — these tables are not org-scoped, so this is not merely "wrong org," it is global entitlement data any logged-in user can write.
- **Inconsistent audit coverage inside one module.** `PlanController` has `@AuditLog` on create/update/delete/assignApps (`controllers/plan.controller.ts:30,60,71,90`), but `OrganizationSubscriptionController`, `PlanModuleController`, and `PlanModuleActionController` have none at all, and their services never call `auditContext.setPreValue()` either — entitlement changes (which modules/actions a plan grants) and subscription CRUD leave no audit trail while plan price/name changes do.
- **Redundant entity declaration.** `Plan.description` is re-declared with `declare description: string` (`entities/plan.entity.ts:35-36`) with an identical column definition to the one it already inherits from `BaseEntity` — harmless but confusing; a future edit to one and not the other would silently diverge.

### Technical (performance, security, scalability, test coverage, observability/logging)

- **Downgrade "enforcement" is contingent on payment succeeding, with no fallback.** As detailed in §3, the pending-plan swap only happens inside a payment-webhook success handler (`billing/services/payment.service.ts:342-345`). If payment never completes, the subscription keeps its old (higher) `plan_id` and `sub_status` may still read `Active` until some other mechanism (not found in this module) flips it — there is no cron or check anywhere in `modules/plan` or `modules/billing/crons` that force-applies an overdue pending downgrade after a grace period. This is a real entitlement-integrity gap: the org can end up paying for (or not paying for) a lower tier while technically still holding higher-tier access indefinitely.
- **No seat/employee-count enforcement exists at all.** Neither `Plan` nor any related entity has a seat/employee-limit column, and no service anywhere checks employee headcount against a plan limit — so the "does downgrade enforce seat limits" question has no code to answer it with; this dimension of entitlement simply isn't modeled.
- **Zero automated test coverage.** No `*.spec.ts` file exists anywhere under `modules/plan/` — the intersection logic in `PermissionResolverService.applyPlanRestrictions()` and the immediate/deferred branching in `SubscriptionBillingService.changePlan()` (both security/billing-sensitive) are unverified by tests.
- **No notification on plan change**, confirmed absent in both `modules/plan` and `modules/billing` (see §3) — org admins get no email/in-app signal for upgrade, downgrade, or cancellation, and end users get no warning before losing module access at the next renewal.

## 5. Top 3 Priorities

1. **Add an authorization guard (`PlatformAdminGuard` or equivalent) to `PlanController`, `OrganizationSubscriptionController`, `PlanModuleController`, and `PlanModuleActionController`.** Right now any authenticated user, regardless of organization, can create/modify/delete global plan pricing and entitlement mappings (`plan_module`, `plan_module_action`) that are not org-scoped — this is a live privilege-escalation surface, not just a missing "nice to have," and the fix is small relative to the risk.
2. **Make the pending-downgrade swap resilient to non-payment.** Today `pending_plan_id` only ever becomes `plan_id` inside a successful Razorpay webhook handler; add an explicit grace-period cron (or extend `RenewalInvoiceCron`) that force-applies (or expires) a pending downgrade when payment doesn't arrive, so entitlement can't silently stay stuck on a higher tier forever.
3. **Wire `NotificationService` into `SubscriptionBillingService.changePlan()` / `cancelSubscription()`.** Org admins currently receive no signal at all when their plan changes or is scheduled to change — this is a straightforward, well-scoped addition (the injection point and business logic already exist in one file) that closes the notification gap identified for this module.
