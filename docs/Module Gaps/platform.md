# Module: Platform

## 1. Purpose & Current Usage

- `brello_server/src/modules/platform/` is a thin, narrowly-scoped module: it defines **no entities of its own**. `platform.module.ts:3-11` imports `Department` (from `../departments/entities/department.entity.ts`) and `Designation` (from `../designations/entities/designation.entity.ts`) via `TypeOrmModule.forFeature`, and wires two controller/service pairs on top of those shared entities:
  - `PlatformDepartmentController` (`controllers/platform-department.controller.ts:23-57`) тЖТ `/platform-admin/departments`, CRUD via `PlatformDepartmentService` (`services/platform-department.service.ts`).
  - `PlatformDesignationController` (`controllers/platform-designation.controller.ts:23-57`) тЖТ `/platform-admin/designations`, CRUD via `PlatformDesignationService` (`services/platform-designation.service.ts`).
- Both controllers are correctly gated with `@UseGuards(JwtAuthGuard, PlatformAdminGuard)` (`platform-department.controller.ts:24`, `platform-designation.controller.ts:24`), and every mutating endpoint carries an `@AuditLog(AuditLogModule.PLATFORM_SETUP, ...)` decorator (`platform-department.controller.ts:34,41,51`; `platform-designation.controller.ts:34,41,51`), so audit trail integration for this slice is solid.
- The services operate purely on the `is_default = true` slice of the shared `Department`/`Designation` tables: `findAll()`/`findOne()` filter `is_default: true, is_deleted: false` (`platform-department.service.ts:18-33`; `platform-designation.service.ts:18-33`, the latter also filters `org_id: IsNull()`). `create()` upper-cases the code and force-sets `is_default: true` (`platform-department.service.ts:43-51`; `platform-designation.service.ts:48-56`). `remove()` is a soft delete (`is_deleted = true`, `status = DELETED`) тАФ no hard delete anywhere.
- **Who calls into it:** Only the module's own controllers call the services (`grep` for `PlatformDepartmentService`/`PlatformDesignationService` outside `src/modules/platform/` returns nothing but the `app.module.ts:35,90` registration). Registered in `app.module.ts:35,90`. Frontend calls land on `pages/platform/PlatformDepartmentPage.tsx` and `PlatformDesignationPage.tsx`, routed via `brello_webapp/src/routes/platformAdminRoutes.tsx:34-35` (`setup/departments`, `setup/designations`).
- **Dead/unused parts:** none inside the module itself тАФ everything defined is reachable from a controller. However, the module is much smaller than what the frontend groups under "Platform Admin" (see ┬з3) тАФ most of the platform-admin surface (apps, modules, actions, industry types, plans, roles, leads, enterprises, organizations) lives in **other** top-level modules, not here.

## 2. Intended / Ideal Usage

- `docs/prd/platform_admin/` contains 3 PRDs: `platform-apps-modules.md`, `platform-plans.md`, `unified-login-flow.md`. None of them document Departments/Designations/Industry-Types specifically (that PRD apparently predates or was never written), but they do establish the pattern this module should follow:
  - `platform-apps-modules.md:65` states the `apps` controller is "guarded by `JwtAuthGuard` + `PlatformAdminGuard`" тАФ matching what `platform/` actually does.
  - `platform-plans.md:124` documents `POST /plans` as `JwtAuthGuard, isPlatformAdmin: true in JWT`.
  - The same PRD (`platform-apps-modules.md:98`) documents the sibling `app-modules` controller as guarded by **`JwtAuthGuard` only** тАФ i.e., even the PRD's own account of the platform-admin surface is inconsistent about whether `PlatformAdminGuard` is required, and the real code follows that inconsistency (see ┬з4 Technical).
- Within its own scope, `platform/` fully matches the "ideal": guard stack, audit logging, soft-delete, uppercase code normalization are all present and consistent between the department and designation implementations. No PRD gaps found for the part that *is* documented.

## 3. Cross-Module Connections

- **Depends on:** `departments` module (`Department` entity), `designations` module (`Designation` entity), `audit` module (`AuditContextService`, `AuditLog` decorator, `AuditLogModule`/`AuditAction` enums), `core/guards` (`JwtAuthGuard`, `PlatformAdminGuard`).
- **Depends on this module:** `organization.service.ts` clones the `is_default` rows this module manages into every newly-created org, but **does so by querying the `Department`/`Designation` repositories directly through the transaction manager**, not by calling `PlatformDepartmentService`/`PlatformDesignationService`:
  - Departments: `organization/services/organization.service.ts:250-266` тАФ `manager.find(Department, { where: { is_default: true, is_deleted: false } })`, duplicating the exact filter already encapsulated in `platform-department.service.ts:19-22`.
  - Designations: `organization.service.ts:269-285` тАФ `manager.find(Designation, { where: { is_default: true, is_deleted: false } })`. Note this omits the `org_id: IsNull()` filter that `platform-designation.service.ts:20` applies, so the two "same" queries are not actually identical тАФ a latent inconsistency if a non-platform designation ever gets `is_default: true` set incorrectly.
  - This means `platform/`'s services are not the source of truth in practice тАФ the entities are, and two different code paths read/interpret the `is_default` flag with slightly different filters.
- **Boundary vs `enterprise-admin`:** clean split, no real overlap. `src/modules/enterprise-admin/` is **empty scaffolding** (folders `billing/`, `dashboard/`, `guards/`, `organization/`, `rbac/{controllers,services}` with zero files, no `.module.ts`, not imported in `app.module.ts`) тАФ confirmed dead per the existing audit at `docs/Module Gaps/enterprise-admin.md`. It poses no confusion risk in code, only in navigation (a developer browsing `src/modules/` sees 5 empty folders next to the working `platform` module).
- **Boundary vs `organization`:** clean, conceptually correct split тАФ `platform` owns org-agnostic **defaults** (`is_default: true` rows, no `organization_id`), `organization` owns the **per-tenant** copies created at signup (see clone logic above). The two modules never share a service layer, only the same two entity classes.
- **Missing/expected connections тАФ platform-admin alert gap (per prior audit context):** confirmed still true. `grep` for `NotificationService`/`Notification` inside `src/modules/platform` returns nothing, and `organization/services/organization.service.ts` (the file that handles org creation, `create-company` flow) has **zero** notification calls anywhere in it. The only `NotificationService` usage found in the lead-capture path is `lead/services/lead.service.ts:179-184`, which sends an OTP-verification email **to the lead**, not an alert to platform admins тАФ so "new lead captured" is not wired to notify platform admins either. No cron or hook was found firing on subscription-expiry-without-renewal (`billing/crons/trial-reminder.cron.ts` exists but is a trial reminder to the org, not a platform-admin alert). This corroborates the known gap: platform-admin alerts for new-org-registered, new-lead-captured, and subscription-expired-with-no-renewal have no notification wiring anywhere in the backend.

## 4. Gaps

### Structural
- **The "platform admin" conceptual surface is scattered across ~8 top-level modules with inconsistent RBAC gating, and `src/modules/platform/` covers only 2 of them.** `platformAdminRoutes.tsx:24-40` (frontend) routes to industry-types, departments, designations, apps, modules, actions, plans, roles, leads, enterprises, organizations, audit-logs тАФ but only departments/designations live under `src/modules/platform/`. This matters because "audit the platform module" cannot be done in isolation from `industry-type`, `app-module`, `app`, `lead`, `plan` тАФ the actual admin-guard risk (┬з4 Technical) lives in those sibling modules, not in `platform/` itself, and nothing in the codebase documents this split.
- **`organization.service.ts` bypasses `PlatformDepartmentService`/`PlatformDesignationService` and re-implements their query filters inline** (`organization.service.ts:250-251, 269-271` vs `platform-department.service.ts:19-22`, `platform-designation.service.ts:19-22`), and the two implementations of "find default designations" disagree on whether to filter `org_id: IsNull()`. This is a DRY/layering violation тАФ if the definition of "what counts as a platform default" ever changes, it now needs updating in two places, and one of them already silently diverged.

### Coding
- **Default department/designation code uniqueness is enforced only at the application layer with a TOCTOU race, and the DB index doesn't actually back it up for default rows.** `platform-department.service.ts:36-41` does a `findOne` then `save` with no unique constraint or transaction wrapping it; the DB-level `@Index(['organization_id', 'code'], { unique: true })` (`departments/entities/department.entity.ts:5`) does not protect against duplicate codes among default rows because `organization_id` is `NULL` for all of them and Postgres treats `NULL` values as distinct in a unique index. Same pattern in `designations/entities/designation.entity.ts:12` (`IDX_DESIGNATION_ORG_CODE` on `org_id, code`, also NULL for defaults). Two concurrent `POST /platform-admin/departments` (or designations) requests with the same code can both pass the `findOne` check and both insert, matters because it silently produces duplicate "default" rows that then get cloned twice into every new org (`organization.service.ts:253-266`).
- **`update()` on both services does an unchecked `Object.assign(entity, updateDto)`** (`platform-department.service.ts:56`, `platform-designation.service.ts:62`) with no re-check of the code-uniqueness constraint тАФ moot today only because the update DTOs don't expose `code` as editable (`update-platform-department.dto.ts`, `update-platform-designation.dto.ts` both omit `code`), but there's no comment/test documenting that this is intentional, so it's one accidental DTO field away from silently breaking uniqueness.

### Technical
- **Within `platform/` itself, RBAC gating is correct on every endpoint** тАФ both controllers apply `JwtAuthGuard` + `PlatformAdminGuard` (`platform-department.controller.ts:24`, `platform-designation.controller.ts:24`). However, the sibling parts of the same conceptual "platform admin" surface are inconsistently gated, which is the real security-relevant finding for this audit:
  - `src/modules/industry-type/controllers/industry-type.controller.ts:20` тАФ `@Controller('industry-types')` has **no `@UseGuards` at all** (not even `JwtAuthGuard`, despite it being imported at line 8 and unused). Every CRUD endpoint (`create`, `findAll`, `findOne`, `update`, `remove`) is reachable with **no authentication**, even though the frontend treats it as a platform-admin-only screen (`pages/platform/PlatformIndustryTypePage.tsx`, routed at `platformAdminRoutes.tsx:33`, calling `industry-types` directly via `features/platform/industryTypes/api.ts:10`).
  - `src/modules/app-module/controllers/app-module.controller.ts:22` and `action.controller.ts:21` both apply `@UseGuards(JwtAuthGuard)` only тАФ no `PlatformAdminGuard` тАФ even though `/app-modules` and `/actions` are platform-admin-only screens per `platformAdminRoutes.tsx:32,36` (`setup/actions`, `modules`). Contrast with the sibling `app.controller.ts:23`, which correctly applies `@UseGuards(JwtAuthGuard, PlatformAdminGuard)`. This means **any authenticated user of any organization** can currently create/update/delete app modules and actions that drive sidebar navigation and RBAC permission trees platform-wide.
  - This matters because the blast radius is platform-wide (these are org-agnostic, shared-tenant tables), and the inconsistency is easy to miss precisely because the two correctly-guarded controllers (`platform/*`, `app.controller.ts`) sit right next to the incorrectly-guarded ones and look identical in shape.
- **Zero test coverage.** No spec/test file matching `*platform*` exists anywhere in the repo (`find . -iname "*platform*spec*" -o -iname "*platform*test*"` returns nothing) тАФ no unit tests for `PlatformDepartmentService`/`PlatformDesignationService`, no e2e test asserting the `PlatformAdminGuard` actually rejects non-admins on these two controllers.
- **No pagination on `findAll()`** in either service (`platform-department.service.ts:18-23`, `platform-designation.service.ts:18-23`) тАФ low severity today since default rows are admin-curated and small in number, but there's no guard rail if that assumption changes.
- **Logging/observability:** beyond the generic `@AuditLog` decorator (which records CRUD to the audit trail), there is no application-level logging (no `Logger` instance) in either service тАФ failures (e.g., the race condition above) leave no trace besides the audit log's post-hoc diff.

## 5. Top 3 Priorities

1. **Add `PlatformAdminGuard` to `app-module.controller.ts` and `action.controller.ts`** (currently `JwtAuthGuard`-only) тАФ this is the highest-impact fix because it's a live authorization hole letting any authenticated user of any org mutate platform-wide navigation/RBAC-module definitions, not a hypothetical.
2. **Add `JwtAuthGuard` + `PlatformAdminGuard` to `industry-type.controller.ts`** тАФ currently has zero guards, so industry-type CRUD is fully unauthenticated despite being surfaced only in the platform-admin UI; trivial fix, high severity.
3. **Fix the default-code uniqueness race**: back it with a real partial unique index (e.g., a Postgres partial unique index on `code` `WHERE is_default = true`) or a `SERIALIZABLE`/`SELECT ... FOR UPDATE` guard in `create()`, and reconcile `organization.service.ts`'s inline `Department`/`Designation` queries with `PlatformDepartmentService`/`PlatformDesignationService` (or extract a shared query helper) so the "what is a platform default" definition can't silently diverge again.
