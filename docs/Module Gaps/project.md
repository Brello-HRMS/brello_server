# Module: Project

## 1. Purpose & Current Usage
- Implements client-scoped project CRUD, team assignment, and contract attachment. `Project` (`entities/project.entity.ts:20-65`) belongs to a `Client` (`client_id`, unique with `name`), has `project_type`/`project_status`/`priority` enums (`enums/project-enums.ts`), and two child tables: `ProjectTeam` (`entities/project-team.entity.ts`, table `project_team_mappings`) and `ProjectContract` (`entities/project-contract.entity.ts`, table `project_contracts`).
- `ProjectService` (`services/project.service.ts`) implements: `create`/`findAllByClient` (org+client scoped, `:38-111`), `findAll`/`findOne` (org-wide list/detail, `:113-148`), `update`/`remove` (`:150-205`), `assignTeam`/`getTeam`/`removeTeamMember` (`:207-223`, `:248-252`, `:260-264`), and `uploadContract`/`getContracts`/`removeContract` (`:225-246`, `:254-258`, `:266-299`), which delegates to `DocumentService` for file linkage/cleanup.
- Two controllers expose this: `ClientProjectController` (`controllers/client-project.controller.ts`) mounts `POST/GET /clients/:clientId/projects` (create + client-scoped list); `ProjectController` (`controllers/project.controller.ts`) mounts the rest under `/projects/:id` (get/update/delete, `/team`, `/contract(s)`). All endpoints use `@AuditLog(...)` decorators for create/update/delete/assign/unassign (`project.controller.ts:50,61,71,110`, `client-project.controller.ts:28`).
- Real callers confirmed on the frontend: `brello_webapp/src/features/project/api/projectApi.ts`, `useAddProjectTeam.ts`, `useProjectTeam.ts`, `TeamTab.tsx`, and `pages/project/ProjectPage.tsx` / `pages/client/ClientDetailPage.tsx` — this is an actively used module, not dead code.
- Dead/unused: none of the module's own code paths are unreachable, but `created_by` is set at creation time (`project.service.ts:79`) even though neither `Project` nor `BaseEntity` declares a `created_by` column (`entities/project.entity.ts`, `common/entities/base.entity.ts:9-45`) — the value is silently dropped on save and never persisted or queryable.

## 2. Intended / Ideal Usage
- A project is created under a client, staffed via `assignTeam` (single source of truth for "who is on this project"), and that team roster should drive downstream authorization (e.g., timesheet entry) and should notify assigned/removed members. Status/priority/date fields should support health tracking (e.g., approaching-deadline alerts) and reporting. Contract documents should be attached/detached via `DocumentModule` with no orphaned files. All of this should stay within organization boundaries and be consistent between the client-scoped and top-level project APIs (same filters, same status field).

## 3. Cross-Module Connections
- Depends on: `ClientModule` (`project.module.ts:10,18`, validates client exists/org match, `project.service.ts:46-51`), `DocumentModule` (`project.module.ts:13`, contract file linkage, `project.service.ts:33,235,292`), `GlobalSearchModule` (`project.module.ts:11`, `SearchIndexingService.indexProject`/`removeProject`, `project.service.ts:83,193,204`), `UserModule` (`project.module.ts:12`, for `ProjectTeam.user` relation), `AuditContextService` from the `audit` module (`project.service.ts:24,158,201`).
- Depends on it: `TimesheetModule` reaches directly into this module's persistence layer rather than going through `ProjectService`/`ProjectRepository` — `timesheet.module.ts:7-8,12` imports the `Project` and `ProjectTeam` entities and registers its own `TypeOrmModule.forFeature([...ProjectTeam, Project])`, and `timesheet/repositories/timesheet.repository.ts:105-119` runs its own `createQueryBuilder('ptm')` against `project_team_mappings` to check `isUserAssignedToProject`, and again at `:186-220` to list a user's assigned projects for the timesheet dropdown/dashboard. This *is* the timesheet integration — it works, but it's a layering violation: two modules' repositories both own direct read/write access to `project_team_mappings`.
- Missing/expected connections:
  - **Notifications**: `assignTeam` (`project.service.ts:207-223`) and `removeTeamMember` (`:260-264`) never call any notification service — confirmed via repo-wide grep, no `NotificationService`/`notificationService` reference anywhere under `src/modules/project/`. Compare to `lead`, `feedback`, `user/employee`, and `letter-management` services, which all inject `NotificationService` for equivalent "something changed for a user" events. A team member added/removed from a project gets zero email/in-app/push signal.
  - **Deadline reminders**: no cron exists anywhere for approaching `project.end_date`. Confirmed by listing every `@Cron`-bearing file in the repo (`attendance-cron.service.ts`, `auto-checkout.service.ts`, `search-cleanup.service.ts`, `otp-cleanup.task.ts`, `offboarding-cron.service.ts`, `org-setup.cron.ts`, `payroll-reminder.cron.ts`, `payroll-auto-run.cron.ts`, `subscription-expiry.cron.ts`, `trial-reminder.cron.ts`, `renewal-invoice.cron.ts`) — none touch `project`. `billing` and `payroll` both have this exact pattern (expiry/reminder crons) that `project` lacks entirely.

## 4. Gaps

### Structural
- Timesheet bypasses the project module's service/repository layer entirely and owns direct TypeORM access to `Project`/`ProjectTeam` (`timesheet.module.ts:7-8,12`, `timesheet.repository.ts:105-119,186-220`) instead of calling an exported method on `ProjectService`/`ProjectRepository` (only `ProjectService` is exported from `project.module.ts:25`, not the repository's team-query capability in a reusable form). This matters because any future change to team-assignment semantics (e.g., soft-deleted members, role-based eligibility) has to be replicated in two modules' query logic instead of one shared source of truth.
- No RBAC/permission guard beyond `JwtAuthGuard` on any project endpoint (`project.controller.ts:28`, `client-project.controller.ts:24`) — any authenticated user in the org can assign/remove team members or delete a project. This is consistent with `client`/`user` controllers elsewhere in the codebase (not project-specific), so it is noted but not counted as a top priority.

### Coding
- `created_by: user.userId` is set on project creation (`project.service.ts:79`) but there is no `created_by` column on `Project` or `BaseEntity` (`entities/project.entity.ts:20-65`, `common/entities/base.entity.ts:9-45` only has `modified_by`/`modified_at`/`deleted_by`/`deleted_at`) — the field is silently discarded by TypeORM's `create()`/`save()`, so "who created this project" can never be answered from the data.
- `remove()` is named/typed as a soft delete but performs a hard delete: `projectRepository.softDelete(id)` (`project.service.ts:203`) calls `this.repository.delete(id)` (`repositories/project.repository.ts:53-55`), not TypeORM's `softRemove`/`softDelete` — even though `BaseEntity` carries unused `deleted_at`/`deleted_by` columns (`base.entity.ts:41-44`) that suggest soft delete was the intended design. Deleting a project this way also has no cleanup of its `project_team_mappings`/`project_contracts` rows or of `DocumentService`-linked contract files (contrast with `removeContract`, which does explicitly clean up its linked document at `project.service.ts:290-297`) — orphaned rows/files are the likely result unless a DB-level `ON DELETE CASCADE` exists outside this codebase (no migration files were found to confirm either way).
- Client-scoped listing filters on the wrong column name: `findAllByClient` passes `filterFields: ['status', 'priority']` (`project.service.ts:108`), but the entity's status column is `project_status`, not `status` (`entities/project.entity.ts:44-45`); `ListingHelper.apply` builds the filter as `${alias}.${field} = :value` (`common/utils/listing.helper.ts:60-67`), so passing `?status=ACTIVE` to `GET /clients/:clientId/projects` generates `project.status = :status` against a non-existent column and will throw a SQL error at runtime. The top-level `findAll` (`:133`) correctly uses `project_status`, showing this is an inconsistency/copy-paste bug rather than an intentional difference.
- `CreateProjectDto.start_date`/`end_date` are validated with `@IsDate()` (`dto/create-project.dto.ts:32-38`) with no `@Type(() => Date)` transform; incoming JSON bodies carry these as strings, and correctness depends entirely on the global `ValidationPipe`'s `enableImplicitConversion` (`main.ts:33-35`) doing the string→Date coercion before validation — there is no explicit, self-documenting transform on the DTO itself, unlike the common pattern of `@IsDateString()` used elsewhere for date-only fields (e.g. `list-projects.dto.ts` filters don't have this issue, but this is fragile if `enableImplicitConversion` is ever turned off globally).

### Technical
- Zero automated test coverage: no `.spec.ts` file exists anywhere under `src/modules/project/` (confirmed via `find ... -iname "*.spec.ts"`), so none of the uniqueness checks, date validation, team-lead-count validation (`project.service.ts:216-219`), or contract cleanup logic has regression protection.
- No structured metrics/observability beyond `Logger.log` calls at the start of each service method (`project.service.ts:43,92,117,139,155,198,212,230,249,255,261,271`) — there's no signal for how often team reassignment happens, contract upload failures, or project counts by status, which matters once deadline reminders or notification volume become a concern.

## 5. Top 3 Priorities
1. Add notification calls to `assignTeam`/`removeTeamMember` (`project.service.ts:207-223`, `260-264`) — this is the most user-visible gap: team members are added/removed from projects with no email/in-app/push signal, unlike every comparable "something changed for a user" flow elsewhere in the codebase (lead, feedback, employee, letters).
2. Fix the `findAllByClient` filter bug (`project.service.ts:108`, `'status'` → `'project_status'`) — this is a live runtime SQL error waiting to happen the first time the client-scoped project list UI passes a status filter, and it's a one-line fix.
3. Decide and implement real soft-delete semantics for `remove()` (`project.service.ts:203`, `repositories/project.repository.ts:53-55`) — today it's a hard delete masquerading as `softDelete`, with unused `deleted_at`/`deleted_by` columns on `BaseEntity` and no verified cleanup of child `project_team_mappings`/`project_contracts` rows, risking orphaned data or FK failures in production.
