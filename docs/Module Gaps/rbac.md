# Module: RBAC

## 1. Purpose & Current Usage

- Implements the app's authorization layer on top of authentication: assigns users to `Role`s per organization (`UserRoleMap`), grants roles per-module/per-action permissions (`ModuleAccess`), restricts those grants by the organization's active subscription `Plan` (`PlanModule` / `PlanModuleAction`), and propagates access up the module WBS hierarchy (child access ⇒ parent gets at least `view`). The core engine is `PermissionResolverService` (`brello_server/src/modules/rbac/services/permission-resolver.service.ts`), which exposes `resolve(user)` (full permission set, used to build the app menu) and `hasPermission(user, moduleCode, actionName)` (single check, used by the guard).
- Enforcement mechanism: controllers apply `@UseGuards(JwtAuthGuard, AccessGuard)` at the class level and annotate individual routes with `@RequirePermission(moduleCode, actionName)` (`brello_server/src/core/guards/require-permission.decorator.ts`). `AccessGuard` (`brello_server/src/core/guards/access.guard.ts`) reads that metadata via `Reflector`, pulls `userId`/`organizationId`/`appId` off the JWT payload, and calls `PermissionResolverService.hasPermission`. If no `@RequirePermission` metadata is present, the guard is a silent no-op (`return true`) — it never fails closed. This pattern is repeated by every controller that wants RBAC enforcement (26 of 74 controllers actually wire up `AccessGuard`; see Gaps).
- `MenuController` (`brello_server/src/modules/rbac/controllers/menu.controller.ts`) calls `resolve()` directly (not via the guard) to serve `GET /menu` and `GET /menu/permissions` for building the frontend nav — this is legitimate self-service (a user always may see their own permission set) and only needs `JwtAuthGuard`.
- Dead/unused: `PermissionResult.planId` is never actually populated on the hot path. `resolve()` computes `planId` via `getActivePlanId()` but then returns `buildModuleResult(...)`, which hardcodes `planId: null` (`permission-resolver.service.ts:337-341`) with the comment "filled by caller when needed" — no caller ever fills it. Only the empty-result branch (`emptyResult`, line 135-137) returns the real value. The field is effectively broken/inconsistent.

## 2. Intended / Ideal Usage

- A permission model resolved once per request (or cached per user/org/app with short-lived invalidation), not recomputed from 6+ SQL queries on every single guarded call.
- Every controller exposing organization data enforced uniformly through the same decorator + guard pair, with no controller silently relying on "authenticated" as a stand-in for "authorized."
- A platform-admin bypass that is narrowly scoped, time-bounded, and independently audited (not a blanket, unlogged `if (isPlatformAdmin) return true`).
- Permission denials (and platform-admin bypass usage) flowing into the same structured, queryable audit trail (`system_audit_logs`) that the rest of the app already uses for state-changing actions, not just an ephemeral `Logger.warn` line.
- Unit/integration test coverage for the resolution algorithm (role aggregation, plan intersection, WBS propagation) given it is a security-critical, non-trivial piece of logic.

## 3. Cross-Module Connections

- **Depends on:** `Role`/`RoleApp` (role module), `AppModule`/`Action`/`ModuleAccess` (app-module), `OrganizationSubscription`/`PlanModule`/`PlanModuleAction` (plan module), `User` and `LoggedInUser`/`JwtPayload` (auth module), `GlobalSearchModule` (imported into `RbacModule` but `RbacModule` itself doesn't obviously need search — see Gaps).
- **Depended on by:** essentially the entire backend. Every controller using `@UseGuards(..., AccessGuard)` + `@RequirePermission(...)` routes through `PermissionResolverService`; `RbacModule` exports `PermissionResolverService`, `RoleService`, `UserRoleMapService`, and repositories so other modules can import them. This is explicitly the module the task description calls "the security backbone of the entire app."
- **Missing/expected connections:** No caching layer between `AccessGuard`/`PermissionResolverService` and Postgres (no `cache-manager`/Redis anywhere in the codebase — confirmed via repo-wide grep). No connection to the audit-log system for denial/bypass events (`AuditInterceptor` only fires on successful handler execution via `@AuditLog`, so a `ForbiddenException` thrown by `AccessGuard` never reaches it). No mechanism to force-invalidate an already-issued JWT's `isPlatformAdmin`/org claims when a platform-admin flag or org membership is revoked mid-session (see Gaps).

## 4. Gaps

### Structural (architecture, module boundaries, coupling, missing abstractions, layering violations)

- **Guard/decorator split across two directories with a duplicated `JwtAuthGuard`.** `RequirePermission`/`AccessGuard`/`PlatformAdminGuard` live in `src/core/guards/`, but there is also `src/modules/auth/guards/jwt-auth.guard.ts` (used by most controllers, e.g. `client.controller.ts:19`) separate from `src/core/guards/jwt-auth.guard.ts` (used by `menu.controller.ts:2` and `access.guard.ts` doc comment). Two identical-purpose guards in two locations is confusing and increases the chance a controller wires the "wrong" one inconsistently.
- **`AccessGuard` fails open, not closed, when metadata is absent.** `access.guard.ts:48-51` — `if (!requirement) { return true; }`. Because enforcement is opt-in per route rather than deny-by-default, a controller that forgets to add `@RequirePermission` (or forgets `AccessGuard` entirely) is silently unprotected with no framework-level signal. This is the direct mechanism behind the biggest gap below.
- **`RbacModule` imports `GlobalSearchModule`** (`rbac.module.ts:21,37`) for reasons unrelated to permission resolution (likely leftover from role/user search indexing in `RoleService`) — a layering smell but not a security issue.

### Coding (bugs, dead code, inconsistent patterns, missing validation/error handling, unsafe assumptions)

- **`PermissionResult.planId` is always `null` on the non-empty path**, contradicting its own inline comment. `permission-resolver.service.ts:337-341` hardcodes `planId: null` even though `resolve()` computed the real `planId` earlier (line 88) and never passes it through to `buildModuleResult`. Any future caller relying on `planId` (e.g. to show "no active plan" messaging) will get wrong data.
- **Case-insensitive action matching only in `hasPermission`, not elsewhere.** `permission-resolver.service.ts:127` lower-cases both sides for the guard check, but `aggregateRolePermissions`/`applyPlanRestrictions` compare action names case-sensitively when building the menu (`resolve()` path used by `MenuController`). If an action name's casing ever differs between `module_access` and `plan_module_action` rows, the menu and the actual enforcement can disagree (menu says "no access", guard says "yes" or vice versa).
- **`UserRoleMapController.findByUser` / `findOne` leak cross-tenant data.** `user-role-map.controller.ts:49-61` — `findByUserId`/`findOne` (in `user-role-map.service.ts:42-48`, `user-role-map.repository.ts:50-69`) take a bare `userId`/`id` with no `organization_id` scoping check against the caller's own org. Any caller with `ACCESS_USERS:view` permission in Org A can pass an arbitrary `userId`/`id` belonging to Org B and read that user's role assignments across organizations.
- **No input validation that `role_id`/`organization_id` in `CreateUserRoleMapDto` actually belong together** — `create()` (`user-role-map.service.ts:15-31`) only checks for a duplicate mapping, never that the role's `app_id`/org matches the target `organization_id`/`user_id`'s org, so a caller could map a user to a role belonging to a different organization if they can guess the UUIDs.

### Technical (performance, security, scalability, test coverage, observability)

- **No caching anywhere — every single `@RequirePermission` check re-resolves the user's entire permission set from scratch.** `AccessGuard.canActivate` (`access.guard.ts:67`) calls `hasPermission`, which calls `resolve()` (`permission-resolver.service.ts:118-123`), which on every invocation runs: `getActivePlanId` (1 query), `getRoleIdsForUser` (1 join query), `aggregateRolePermissions` (1 join query), `applyPlanRestrictions` (2-3 queries: plan modules, plan actions, action-name lookup), and `appModuleRepo.find` (1 query) — roughly 6+ round trips to Postgres for a single permission check, repeated on every guarded request across all ~35 modules, with zero memoization even within the same request if multiple guards fire. A repo-wide grep confirms there is no `cache-manager`/Redis/`CacheModule` usage anywhere in the backend. At any meaningful request volume this is a significant, avoidable DB load multiplier and the single most consequential technical gap in the module.
- **Platform-admin bypass is a blanket, unaudited, unscoped `return true`.** `access.guard.ts:56-59` — `if (user?.isPlatformAdmin) { return true; }` skips all module/action/plan checks for every `@RequirePermission`-guarded route in the app, with no additional condition (no scoping to platform-only modules, no step-up auth, no reason/justification capture). Because `isPlatformAdmin` is a claim baked into the JWT at login (`token.service.ts`, default access-token TTL `15m` per `auth.JWT_ACCESS_EXPIRATION`), revoking platform-admin status does not take effect until the current access token expires — a demoted admin keeps full bypass power for up to the token's remaining lifetime. There is also no audit-log entry generated when a platform admin uses this bypass to touch org data (see next point), so bypass usage is invisible after the fact.
- **Several core business-data controllers have no authorization layer at all — only authentication.** A repo-wide check of all 74 `*.controller.ts` files shows 48 do not wire `AccessGuard`, and spot-checking the largest/most sensitive ones confirms real gaps (not just platform-admin-only or public endpoints): `modules/client/controllers/client.controller.ts:26-27` (`@UseGuards(JwtAuthGuard)` only, full CRUD on clients), `modules/user/controllers/user.controller.ts:26-27` (full CRUD on users/employees), `modules/project/controllers/project.controller.ts:27-28`, `modules/departments/controllers/department.controller.ts:27-28`, `modules/document/controllers/document.controller.ts:28-29`, `modules/reimbursement/controllers/reimbursement.controller.ts:29-30`, `modules/timesheet/controllers/timesheet.controller.ts:23-24`, and the base `modules/attendance/controllers/attendance.controller.ts:24-25` (note sibling controllers in the *same* attendance module, e.g. `admin-attendance.controller.ts` and `attendance-rule.controller.ts`, *do* use `@RequirePermission`, showing this is an inconsistency rather than an intentional design). Concretely: any authenticated user, regardless of assigned role or granted permissions, can create/update/delete clients, users, projects, and reimbursements today, because the route never reaches `AccessGuard`/`hasPermission` at all. This is the single biggest gap in the module — RBAC exists and is well-built, but a large share of the app's actual write surface never calls into it.
- **Permission denials are not audit-logged, only console-logged.** `access.guard.ts:74-76` uses `Logger.warn`, which is ephemeral process log output, not a `system_audit_logs` entry. `AuditInterceptor` (`modules/audit/interceptors/audit.interceptor.ts:31-60`) only fires its `tap()` callback on a *successful* response from `next.handle()`; since `AccessGuard` throws `ForbiddenException` before the controller handler (and thus before `next.handle()`'s success path) runs, denied/forbidden attempts never produce a durable audit record. Combined with the platform-admin bypass point above, there is no queryable trail of "who was denied access to what" or "when did a platform admin use their bypass."
- **Zero test coverage.** `find modules/rbac -name "*.spec.ts"` returns nothing — no unit tests for `PermissionResolverService`'s resolution algorithm (role OR-aggregation, plan AND-restriction, WBS propagation) nor for `AccessGuard`'s bypass/deny logic, despite this being explicitly called out as the highest-stakes module in the app.

## 5. Top 3 Priorities

1. **Close the coverage gap on core business controllers** (`client`, `user`, `project`, `department`, `document`, `reimbursement`, `timesheet`, base `attendance`) by adding `AccessGuard` + `@RequirePermission` to every mutating and data-exposing route — right now RBAC is well-designed but is simply not being invoked for a large share of the app's actual write surface, which is a live privilege gap, not a theoretical one.
2. **Add caching to `PermissionResolverService.resolve()`** (e.g., short-TTL per user+org+app cache invalidated on role/module-access/plan changes) — every guarded request currently costs 6+ DB queries with zero reuse, which will not scale and is pure unforced overhead today.
3. **Audit-log permission denials and platform-admin bypass usage**, and shrink the platform-admin bypass's blast radius (e.g., require the bypass to be explicit/scoped rather than an unconditional `return true` for every `@RequirePermission` route) — right now the most powerful account type in the system operates with no forensic trail.
