# Module: Role

## 1. Purpose & Current Usage

- **Entities**: `Role` (`role/entities/role.entity.ts:1-27`, table `role`, extends `BaseEntity`) with own columns `name`, `app_id` (FK→App), `roleApps` (OneToMany→RoleApp), `is_system_role`, `is_default`, unique index `(name, app_id, organization_id)`. `RoleApp` (`role/entities/role-app.entity.ts:1-35`, table `role_apps`) is a join table `{role_id, app_id}` used when a role spans multiple apps beyond its single "primary" `app_id`.
- **Services**: `RoleService` (`role/services/role.service.ts:1-311`) provides tenant-scoped CRUD (`create`, `findAll`, `findByFilter`, `findOne`, `update`, `remove`) plus a parallel platform-admin CRUD surface (`findPlatformRoles`, `createPlatformRole`, `updatePlatformRole`, `deletePlatformRole`, `:222-290`) for global template roles (`organization_id IS NULL`), and a private `validateSystemRoleAccess` guard (`:292-309`).
- **Controllers**: `RoleController` (`/roles`, 6 endpoints, all gated by `@RequirePermission('ACCESS_ROLES', ...)`, `role.controller.ts:1-97`) and `PlatformRoleController` (`/platform-roles`, gated by `PlatformAdminGuard`, `role.controller.ts` sibling `platform-role.controller.ts:1-60`).
- **Repositories**: `RoleRepository` and `RoleAppRepository` (`role/repositories/`), thin TypeORM wrappers; `RoleRepository.softDelete` (`role.repository.ts:88-93`) only flips `status`, never sets `deleted_at`.
- **Actual callers today**:
  - `organization.service.ts` `setupCompany` clones platform default roles per new org **by importing `Role`/`RoleApp` entities directly** (`organization.service.ts:29-30`) and writing through a raw `queryRunner.manager` — fetching templates (`:144-151`), cloning `Role` rows (`:158-168`), cloning `ModuleAccess` permission rows (`:171-183`), cloning `RoleApp` rows (`:186-194`), and creating `UserRoleMap` rows (`:204-212`) — entirely bypassing `RoleService`, `ModuleAccessService`, and the `rbac` module's own services.
  - `module-access.service.ts` (in the **third** module, `app-module`) injects `RoleService`, `RoleRepository`, `RoleAppRepository` directly (`:23-25, 40-42`) for `getPermissionsList()` (`:202-276`) and `propagateToOrgRoles()` (`:361-398`) — a second, independent implementation of "propagate platform role permissions to cloned org roles" that overlaps conceptually with `organization.service.ts`'s cloning logic.
  - `seeds/seed-brello-v2-base.ts` writes to `role`/`module_access` via raw SQL (`upsertRole` `:239-254`, `grantAll` `:302-317`), independent of both `RoleService` and the ORM entities.
  - Frontend: `brello_webapp/src/api/roles.ts:4-21` calls `GET/POST/PUT/DELETE /roles`, consumed by `features/access/roles/hooks/useRoles.ts` and `hooks/useRoles.ts`; a separate `features/platform/roles/api.ts` hits `/platform-roles`.
- **Dead/unused parts**: `RoleRepository.findAll()` (`role.repository.ts:23-28`) has zero callers anywhere in `src`. `Role.roleApps` (`role.entity.ts:19-20`) is never populated via the declared `OneToMany` relation — it's manually assigned post-fetch in three places (`role.service.ts:228, 252, 276`), making the relation decorative.

## 2. Intended / Ideal Usage

- Orgs are expected to start from **cloned platform-default roles** (system templates flagged `is_system_role=true, is_default=true, organization_id=NULL`) rather than defining permissions from scratch; `setupCompany` performs this clone at org creation.
- Ideal usage would have org admins able to edit permissions per cloned role (via the `ModuleAccess` grid, owned by `app-module`) and, per `validateSystemRoleAccess`, non-platform-admins should only be able to touch system roles that belong to their own org — implying custom, org-created roles are also a supported path through `RoleController.create`.
- A single owner should perform "clone defaults for a new org" and "propagate a platform template's permission change to already-cloned org roles" — today this is implemented twice (`organization.service.ts` clone path and `module-access.service.ts:propagateToOrgRoles`), which is not the ideal shape.

## 3. Cross-Module Connections

- **Depends on**: `App` entity (role's `app_id`/`RoleApp.app_id` FK), `SearchIndexingService` (indexing roles for global search), `AuditContextService` (pre-value capture for audit log).
- **Depended on by**: `rbac` module — `UserRoleMap.role_id` is a real FK→`Role` (`rbac/entities/user-role-map.entity.ts:36-38`); `app-module`'s `ModuleAccess.role_id` is FK→`Role` too; `organization` module for org-bootstrap cloning; `rbac.module.ts` re-declares `RoleService`/`RoleRepository`/`RoleAppRepository` as its own providers (`rbac.module.ts:11, 16-17, 35-68`) and wires `RoleController` into its own `controllers` array (`:50`) rather than importing `RoleModule` and re-exporting — `role.module.ts` itself only registers `PlatformRoleController` (`role.module.ts:15`), with a doc-comment acknowledging `RoleController` lives in `RbacModule` (`:11-12`).
- **Missing/expected connections — role vs rbac boundary**: In principle the split is clean (`role` = pure CRUD + tenant/system-role authorization; `rbac`'s `PermissionResolverService` = permission enforcement, querying `UserRoleMap`/`ModuleAccess`, never touching `Role`/`RoleApp` directly except via `UserRoleMap`'s join, `permission-resolver.service.ts:140-158`). In practice it is muddied by (a) duplicate DI provider registration of `Role`-related classes in both `role.module.ts` and `rbac.module.ts` against the same `TypeOrmModule.forFeature([Role, RoleApp])`, (b) `app-module/app-module.module.ts` importing **both** `RoleModule` and `RbacModule` (`:34-35`) to obtain `RoleService`, leaving which provider instance actually gets injected dependent on Nest's module-resolution order rather than one canonical source, and (c) `ModuleAccess` (the real permission grid, fine-grained module+action) and `RoleApp` (coarse app-membership) both being written from three different modules (`role`, `app-module`, `organization`) with no single service enforcing consistency between a role's `RoleApp` set and its `ModuleAccess` grants.

## 4. Gaps

### Structural (architecture)
- **`rbac.module.ts` redeclares `role` module's providers instead of importing `RoleModule`.** `rbac.module.ts:11, 16-17, 35-68` imports `Role`/`RoleApp` entities and `RoleRepository`/`RoleAppRepository`/`RoleService` classes directly and re-registers them as its own providers/exports, and also owns `RoleController` in its `controllers` array (`:50`), while `role.module.ts:15` registers only `PlatformRoleController`. This means `RoleModule` cannot be understood, tested, or deployed as a self-contained unit — its primary controller and DI registration live in a different module.
- **`organization.service.ts` bypasses every service-layer abstraction during org bootstrap.** `setupCompany` (`organization.service.ts:29-30, 62, 143-212`) writes directly to `Role`, `RoleApp`, `ModuleAccess`, and `UserRoleMap` entities via a raw `queryRunner.manager`, duplicating logic that already exists in `RoleService.createPlatformRole`/`ModuleAccessService` and skipping their validation (name-uniqueness, `validateSystemRoleAccess`, audit hooks, search indexing) — this is the clearest layering violation in the module and the single largest source of drift risk between "how a role is created via API" and "how a role is created at org setup."
- **Two independent implementations of "propagate role/permission changes to cloned org roles."** `module-access.service.ts:361-398` (`propagateToOrgRoles`) and `organization.service.ts:170-194` (clone-time copy) both encode the same conceptual rule from different modules with no shared abstraction, so a future permission-model change has to be updated in two places or will silently diverge.

### Coding (bugs, dead code, inconsistent patterns)
- **`GET /roles?search=...` is broken.** `RoleService.findAll()` passes `searchFields: ['name', 'context']` to `ListingHelper.apply` (`role.service.ts:111`), but `Role` has no `context` column (`role.entity.ts:1-27`); `ListingHelper.apply` builds `role.context ILIKE :search` verbatim (`listing.helper.ts:31-40`), so any search query will throw a Postgres "column does not exist" error. This is a straightforward, currently-shipping bug on a commonly-used list endpoint.
- **`is_system_defined` maps to two different columns depending on which method handles it.** In `create`/`update` it maps to `is_system_role` (`role.service.ts:57-61, 188-190`); in `createPlatformRole`/`updatePlatformRole` the same input field instead maps to `is_default` (`:242-243, 269`), with `is_system_role` hardcoded `true`. One DTO field silently means two different entity columns depending on the endpoint — a real correctness hazard for anyone extending this API.
- **Dead code**: `RoleRepository.findAll()` (`role.repository.ts:23-28`) has no callers anywhere in `src`. `Role.roleApps` OneToMany relation (`role.entity.ts:19-20`) is never populated by the ORM — it's manually assigned post-fetch in three call sites (`role.service.ts:228, 252, 276`), so the relation declaration is decorative.
- **`UpdateRoleDto` has no update-specific validation.** It is a bare `PartialType(CreateRoleDto)` (`update-role.dto.ts:4`); nothing stops a caller from changing a role's `app_id` on `PUT /roles/:id` even though the role may already have `ModuleAccess` grants scoped to modules under the old app — those grants are not re-validated against the new `app_id`.
- **Seed script's existence check is inconsistent with the live soft-delete implementation.** `seed-brello-v2-base.ts:242` filters on `WHERE ... AND deleted_at IS NULL`, but `RoleRepository.softDelete` (`role.repository.ts:88-93`) never sets `deleted_at` — it only flips `status` — so a soft-deleted role's row still passes the seed's "not deleted" check and could be silently reused/upserted instead of a fresh row being created.

### Technical (performance, security, scalability, test coverage, observability)
- **A role can be deleted or reassigned while users still hold it, with no check against `UserRoleMap`.** `RoleService.remove()` (`role.service.ts:203-220`) and `RoleService.update()` (`:154-201`) never import or query `UserRoleMapRepository`. `UserRoleMap.role_id` has `onDelete: CASCADE` at the DB level (`rbac/entities/user-role-map.entity.ts:36`), but since `remove()` only soft-deletes (sets `status`, never removes the row), the CASCADE never fires — stale `UserRoleMap` rows pointing at a deleted role persist silently. `PermissionResolverService.getRoleIdsForUser` does filter `role.status = ACTIVE` (`permission-resolver.service.ts:152-154`) so permissions correctly stop applying, but the user is left with a dangling role assignment with no cleanup or warning surfaced anywhere. This matters because it's an easy path to silent permission loss (user thinks they have a role, has none) with no operator visibility.
- **`UserRoleMapService.remove()` does a bare hard delete with no "last role" check.** (`rbac/services/user-role-map.service.ts:50-53`) — a user's final role assignment in an org can be removed without warning, potentially leaving them with zero access and no signal to the caller.
- **Zero test coverage.** No `.spec.ts` files exist for `role` or `rbac` (confirmed by directory search); the only 5 spec files in the entire backend are under `payroll`/`letter-management`. This is notable given `PermissionResolverService`'s multi-step resolution logic (role→module-access→plan-restriction→WBS propagation) and the cross-module coupling documented above have no regression safety net at all.
- **Two divergent `JwtAuthGuard` implementations are in play across this exact cluster of controllers** — `core/guards/jwt-auth.guard.ts` (used by `RoleController`) vs. `auth/guards/jwt-auth.guard.ts` (used by `PlatformRoleController`, `UserRoleMapController`), which differ in `@Public()`-bypass support. Not role/rbac-specific in origin, but it means the guard behavior on `/roles` vs `/platform-roles` is not actually uniform despite looking that way at a glance.

## 5. Top 3 Priorities

1. **Add a `UserRoleMap` check before role delete/update, or at minimum surface a warning/reassignment flow** (`role.service.ts:154-220`) — this is a live data-integrity and security gap: users can be left holding orphaned role assignments with silently-revoked permissions and no operator visibility.
2. **Collapse `organization.service.ts`'s raw-entity role/permission cloning (`:143-212`) and `module-access.service.ts`'s `propagateToOrgRoles` (`:361-398`) into one canonical "clone/propagate" service**, and have `RbacModule` `imports: [RoleModule]` instead of redeclaring its providers — this is the root cause of the duplicate-DI-registration smell and the two-implementations-of-one-rule problem, and the biggest structural risk to future changes in the permission model.
3. **Fix the `context` search-field bug** (`role.service.ts:111`) — small effort, but it's a currently-broken production code path (`GET /roles?search=`) that will throw on any real usage.
