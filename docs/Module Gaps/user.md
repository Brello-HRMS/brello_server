# Module: User

## 1. Purpose & Current Usage

- **Entities**: `User` (auth/identity record — email, phone, password hash, `status`, `department_id`, `designation_id`, `reports_to_id`, `plan_id`) and `UserProfile` (HR/employee data — `employee_status`, DOB, salary strings, exit info, 1:1/1:N links to education, experience, assets, documents, gov info, bank info, emergency contacts). `User.status` (common `Status` enum: PENDING/ACTIVE/INACTIVE/DELETED) is the field auth actually gates login on; `UserProfile.employee_status` (DRAFT/INVITED/ACTIVE/OFFBOARDING/INACTIVE) is a separate parallel lifecycle enum. Both must be kept in sync manually by each call site — there is no single source of truth.
- **Services**: `EmployeeService` (`brello_server/src/modules/user/services/employee.service.ts`, 1358 lines) owns the entire employee lifecycle — creation, profile/personal/employment/payroll/system-access updates, education/experience/asset/document/emergency-contact CRUD, onboarding invite, activation, exit/offboarding initiate-update-cancel, photo upload, dashboard stats, new-hires, birthdays. `UserService` (`services/user.service.ts`) is much thinner — generic user CRUD, department/designation mapping, listing. `OffboardingCronService` (`services/offboarding-cron.service.ts`) runs one daily cron that finalizes offboarding.
- **Controllers**: `EmployeeController` (`controllers/employee.controller.ts`) exposes ~30 routes under `/employees`; `UserController` (`controllers/user.controller.ts`) exposes ~9 routes under `/users`. Both are thin pass-throughs to their services.
- **Callers**: `AuthService` (`modules/auth/services/auth.service.ts:207-217, 242-247`) calls `employeeService.activateEmployee()` and `employeeService.findProfileByUserId()` during OTP login to auto-activate first-time users and to gate login for INVITED-but-PENDING accounts. `OrganizationService` and `DepartmentService` call into `UserService`/`UserRepository` for org setup and department listings. Nearly every other module (attendance, leave, payroll, letters, documents, notification, audit, global-search) references `user_id`/`employee_id` as a foreign key — this module is the identity anchor for the whole system.
- **Dead/unused code**: `UserResponseDto` (`dto/user-response.dto.ts`) is fully built (`@Exclude`/`@Expose`, explicitly documents that it hides `password_hash`) but is never imported or applied anywhere (`grep` finds zero usages outside its own definition) — no `ClassSerializerInterceptor` is registered globally either, so the DTO's protection is entirely inert. `NotificationEventType.EMPLOYEE_INVITED` and `EMPLOYEE_ACTIVATED` (`common/enums/notification-event-type.enum.ts:23-24`) are defined but never referenced anywhere in the codebase — dead enum values that suggest notification wiring was planned but abandoned.

## 2. Intended / Ideal Usage

A correct employee-lifecycle module should: (1) maintain exactly one authoritative status field (or keep two in lockstep via a single transition function) so "can this person log in" is never ambiguous; (2) fire a notification/event at every lifecycle transition (invite, activation, offboarding initiated/finalized, birthday, work anniversary) so downstream consumers (email, in-app, Slack, admins) can react without polling; (3) have automated crons for every calendar-driven HR event, not just offboarding; (4) never leak authentication secrets (`password_hash`) through generic list/get endpoints; (5) route all system-access/role grants through the actual RBAC assignment table rather than repurposing unrelated columns; (6) keep one audit trail per write, not two or three competing ones.

## 3. Cross-Module Connections

- **Depends on**: `departments`/`designations` (FK + repositories for listing/mapping), `enterprise`/`organization` (tenant scoping, validated in `UserService.create`/`update`), `document` module (`DocumentService.uploadDocument`/`buildViewUrl`, `FolderType.EMPLOYEE_IMAGE` for profile photos — `employee.service.ts:1062-1071`), `global-search` (`SearchIndexingService.indexEmployee`/`removeEmployee` — `employee.service.ts:239-244, 1101`), `notification` (`NotificationService.send` for invite emails only), `audit` (`IAuditService`/`AuditContextService`, `AuditLogModule.EMPLOYEE`), and its own legacy `audit_log` table/repository.
- **Depended on by**: virtually every other module via `user_id`/`employee_id` FKs (attendance, leave, payroll, reimbursement, letters, rbac's `user_role_map`, etc.) — this is the expected fan-out pattern for an identity table and isn't itself a gap.
- **Missing/expected connections** (per known context, confirmed in code):
  - `activateEmployee()` (`employee.service.ts:1175-1237`) has zero notification calls — an employee moving INVITED→ACTIVE gets no email/in-app confirmation, despite `NotificationEventType.EMPLOYEE_ACTIVATED` existing specifically for this.
  - `initiateOffboarding`/`updateOffboarding`/`cancelOffboarding` (`employee.service.ts:944-1034`) and `OffboardingCronService.handleOffboarding` (`offboarding-cron.service.ts:18-48`) have zero notification calls, and there is no `NotificationEventType` value for any offboarding event (initiate, cancel, or finalize) — HR/IT/the employee's manager learn about an exit only by opening the UI.
  - `findBirthdaysThisMonth` (`repositories/user.repository.ts:171-181`) and `getBirthdays` (`employee.service.ts:1336-1357`) are pull-based only, exposed via `GET /employees/birthdays` — nothing pushes a birthday notification; `grep` for "anniversary"/"work anniversary" across the whole `src` tree returns zero hits — there is no work-anniversary feature at all (no query, no cron, no endpoint).
  - New-device/new-location login and password-changed notifications are out of scope here (they live in `auth`), but that module has the identical gap — noted only as a boundary, not re-audited.

## 4. Gaps

### Structural

- **`EmployeeService` is a god-service.** `employee.service.ts` is 1358 lines and owns onboarding, activation, offboarding, payroll fields, education/experience/asset/document/emergency-contact CRUD, dashboard analytics (`getDashboardStats`, raw SQL against `attendance_records` — `employee.service.ts:1239-1311`), and birthday/new-hire queries. It mixes at least four responsibilities (profile-section CRUD, lifecycle state machine, HR analytics, and document/photo handling) that would normally be split into separate services; this makes the lifecycle logic (the part most in need of careful review) hard to find and change safely.
- **Three competing audit trails write for the same actions.** The module writes to its own legacy `audit_log` table via the private `createAuditLog()` helper (e.g. `employee.service.ts:520-527` inside `updatePersonalDetails`), while the same action is *also* captured by the centralized `@AuditLog(AuditLogModule.EMPLOYEE, AuditAction.UPDATE, 'employee')` decorator on the controller method one layer up (`controllers/employee.controller.ts:103-111`), and a third, unrelated centralized path (`IAuditService`/`system_audit_logs`) is used only in `createEmployee` (`employee.service.ts:246-259`). `updatePersonalDetails` is thus double-logged into two different audit systems while most other mutations (`updateEmploymentDetails`, `updateSystemAccess`, offboarding actions) only hit the legacy local table and never reach the new centralized audit log — inconsistent coverage plus duplicate writes.
- **`User.status` and `UserProfile.employee_status` are two parallel lifecycle fields with no single transition function.** Every lifecycle method hand-rolls which of the two fields to touch (`activateEmployee` updates both in a transaction; `cancelOffboarding` updates both but via two separate non-transactional calls — `employee.service.ts:1008-1034`; the offboarding cron updates only `UserProfile`, see Coding gap below). This split state makes it easy — and it has already happened — to leave the two out of sync.

### Coding

- **Offboarding cron never revokes login access.** `OffboardingCronService.handleOffboarding()` (`offboarding-cron.service.ts:35-44`) only calls `this.profileRepository.update(profile.id, { employee_status: EmployeeStatus.INACTIVE, status: Status.INACTIVE })` — this writes to the `user_profile` table only (confirmed in `UserProfileRepository.update`, `repositories/user-profile.repository.ts:48-54`, a plain `repository.update` on the `UserProfile` entity). It never touches the `User` row's `status` column. But `AuthService.findActiveUserByEmail()` (`modules/auth/services/auth.service.ts:235-247`) gates login on `user.status === Status.ACTIVE` (falling back to a PENDING+INVITED check for first login only) — it never consults `UserProfile.employee_status`. The result: once an employee's last working day passes and the cron "finalizes" their offboarding, they can still log in indefinitely, because the field auth actually checks was never updated. Contrast with `cancelOffboarding()` (`employee.service.ts:1008-1034`), which explicitly does `this.userRepository.update(id, { status: Status.ACTIVE })` — proving the developer knew `User.status` is the gating field elsewhere, just not in the cron. This is a live security hole for every terminated employee.
- **`updateSystemAccess` writes a "role" into the wrong, unrelated column.** `employee.service.ts:590-592`: `if (dto.roleId) { await this.userRepository.update(id, { plan_id: dto.roleId }); }`. `User.plan_id` is the org's **billing/subscription plan** reference — it is set once during `OrganizationService.setupCompany()` (`modules/organization/services/organization.service.ts:76-81, 138-140, 224`) to seed the org's `OrganizationSubscription` and enabled apps, and RBAC's actual permission resolution reads the plan ID from `organization_subscription` (`modules/rbac/services/permission-resolver.service.ts:186-195`), never from `User.plan_id`. Meanwhile the real per-user role assignment mechanism is the `user_role_map` junction table (`modules/rbac/entities/user-role-map.entity.ts`). So `updateSystemAccess` silently no-ops as a role-granting feature (the role never reaches `user_role_map`, so it has no effect on the employee's actual permissions) while also stomping on a column that, for the one user where it's meaningful (the org's founding admin), is tied to billing/subscription setup. The inline comment `// Assuming role/plan link or similar` (line 586) shows this was never actually verified.
- **`getEmployeeAggregate` and generic `User` CRUD leak `password_hash`.** `UserController.findOne`/`update`/`create` (`controllers/user.controller.ts:91-107`) return the raw `User` entity from `UserService.findOne`/`update` (`services/user.service.ts:310-380`), and `UserRepository.findById` (`repositories/user.repository.ts:32-37`) has no `select` exclusion, so `password_hash` is serialized straight into the JSON response for `GET /users/:id` and `PATCH /users/:id`. The purpose-built `UserResponseDto` that would prevent this (see §1) is never wired in, and there is no global `ClassSerializerInterceptor`. This is the concrete instance of the PII/security gap called out in §4 Technical below.
- **`updateOffboarding` silently allows extending the last working day past what the cron already finalized**, and `cancelOffboarding` restores a user without checking whether the cron already ran and moved them to `INACTIVE` — there is no `is_processed`/finalized guard on `EmployeeOffboarding`, only `is_cancelled` (`entities/offboarding.entity.ts:37-38`), so `updateOffboarding`/`cancelOffboarding` can be called after the cron has already deactivated the profile, leaving the two out of sync again in the opposite direction.
- **`removeDocument` has a documented but unresolved ID-type ambiguity**: `employee.service.ts:906` — `// docId from parameter could be the user_document table ID, not S3 doc ID` — the method proceeds to soft-delete assuming the join-table ID, which is a latent bug if callers ever pass the underlying `Document` ID instead.

### Technical

- **PII/sensitive-field exposure**: as above, `password_hash` is returned by `GET/PATCH /users/:id`. Additionally `UserProfile` carries bank account numbers, PAN/Aadhaar (`user-gov-info`/`user-bank-info` entities), and salary fields directly on the aggregate returned by `getEmployeeAggregate` (`employee.service.ts:293-319`) with no field-level redaction based on caller's role — anyone who can call this endpoint sees full bank/gov ID data.
- **No test coverage**: there are no `.spec.ts` files alongside any service/controller/repository in `modules/user/` (confirmed via directory listing) — the lifecycle logic identified above (offboarding cron, activation, system access) has zero automated regression protection.
- **Dashboard stats run raw parameterized SQL** against `attendance_records` inside `EmployeeService.getDashboardStats` (`employee.service.ts:1264-1298`) rather than going through a repository/query builder — a layering inconsistency (this service reaching into another module's table directly) and a maintenance risk if that table's schema changes.
- **N+1 query pattern in the offboarding cron**: `handleOffboarding()` loops over every pending offboarding record and issues a separate `profileRepository.findByUserId()` + `update()` per record (`offboarding-cron.service.ts:35-44`) instead of a single batched query — fine at current scale, will not scale with organization/employee count growth.

## 5. Top 3 Priorities

1. **Fix `OffboardingCronService` to also set `User.status = INACTIVE`** (and add an idempotency guard) — this is a live security gap where terminated employees retain full login access indefinitely after their last working day, and it's a one-line fix given the pattern already used in `cancelOffboarding`.
2. **Redirect `updateSystemAccess`'s `roleId` write to `user_role_map` instead of `User.plan_id`** — the feature currently does nothing useful for permissions while corrupting an unrelated billing-plan column; this is presented to admins as "granting system access" but silently fails.
3. **Wire `UserResponseDto`/a serializer onto `UserController`'s CRUD endpoints** to stop leaking `password_hash`, and add the missing `NotificationEventType` values + calls for `activateEmployee` and the offboarding lifecycle (initiate/cancel/finalize) so the two enum values that already exist (`EMPLOYEE_INVITED`, `EMPLOYEE_ACTIVATED`) stop being dead code and HR/managers actually learn about lifecycle events.
