# Chapter 8 — Frontend UX, Screens & User Flows

---

# 8.1 Overview

The Letter Management module is designed around a simple principle:

> **Generating a letter should require the fewest possible clicks while maintaining complete control and accuracy.**

The UI should feel consistent with the rest of Brello:

- Clean
- Fast
- Minimal
- Predictable
- Responsive

Every screen follows Brello's design system and shared components.

# 8.31 Frontend State Management

The frontend should separate state into three layers:

- **Server State**: Lists, templates, issued letters, categories (managed using TanStack Query or the project's standard data-fetching layer).
- **UI State**: Open dialogs, selected rows, filters, loading indicators.
- **Draft State**: Template editor content, unsaved manual inputs.

Avoid duplicating server state in global stores.

---

# 8.32 Chapter Summary

The frontend experience prioritizes operational efficiency. HR users can generate letters in three simple steps, template management remains intentionally structured rather than free-form, and every screen follows consistent navigation, validation, and feedback patterns. By reusing Brello's design system and separating server, UI, and draft state, the module remains scalable and maintainable as new document types are introduced.

---

## Product Review & Final UX Recommendations

Compared to the previous design, these improvements make the module more production-ready:

1. **Use a drawer instead of full-page navigation** for letter generation to keep HR in context and reduce navigation overhead.
2. **Keep Preview and Generate as separate actions**, ensuring users can verify content before committing an immutable legal document.
3. **Adopt optimistic UI only for configuration actions** (e.g., category updates), but **never** for letter generation, which must always wait for backend confirmation due to its transactional and legal nature.
